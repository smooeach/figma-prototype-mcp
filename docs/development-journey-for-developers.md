> 📌 이 문서는 **개발자용**입니다. 버전·API·아키텍처 중심의 기술 회고예요.
> 비기술 독자(디자이너·기획)를 위한 서사 버전은 [개발 여정 (디자이너용)](./development-journey-for-designers.md)을 보세요.

> 한 줄 요약: "디자이너가 만든 화면을, 자연어로, 프로토타입 인터랙션으로 엮어 주는 MCP 서버"를
> v0.1.0부터 v0.34.0까지 어떻게 만들어 왔는지에 대한 시간순 회고.

이 문서는 코드/커밋 히스토리가 말해 주지 않는 것 — **왜 그 순서로 만들었고, 무엇에 막혔으며,
무엇을 배웠는지** — 를 남기기 위한 기록이다. 사실관계(버전·도구 개수)는 작성 시점(2026-06-17)의
`package.json`(0.34.0)·git 태그(v0.1.0~v0.34.0)·`src/server/tools.ts`(등록 도구 21개)와 대조해 확인했다.

처음에는 기능 사이클을 내부적으로 "v1.x"로 세었지만, 서버 통합(아래 5막) 이후로는 공개 릴리즈 태그
"v0.x.0"로 번호 체계를 일원화했다. 이 문서는 **공개 릴리즈 태그를 기준선**으로 삼는다.

## 1막 — 와이어링 MVP (v0.1.0)

시작점의 제품 가설은 분명했다. **노드/화면을 "만드는" 일은 우리 몫이 아니다.** 그건 공식 Figma MCP가
이미 한다. 우리가 채울 빈칸은 그 다음 — 이미 그려진 화면들을 **프로토타입 인터랙션으로 엮는(wire)**
일이다. 이 경계는 끝까지 흔들리지 않았고, 나중에 "생성 기능을 추가하자"는 유혹을 매번 거절하는 근거가 됐다.

첫 릴리즈는 실제 Figma 파일에서 5개 E2E 시나리오를 통과시키는 것으로 합격선을 잡았다. 그리고 곧바로
구조적 진실 하나를 만났다: **플러그인은 선택이 아니라 필수다.** REST API로는 reaction을 쓸 수 없고,
헤드리스 경로가 없다. 그래서 "stdio MCP 서버 ↔ WebSocket ↔ Figma 플러그인"이라는 3단 구조가
태생부터 정해졌다.

## 2막 — 인터랙션 표면을 넓히다 (v0.2.0 ~ v0.5.0)

MVP가 서면, 다음은 Figma 프로토타입이 표현할 수 있는 인터랙션을 하나씩 따라잡는 일이었다.

- **v0.2.0** — Scroll To, `create_reactions` union 정리
- **v0.3.0** — Open Overlay / Close Overlay
- **v0.4.0** — Back / Open URL / Swap Overlay
- **v0.5.0** — URL `openInNewTab`

이 구간에서 반복적으로 부딪힌 것이 이후 모든 작업을 지배하는 1번 교훈이 됐다: **라이브 API가
typings보다 우선한다.** Figma의 공식 타입 선언이 "필수"라고 적어 둔 필드를 막상 런타임이 거부하거나,
overlay 액션의 실제 wire 형태가 문서와 다른 식의 "라이브 검증에서만 드러나는 놀라움"이 연속으로 나왔다.
SCROLL_TO가 액션 타입이 아니라 노드의 navigation 속성이었던 것도 이때의 교훈이다.

## 3막 — 연결을 단단하게, 모션을 풍부하게 (v0.6.0 ~ v0.13.0)

인터랙션 종류가 늘자 두 갈래로 안정화가 필요했다.

**연결 안정성** — 플러그인 자동 연결(`figma.clientStorage`, v0.6.0), 지수 백오프 자동 재연결
(v0.7.0), 서버 브리지 재연결(v0.8.0). 데모 중 끊기면 신뢰를 잃으므로, 화려한 기능보다 먼저 깔아야 할
바닥이었다.

**모션과 트리거** — AFTER_TIMEOUT 트리거(v0.9.0)에 이어, 트랜지션을 3단계로 채웠다: 커스텀
duration + cubic/back easing(v0.10.0), 스프링 프리셋 + 커스텀 easing(v0.11.0), 방향성 트랜지션 +
신규 트리거 7종(v0.12.0).

그리고 **v0.13.0 `set_frame_scroll`** — 처음으로 reaction이 아니라 프레임 자체의 속성
(`overflowDirection`)을 세팅하는 도구. "세로 스크롤을 수동으로 켜 둬야 한다"는 시나리오상의 전제를
제거했다. 단순 가변 속성이라 라이브 놀라움이 없었던, 모처럼 매끄러운 릴리즈였다.

## 4막 — 조건부와 스크롤 옵션 (v0.14.0 ~ v0.16.0)

- **v0.14.0** — 스크롤 옵션 번들: `fixedChildren` + `resetScrollPosition`, 그리고 deprecated된
  `preserveScrollPosition` 제거. 서브에이전트 리뷰가 놓친 라이브 버그를 라이브 검증이 3건 잡았고,
  여기서 "계획→코드 사이의 갭" 패턴을 문서화했다.
- **v0.15.0** — 조건부 reaction MVP: 단일 비교 + IF/ELSE, 변수를 이름으로 참조.
- **v0.16.0** — `set_variable` + `toggle_variable`. BOOLEAN 토글에 "쓸 때 desugar / 읽을 때 match"
  패턴을 도입하고, 조건부와 `set_variable`이 변수 리터럴 호환성 검증 로직을 공유하게 했다.

## 5막 — 변수, 그리고 서버 통합 Phase A/B/C (v0.17.0 ~ v0.20.0)

**v0.17.0**에서 `set_variable`이 hex 문자열로 COLOR까지 다루게 되며 변수 축이 일단락됐다(alpha 필수가
런타임에서만 드러난 또 하나의 라이브 발견).

이어 아키텍처를 한 번 크게 손봤다 — **서버 통합 3부작**:

- **Phase A (v0.18.0)** — stdio MCP와 WS 릴레이를 하나의 24/7 Express 서버로 합쳤다. 단일 활성
  세션 모델, 3초 타임아웃 가드. 라이브 프로브가 Express 5의 body-parser 버그를 잡았다.
- **Phase B (v0.19.0)** — `proto_*` 고수준 래퍼의 첫 형태(`proto_wire`/`overlay`/`scroll`)와
  모션 프리셋 10종. SMART_ANIMATE를 overlay에 걸 때의 제약이 컴파일타임 재작성을 요구함을 라이브가 잡았다.
- **Phase C (v0.20.0)** — 인메모리 히스토리 스택 + `proto_get_last_history`. 순수 인프로세스
  상태라서, **v0.x 시리즈 최초로 라이브 API 놀라움이 0건**이었던 릴리즈.

## 6막 — proto_* 고수준 표면의 완성 (v0.21.0 ~ v0.23.1)

자연어로 다루기 좋은 `proto_*` 표면을 완성하는 구간.

- **v0.21.0** — `proto_back` + `proto_url`로 고수준 내비게이션 클러스터 완성.
- **v0.22.0** — `proto_set_variable` + `proto_toggle_variable`(COLOR hex 포함).
- **v0.23.0** — `proto_conditional`. 고수준 표면이 일단 완성됐고, 라이브 프로브 2종 PASS,
  3연속 무놀라움.
- **v0.23.1** — MCP 클라이언트(SSE) 쪽도 "최신 우선 단일 활성"으로 만들어 플러그인 쪽과 대칭을 맞췄다.

여기서 Claude Desktop 엔드투엔드 검증의 중요한 발견이 나왔다: `proto_get_last_history`가 잘 안 불린다 —
Claude Desktop의 네이티브 채팅 메모리가 이겨 버리기 때문. 도구가 좋아도 호스트 환경이 다르게 행동할 수
있다는 교훈.

## 7막 — 플랫폼의 벽 (v1.4 시절 · v0.24.0 작업 중)

두 번, **Figma 플랫폼 자체가 길을 막았다.**

1. **overlay 프레임 속성(readonly)** — overlay의 프레임 속성이 Figma typings에서 readonly이고
   플러그인 세터가 없었다. 스펙/계획을 다 세운 뒤에야 막혔음을 알았다. 그 대가로 브레인스토밍 흐름에
   **"readonly 체크를 스펙 단계에서 의무화"** 한다는 규칙이 생겼다(2번 교훈).
2. **조건부 else-if 체인** — 357 테스트까지 다 초록으로 만들고 나서 폐기했다. Figma 프로토타입에는
   else-if가 없다(UI에도 없고, `setReactionsAsync`가 3블록을 2블록으로 접어 버린다). 여기서
   "프로토타입 UI가 실제로 그 기능을 갖는지"를 브레인스토밍에서 먼저 확인하는 체크가 추가됐다.

막혔다는 사실 자체를 역사적 기록으로 남긴 것이, 이후 같은 길을 다시 파지 않게 해 줬다.

## 8막 — 변수 발견성과 자연어 조준 R2 (v0.24.0 ~ v0.26.0)

이 무렵부터 "기능"보다 "LLM이 우리 도구를 제대로 고르고 채우게 하는 것"이 과제가 됐다.

- **v0.24.0** — `list_variables` + 라이브러리/원격 변수 해석. LLM이 stale한 "local" 설명을 보고
  라이브러리 변수를 미리 거절하던 문제를 고쳤다.
- **v0.25.0** — 변수 이름 충돌 시 `collection` 필드로 모호성 해소. 충돌을 조용히 첫 항목으로 고르지
  않고 에러로 만들었다.
- **v0.25.1 / v0.26.0** — 자연어 조준(NL steering) 다듬기. overlay "돌아가"가 back인지 close인지
  애매하면 LLM이 사용자에게 묻게 했고, no-matching-layer일 때 SMART_ANIMATE를 DISSOLVE로
  자동 강등하는 등 그린필드 검증에서 나온 갭을 메웠다.

여기서 3번 교훈이 확립됐다: **스키마의 `.describe()`가 자연어 동작의 단일 진실 공급원(source of
truth)이다.** 기능 코드를 안 건드리고 describe만 고쳐서 LLM 행동을 바꾼 릴리즈가 여럿 있었다.

## 9막 — change_to와 복합 조건 (v0.27.0 ~ v0.28.0)

- **v0.27.0** — `proto_change_to`: 인스턴스를 형제 variant로 전환. 마지막 남은 인터랙션 표면 갭을
  메웠다. overlay와 달리 모션 제약이 없었고, "현재 variant" 가드를 새로 넣었다.
- **v0.28.0** — 복합 조건(AND/OR, 한 단계). `proto_conditional`/`create_reactions`의 `if`가
  2개 이상 비교에 대한 `all`/`any`를 받는다. **첫 풀 서브에이전트 주도 계획 실행**이었고,
  타입이 얽힌 작업은 묶어서/순서대로 처리해 커밋을 항상 초록으로 유지한다는 교훈을 남겼다.

## 10막 — 읽기 축의 등장 (v0.29.0)

`get_prototype_flow`(18번째 도구). 이때까지는 전부 "쓰기"였는데, 처음으로 **읽고 이해하는** 도구가
나왔다 — 페이지 전체의 인터랙션 그래프를 프레임 + 모든 인터랙션으로 반환한다. 라이브 프로브가 중요한
한계를 잡았다: 프레임이 Section 안에 중첩되므로 `page.children`이 아니라 `page.findAll`로 열거해야 한다
(`get_canvas_overview`도 같은 잠복 한계를 갖고 있었다).

## 11막 — 공개, npm, 그리고 데스크톱 확장 (v0.30.0 ~ v0.33.0)

제품을 세상에 내보내는 구간.

- **v0.30.0** — 저장소를 공개로 전환(내부 문서/설정 정리), npm 게시 가능 형태로 패키징(bin +

- **v0.30.1 ~ v0.30.3** — 연결 오류 안내 메시지(이중 언어), 그리고 `--stdio` 서버 모드로

- **v0.31.0 / v0.32.0** — `export_interactions`: 인터랙션을 정규 JSON으로 내보내는 개발자

이로써 3개 채널(npm · GitHub 릴리즈 · Figma Community 플러그인)이 모두 가동됐다.

## 12막 — 핸드오프에서 코드 생성으로 (v0.34.0)

`export_interactions`가 "인터랙션을 JSON으로"였다면, 다음 단계는 "인터랙션을 **코드로**"였다.

- `create_variable`(19번째 도구) — 첫 변수 **생성** 도구(찾거나-만들기, 재사용 우선).
- `generate_interaction_code`(20번째 도구) — 인터랙션 스펙으로부터 결정론적으로 React 코드
  (routes/store/hooks/README)를 생성. 화면 UI가 아니라 **인터랙션 레이어**만 만든다 — 디자인→UI
  코드 생성과 짝을 이루는 보완재.

두 도구 모두 최종 opus 리뷰와 라이브 프로브가 각각 Critical을 하나씩 잡았다. (이때 `create_variable`이
플러그인 코드를 건드렸기 때문에 Figma Community 재게시가 필요했고, 2026-06-16에 재게시했다.)

## 13막 — 멀티타깃 코드 생성: React Native + SwiftUI (현재)

코드 생성을 단일 React에서 **멀티 프레임워크**로 확장하는 작업.

- **React Native** — 공유 코어(변수 스토어 + 조건 로직)를 React와 나눠 쓰고, `@react-navigation`
  기반 네비게이터를 emit.
- **SwiftUI** — `NavigationStack` Router(ObservableObject) + 변수 스토어 + 화면별 액션 핸들러.
  Swift는 JS와 달리 **컴파일 언어**라서 새로운 종류의 제약이 생겼다: 알 수 없는 화면으로의 navigate는
  런타임 no-op가 아니라 컴파일 에러가 되므로 주석 처리하고, 핸들러 함수 이름은 유효한 Swift 식별자여야
  한다(숫자/빈 시작 방지). "생성된 코드가 실제로 컴파일된다"가 이 막의 합격선이었고, 최종 브랜치 리뷰가
  식별자 버그를 하나 더 잡아냈다.

현재 등록 도구는 **21개**, 누적 테스트는 **594개**다(초기 안전망 409개에서 꾸준히 성장).

## 반복된 교훈 (5줄 요약)

1. **라이브 API가 typings를 이긴다.** Figma의 타입 선언이 필수라 적은 필드를 런타임이 거부하고, 문서에
   없는 제약이 라이브 검증에서만 드러난다 — 항상 첫 라이브 검증에서 버릴 준비를 하고 간다.
2. **readonly·플랫폼 능력은 스펙 단계에서 확인한다.** overlay readonly와 else-if 두 번의 차단이
   준 값비싼 교훈. 코드를 다 짜고 막히지 않으려면 브레인스토밍에서 먼저 검증한다.
3. **`.describe()`가 자연어 동작의 단일 진실 공급원이다.** 기능을 안 건드리고 describe만 고쳐 LLM의
   도구 선택·인자 채움을 바꾼다.
