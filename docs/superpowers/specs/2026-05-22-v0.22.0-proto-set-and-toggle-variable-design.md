# v0.22.0 — `proto_set_variable` + `proto_toggle_variable`

Status: design (2026-05-22).
Predecessor: v0.21.1 (proto_scroll description disambig patch).
Release type: minor.
Scope: lift the existing `set_variable` / `toggle_variable` action types from the `create_reactions` escape hatch into the `proto_*` namespace as two atomic tools. No new low-level surface; both wrap existing v1.17 / v1.18 actions.

## Goal

Designer-intent natural language like "이 버튼을 누르면 showMenu 변수를 true로 만들어줘" or "다크모드 토글" now resolves to a one-call `proto_set_variable` or `proto_toggle_variable` without the LLM needing to construct a low-level `create_reactions` payload with nested action shapes.

## Decisions (from brainstorm)

- **Tool count**: 2 atomic tools — `proto_set_variable` and `proto_toggle_variable`. Same pattern as v0.21.0's back/url split. set takes (variable + value), toggle takes only (variable) — fundamentally different input shapes, no benefit from unifying.
- **Naming**: matches the low-level v1.17 action types (`set_variable` / `toggle_variable`) with the `proto_` prefix only.
- **No `motion` field**: variables change instantly in Figma's prototype runtime; the reaction's transition is visually inert for variable actions. Both schemas use `.strict()` to enforce this (same precedent as proto_url).
- **COLOR handling is automatic**: the value union `boolean | number | string` already accepts a hex string. The plugin's existing v1.18 `validateVariableLiteralCompat` reads the variable's `resolvedType` and dispatches — no schema branching at the proto_* layer.
- **toggle is BOOLEAN-only**: same constraint as v1.17 — the plugin enforces this at runtime by checking `resolvedType === "BOOLEAN"`. proto_toggle_variable passes through without redundant validation.
- **History recording**: both tools are mutating; both record via the existing `recordedHandler`. `ProtoToolName` extends from 5 to 7 entries.
- **Korean disclaimer prefix**: applied to both descriptions (same convention since v1.21).

## Section 2 — Tool signatures

### `proto_set_variable`

```ts
proto_set_variable({
  sets: [
    {
      from: string,                     // sourceNodeId
      variable: string,                 // local Figma variable NAME (plugin resolves to id)
      value: boolean | number | string, // hex string ("#RRGGBB" or "#RRGGBBAA") for COLOR variables
      trigger?: TriggerInput,           // default "ON_CLICK"
    }
  ],
  replaceExisting?: boolean,
})
```

`ProtoSetVariableEntry` is `.strict()` — rejects unknown keys including `motion`, mirroring the no-transition contract.

Compiles to (per entry):
```ts
{
  sourceNodeId: from,
  trigger: trigger ?? "ON_CLICK",
  // No `transition` field — create_reactions zod schema defaults it to "INSTANT".
  action: { type: "set_variable", variable, value },
}
```

The plugin (v1.17 logic) resolves `variable` name → id, validates `value` against the variable's `resolvedType` (BOOLEAN/FLOAT/STRING/COLOR), and writes via Figma's prototype VariableData.

For COLOR variables: pass `value` as `"#FF8800"` (RGB) or `"#FF880080"` (RGBA). The plugin parses with v1.18's `hexToRgb`. Wrong-format hex → plugin rejects with v1.18 error message.

### `proto_toggle_variable`

```ts
proto_toggle_variable({
  toggles: [
    {
      from: string,         // sourceNodeId
      variable: string,     // local Figma BOOLEAN variable NAME (plugin enforces type)
      trigger?: TriggerInput,
    }
  ],
  replaceExisting?: boolean,
})
```

`ProtoToggleVariableEntry` is `.strict()` — rejects unknown keys.

Compiles to:
```ts
{
  sourceNodeId: from,
  trigger: trigger ?? "ON_CLICK",
  // No `transition` field — defaults to INSTANT.
  action: { type: "toggle_variable", variable },
}
```

The plugin (v1.17 logic) desugars `toggle_variable` into `CONDITIONAL + 2 SET_VARIABLE` actions internally. Round-trip via `list_reactions` reconstructs the toggle pattern. Non-BOOLEAN variable → plugin rejects.

Insert 2 entries in `makeTools` AFTER `proto_url` and BEFORE `proto_get_last_history`:

```ts
{
  name: "proto_set_variable",
  description:
    "이 도구는 오직 피그마의 프로토타입 인터랙션/애니메이션 생성·수정 목적으로만 사용합니다. " +
```

## Section 5 — Testing strategy

### Unit tests on top of 268

**`tests/protoTools.test.ts`** — schema tests for the 2 new inputs (~10 cases).
- `ProtoSetVariableInput`:
  - Accepts minimal `{ sets: [{ from, variable, value: true }] }`.
  - Accepts each value type (boolean / number / string / hex string).
  - Accepts `trigger` override.
  - Rejects empty `sets`.
  - Rejects missing `variable` / `value`.
  - Rejects unknown `motion` field (.strict()).
- `ProtoToggleVariableInput`:
  - Accepts minimal `{ toggles: [{ from, variable }] }`.
  - Accepts `trigger` override.
  - Rejects empty `toggles`.
  - Rejects missing `variable`.
  - Rejects unknown keys (.strict()).

**`tests/protoTools-compile.test.ts`** — compile output snapshots (~8 cases).
- `compileProtoSetVariable`:
  - Minimal → connection with `action: { type: "set_variable", variable, value }`, no `transition` field, round-trips through `CreateReactionsInput.safeParse`.
  - Each value type passes through unchanged (boolean / number / string / hex string).
  - Batch of multiple sets compiled in order.
- `compileProtoToggleVariable`:
  - Minimal → connection with `action: { type: "toggle_variable", variable }`, no `transition`.
  - Round-trips through CreateReactionsInput.
  - Batch of multiple toggles.

**`tests/protoTools-history.test.ts`** — integration tests (~4 cases).
- proto_set_variable success → entry recorded with `tool: "proto_set_variable"`, value captured.
- proto_set_variable successCount 0 → no entry (consistency with other proto_*).
- proto_toggle_variable success → entry recorded with `tool: "proto_toggle_variable"`.
- (Optional) mixed ordering check including a variable tool.

Expected total: 268 → ~290 (+22).

## Section 7 — Out of scope (explicit YAGNI)

- No motion / transition field on either tool (per design decision).
- No reference to variables by **id** — only by NAME (consistent with v1.17 / v1.18; plugin resolves).
- No multi-action support (proto_* is 1-action-per-entry).
- No `proto_set_variable_color` specialization — single tool with type-driven value interpretation in the plugin (v1.18 path).
- No partial-batch override semantics — `replaceExisting` is per-batch as everywhere else.
- No new test infrastructure — reuse the v0.20.1 `findHandler(makeTools(store), name)` pattern.
- No tweaks to the `recordedHandler` helper or `HistoryStore` internals.
