# v1.20 — proto_* intent tools + motion presets

Status: design (2026-05-22).
Predecessor: v0.18.0 (Phase A: SSE+WS unified server).
Phase: B.

## Goal

Add a designer-intent-oriented tool surface (`proto_wire`, `proto_overlay`, `proto_scroll`) and a small set of named motion presets (Material 3 + Apple HIG), in parallel with the existing 6 low-level tools. The new surface compiles down to existing primitives; nothing in the plugin or the existing tool layer changes.

100% feature parity preserved: presets are shortcuts, not new capabilities. Anything the new surface does not cover (back/url/conditional/variables/directional transitions) remains available via the existing `create_reactions` escape hatch.

## Decisions (from brainstorm)

- **Tool shape** — three high-level intent tools (`proto_wire`, `proto_overlay`, `proto_scroll`). The existing 6 tools stay intact as the escape hatch.
- **Preset scope** — 10 presets ship in v1.20: M3 (6) + HIG (4).
- **Preset consumption layer** — presets are accepted only at the proto_* layer. `create_reactions` is not modified.

## Section 1 — Architecture & module boundaries

```
src/
├── shared/
│   └── motionPresets.ts          # NEW — preset constants + resolveMotion()
├── mcp-server/
│   ├── tools.ts                  # existing 6 schemas (unchanged)
│   └── protoTools.ts             # NEW — zod schemas for proto_wire/overlay/scroll
└── server/
    └── tools.ts                  # TOOLS array gains 3 proto_* entries
```

Three rules:
1. `motionPresets.ts` is a pure function module — `resolveMotion(motionInput) → TransitionInput`. Importable anywhere, independently testable.
2. proto_* handlers call the existing `CREATE_REACTIONS` plugin command. **No plugin-side code changes.** proto_* is a pure server-side compilation layer.
3. proto_* input zod schemas reuse sub-schemas from `mcp-server/tools.ts` (`TriggerInput`, `TransitionInput`, etc.). To make this possible, those sub-schemas must be exported (currently file-local). No duplicate definitions.

## Section 2 — proto_* tool signatures

All three tools take a batch (matching `create_reactions` `connections[]`). All three accept `motion?: MotionInput` (preset name OR existing `TransitionInput`), default `"M3_EMPHASIZED"`. All three return `{ created: N, failures: [{index, error}] }` — each entry succeeds or fails independently (same semantics as `create_reactions`).

### proto_wire (NAVIGATE only)
```ts
proto_wire({
  wires: [{
    from: string,                  // sourceNodeId
    to: string,                    // targetFrameId
    trigger?: TriggerInput,        // default "ON_CLICK"
    motion?: MotionInput,          // default "M3_EMPHASIZED"
    resetScrollPosition?: boolean,
  }],
  replaceExisting?: boolean,
})
```
Compiles each entry to a `create_reactions` connection with `action: { type: "navigate", targetFrameId: to, resetScrollPosition }`.

### proto_overlay (open / swap / close)
```ts
proto_overlay({
  overlays: [
    | { mode: "open",  from: string, overlay: string, trigger?, motion? }
    | { mode: "swap",  from: string, overlay: string, trigger?, motion? }   // only meaningful from inside an overlay
    | { mode: "close", from: string,                  trigger?, motion? }   // typically a button inside the overlay
  ],
  replaceExisting?: boolean,
})
```
Mode maps to action type:
- `"open"` → `{ type: "overlay",      targetFrameId: overlay }`
- `"swap"` → `{ type: "swap_overlay", targetFrameId: overlay }`
- `"close"` → `{ type: "close" }`

### proto_scroll (SCROLL_TO)
```ts
proto_scroll({
  scrolls: [{
    from: string,                  // sourceNodeId
    to: string,                    // targetNodeId
    trigger?: TriggerInput,
    motion?: MotionInput,
    resetScrollPosition?: boolean,
  }],
  replaceExisting?: boolean,
})
```
Compiles each entry to `action: { type: "scroll", targetNodeId: to, resetScrollPosition }`.

### Out of scope for proto_* (explicit YAGNI)
The following stay on the existing `create_reactions` escape hatch — no proto_* wrapper in v1.20:
- `back` / `url` actions
- `conditional` / `set_variable` / `toggle_variable` actions
- DirectionalTransition (MOVE_IN / MOVE_OUT / PUSH / SLIDE_IN / SLIDE_OUT) inside `motion`
- Multiple actions per reaction (proto_* is one-action-per-entry)
- `from` accepting a selector instead of nodeId (nodeId only — caller does `find_nodes` separately)
- Magical preset partial-override (e.g. `motion: { preset: "M3_EMPHASIZED", duration: 0.7 }`) — use full TransitionInput object instead

## Section 3 — Motion presets (10 total)

### M3 — Material 3, encoded as `CUSTOM_CUBIC_BEZIER` (safe path)

| Name | duration (s) | easing (x1, y1, x2, y2) | Intended use |
|---|---|---|---|
| `M3_EMPHASIZED` | 0.5 | (0.2, 0, 0, 1) | **Global default** — page/overlay transitions |
| `M3_EMPHASIZED_DECELERATE` | 0.4 | (0.05, 0.7, 0.1, 1) | Incoming (enter) |
| `M3_EMPHASIZED_ACCELERATE` | 0.2 | (0.3, 0, 0.8, 0.15) | Outgoing (exit) |
| `M3_STANDARD` | 0.3 | (0.2, 0, 0, 1) | Small components |
| `M3_STANDARD_DECELERATE` | 0.25 | (0, 0, 0, 1) | Small enter |
| `M3_STANDARD_ACCELERATE` | 0.2 | (0.3, 0, 1, 1) | Small exit |

All x1/x2 values are within Figma's `[0, 1]` bezier constraint — no runtime risk on bounds.

### HIG — Apple, mapped to Figma's built-in `NamedEasingEnum` springs

CUSTOM_SPRING is deliberately avoided (cf. the documented runtime rejection of `initialVelocity` despite typings — see comment in `mcp-server/tools.ts`). Named springs are already in the existing `EasingInputUnion` and ship-safe.

| Name | easing | Note |
|---|---|---|
| `HIG_DEFAULT` | `"GENTLE"` | Apple UISpring default tone |
| `HIG_SMOOTH` | `"SLOW"` | No bounce |
| `HIG_SNAPPY` | `"QUICK"` | Light bounce + fast |
| `HIG_BOUNCY` | `"BOUNCY"` | Significant overshoot |

For HIG entries, `duration` is **not emitted** (springs ignore duration in the Figma runtime).

### Compile shape

```ts
// src/shared/motionPresets.ts
export type PresetName =
  | "M3_EMPHASIZED" | "M3_EMPHASIZED_DECELERATE" | "M3_EMPHASIZED_ACCELERATE"
  | "M3_STANDARD"   | "M3_STANDARD_DECELERATE"   | "M3_STANDARD_ACCELERATE"
  | "HIG_DEFAULT"   | "HIG_SMOOTH" | "HIG_SNAPPY" | "HIG_BOUNCY";

export type MotionInput = PresetName | TransitionInput;

export function resolveMotion(m: MotionInput | undefined): TransitionInput {
  if (m === undefined) return resolveMotion("M3_EMPHASIZED");
  if (typeof m === "string" && isPresetName(m)) return PRESETS[m];
  return m;  // already a TransitionInput
}
```

### Default transition `type`

Presets encode `duration + easing` only — not the transition `type` (SMART_ANIMATE / DISSOLVE / INSTANT / etc.). When a preset is used, the compiled transition's `type` defaults to **`SMART_ANIMATE`**. To use a different `type`, pass a full `TransitionInput` object (no preset).

## Section 4 — Testing strategy, live verification, designer summaries

### Unit tests (on top of the existing 173)

1. `motionPresets.test.ts` — each of the 10 presets, when run through `resolveMotion()`, produces a value that passes the existing `TransitionInput` zod schema. Covers shape validity.
2. `protoTools.test.ts` — happy-path and rejection cases for the proto_wire / proto_overlay / proto_scroll zod schemas.
3. `protoTools-compile.test.ts` — snapshot test: a proto_* input compiles to exactly the expected `CreateReactionsInput` connections array.

**Mocking policy** — the plugin command path (`session.sendCommand`) is not mocked. proto_* is a pure compiler; its tests assert on the compiled object only. End-to-end coverage of the resulting Figma side-effects is already provided by the existing `create_reactions` e2e tests.

### Live verification (Phase A pattern + v1.19 bypass-probe lesson)

The bypass-probe must exercise full `tools/call` roundtrip (not just connection establishment) — three shots:

| # | Call | Verifies |
|---|---|---|
| 1 | `proto_wire` × 1 with `motion: "M3_EMPHASIZED"` | M3 bezier (default path) lands in the live runtime — most common case |
| 2 | `proto_overlay` `mode: "open"` with `motion: "HIG_SNAPPY"` | Named spring (`QUICK`) composes correctly with `SMART_ANIMATE` |
| 3 | `proto_scroll` with a custom `TransitionInput` object (no preset) | Escape path (raw transition) still works through proto_* |

After all three probes pass, re-run the v1 acceptance 5 scenarios using proto_* where applicable. If any probe surfaces a runtime quirk, fold it back into the spec before continuing — same protocol as v1.5 onward.

### Designer summary ("디자이너 요약") per plan task

## Live verification outcome (2026-05-22)

- **Probe 1 (proto_wire M3_EMPHASIZED)** — PASS. M3 cubic bezier (0.2, 0, 0, 1) lands cleanly with SMART_ANIMATE on NAVIGATE.
- **Probe 2 (proto_overlay HIG_SNAPPY)** — FAILED initially with "Reaction at index 0 was invalid", **then PASSED after fix**. Two live findings:
  - **F1 (schema bug, our side):** the defensive throw added to `resolveMotion` during Task 2 cleanup rejected valid passthrough TransitionInput strings ("INSTANT" / "DISSOLVE" / "SMART_ANIMATE"). Reverted to the original 3-line implementation; added regression tests asserting these strings round-trip unchanged.
  - **F2 (Figma runtime constraint):** Figma rejects `SMART_ANIMATE` transitions on OVERLAY / SWAP / CLOSE navigation actions. The UI also hides "Smart Animate" from the overlay transition dropdown — only Instant / Dissolve / Move / Slide are offered. Diagnostic probes confirmed all SMART_ANIMATE variants fail while every DISSOLVE variant (including DISSOLVE+named-spring "QUICK") works. Since the entire M3 preset family compiles to SMART_ANIMATE-based transitions, the proto_overlay default would always fail without a rewrite. **Fix:** `compileProtoOverlay` now runs a `rewriteForOverlay()` step over the resolved transition that swaps `type: "SMART_ANIMATE"` → `type: "DISSOLVE"` while preserving `duration` and `easing`. The designer's motion intent (M3 emphasized curve, HIG snappy spring) survives the rewrite. Documented in the proto_overlay tool description.
- **Probe 3 (proto_scroll raw `{ type: "SCROLL_ANIMATE", … }`)** — PASS. Raw TransitionInput passthrough through proto_scroll works. (Server emitted a non-fatal warning because the target frame had no scrollable ancestor — a Figma file setup concern, not a code concern.)
- **Cross-check** — `proto_wire` with `motion: "HIG_SNAPPY"` (named spring on NAVIGATE) also PASSED, confirming named springs are accepted; the rejection is purely the SMART_ANIMATE + OVERLAY combination.

The bypass-probe protocol from v1.19 paid off again: a unit-test-clean implementation (215 tests, typecheck clean) still hit two production-breaking issues that only the live Figma runtime surfaced.
