# figma-prototype-mcp v1.9 — MCP server-side bridge auto-reconnect 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 리뷰 대기)
- 기반 버전: v0.7.0
- 영향 범위: `src/mcp-server/plugin-bridge.ts`만. plugin / relay / 빌더 무관.

## 1. 목적

v0.7.0의 plugin 측은 relay 재시작에 살아남지만, **MCP server 측 bridge는 한 번 끊기면 영구 disconnect**. 사용자가 LLM에 tool을 부탁할 때 "Bridge not connected"를 보게 됨. Claude Code 재시작이 유일한 복구 — UX 비대칭.

v1.9는 plugin 측(v1.8)과 같은 패턴으로 MCP bridge에 지수 backoff 자동 reconnect 추가. 사용자가 보기에는 relay가 끊겼다 살아나는 동안 도구 호출이 잠시 기다리다가 자연스럽게 성공.

## 2. spec-단계 readonly/setter 체크

순수 WebSocket / 타이머 / Promise state machine. Figma plugin API 무관. ✅ 차단 없음.

## 3. 스코프

### v1.9 포함
- `ws.on("close", ...)`에서 의도하지 않은 close 감지 → 지수 backoff retry (1s → 2s → 4s → 8s → 16s → 30s cap)
- 의도한 종료 (`bridge.close()` 호출) → flag로 retry 억제
- 성공 join 시 backoff 1s로 리셋
- `sendCommand`: WS가 OPEN 아니면 `timeoutMs` 안에서 연결을 기다림 (= 자연스러운 자동 복구 UX). timeout 초과 시 기존 에러 throw.
- 단위 테스트 3개 추가 — reconnect-on-close / waitUntilConnected timeout / close()-prevents-reconnect

### v1.9 비-목표
- Max retry attempts
- 콜백 / 이벤트 API (외부에서 reconnect 상태 구독)
- Relay heartbeat
- README 시나리오 신설 (시나리오 12의 라이브 검증은 그대로; bridge auto-reconnect의 효과는 "MCP 호출이 갑자기 끊어진 후에도 자동 복구"로 자연스럽게 드러남)

## 4. 클래스 구조 변경

### 신규 / 변경되는 필드

```ts
class PluginBridge {
  private ws: WebSocket | null = null
  private intentionallyClosed = false           // NEW
  private retryDelay = 1000                     // NEW
  private retryTimer: NodeJS.Timeout | null = null  // NEW
  private connectionWaiters = new Set<{ resolve: () => void; reject: (err: Error) => void; timer: NodeJS.Timeout }>()  // NEW
  private joinResolved: (() => void) | null = null    // 기존
  // ...
}
```

`MAX_RETRY_DELAY` 상수: 30000 (ms).

### 메서드 변화

**`async connect()` (public, 변화 있음)**
- `this.intentionallyClosed = false`로 리셋
- 내부 `attemptConnect()` 호출
- attemptConnect 실패 시 reject — 즉 첫 connect는 즉각 실패하면 호출자에게 에러 (앱 시작 시 relay 부재면 명확히 알림)

**`private async attemptConnect()` (NEW)**
- 새 WebSocket 생성
- open / error / close 핸들러 부착
- join 메시지 보내고 join ACK 대기
- 성공 시 `retryDelay = 1000`로 리셋 + `connectionWaiters` 드레인
- 실패 시 reject (호출자 — connect 또는 scheduleReconnect)

**`private scheduleReconnect()` (NEW)**
- `intentionallyClosed`이면 no-op
- 이미 retryTimer 있으면 no-op (중복 방지)
- `setTimeout(() => { this.retryTimer = null; const next = Math.min(this.retryDelay * 2, MAX); this.attemptConnect().catch(...).then(...); this.retryDelay = next; }, this.retryDelay)`

**`ws.on("close", ...)` 변화**
- 현재: `failAllPending(new Error("WebSocket closed"))`
- 추가: `if (!intentionallyClosed) this.scheduleReconnect()`

**`close()` (변화)**
- `this.intentionallyClosed = true`
- `if (retryTimer) clearTimeout(retryTimer)`
- 기존 failAllPending + ws.close
- 추가: `connectionWaiters`에 있는 모두에게 reject

**`async sendCommand()` (변화)**
- 시작에 `await this.waitUntilConnected(this.timeoutMs)`
- 그 후 기존 로직 (UUID, pending Map, ws.send, 응답 대기)

**`private async waitUntilConnected(timeoutMs)` (NEW)**
- `ws?.readyState === OPEN`이면 즉시 resolve
- 아니면 새 Promise: connectionWaiters에 등록 + 자체 timeout. 등록된 waiter는 attemptConnect 성공 시 또는 close()에서 drain.

## 5. sendCommand 의미 변화 (사용자 시점)

기존: bridge 끊겨있으면 `"Bridge not connected"` 즉시 throw.
v1.9: bridge 끊겨있으면 timeoutMs 동안 reconnect 기다림. Timeout 내 OPEN되면 명령 실행. Timeout 초과 시 `"Bridge not connected (waited 30000ms)"`로 throw.

LLM의 tool 호출 관점에서 30s 안에 bridge가 복구되면 호출은 그냥 약간 느린 정상 응답으로 보임. 30s 초과면 명확한 에러.
