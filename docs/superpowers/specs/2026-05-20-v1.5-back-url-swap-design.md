# figma-prototype-mcp v1.5 — Back / Open URL / Swap Overlay 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 리뷰 대기)
- 기반 버전: v0.3.0
- 직전 spec: v1.3 (`docs/superpowers/specs/2026-05-20-v1.3-overlay-design.md`)
- v1.4는 BLOCKED — `docs/superpowers/specs/2026-05-20-v1.4-configure-overlay-frame-design.md`

## 1. 목적

v0.3.0의 `create_reactions`는 4개 action variant (navigate, scroll, overlay, close)를 지원한다. 자주 쓰이는 패턴 중 남은 빈자리 세 개를 한 번에 채운다: **Back**, **Open URL**, **Swap Overlay**.

## 2. spec-단계 readonly/setter 체크 (v1.4 학습 적용)

`@figma/plugin-typings/plugin-api.d.ts`에서 직접 확인:

```ts
type Action =
  | { readonly type: 'BACK' | 'CLOSE' }
  | { readonly type: 'URL'; url: string; openInNewTab?: boolean }
  | { readonly type: 'NODE'; ...; navigation: Navigation; ... }

type Navigation = 'NAVIGATE' | 'SWAP' | 'OVERLAY' | 'SCROLL_TO' | 'CHANGE_TO'
```

- 세 신규 액션 모두 `Action` union의 일급 멤버.
- Mutation 경로는 v1.3에서 이미 검증된 `node.setReactionsAsync([reactions])`. 새 setter 호출 필요 없음.
- BACK은 CLOSE와 같은 페밀리(필드 없음). URL은 `url` 필수 + `openInNewTab?` 옵셔널. SWAP은 NODE 액션의 `navigation` 값 (별도 top-level type 아님).

✅ **차단 없음**. v1.4 같은 readonly 사고 위험 없음.

## 3. 스코프

### v1.5 포함

- 세 새 액션을 `create_reactions`의 `action` discriminated union에 추가
- 사용자-노출 키워드: `"back"`, `"url"`, `"swap_overlay"`
- 빌더 함수 3개: `buildBackReaction`, `buildUrlReaction`, `buildSwapOverlayReaction`
- Plugin handler 분기 3개 추가
- 새 E2E 시나리오 9 (a/b/c)

### v1.5 비-목표

- URL의 `openInNewTab` 옵션 노출 — 항상 same-tab (default false). 필요 시 v1.6에서 추가.
- Variable 계열 액션 (SET_VARIABLE, SET_VARIABLE_MODE, CONDITIONAL) — 별도 spec.
- CHANGE_TO (component variant swap) — instance/variant 검색 인프라 필요해서 별도.
- AFTER_TIMEOUT 등 새 trigger 타입 — v1.7+ 후보.

## 4. 변경 요약

| 영역 | v0.3.0 | v1.5 |
|---|---|---|
| `action` union variant 수 | 4 | 7 (+ back, url, swap_overlay) |
| 빌더 함수 | 4 | 7 |
| Figma 내부 wire shape | NODE(navigate/scroll/overlay), CLOSE | + NODE(swap), BACK, URL |

기존 4개 액션 분기 + 인프라(트리거·트랜지션·`commandQueue`·`list_reactions` navigation 노출)는 모두 그대로. `list_reactions`는 BACK/URL/SWAP을 자동으로 노출 (action.type 또는 navigation 차이만)이므로 별도 수정 불필요.

## 5. 툴 API

### 5.1 `create_reactions` (입력)

```ts
{
  connections: [
    {
      sourceNodeId: string,
      trigger?: "ON_CLICK" | "ON_HOVER" | "ON_PRESS",
      transition?: "INSTANT" | "DISSOLVE" | "SMART_ANIMATE",
      action:
        | { type: "navigate", targetFrameId: string }
        | { type: "scroll",   targetNodeId: string }
        | { type: "overlay",  targetFrameId: string }
        | { type: "close" }
        | { type: "back" }                            // NEW
        | { type: "url",     url: string }            // NEW
        | { type: "swap_overlay", targetFrameId: string }  // NEW
    }
  ],
  replaceExisting?: boolean
}
```

Zod 표현은 기존 `z.discriminatedUnion("type", [...])`에 새 3개 object 스키마 추가.

### 5.2 transition / trigger 처리

| Action | trigger 의미 | transition 의미 |
|---|---|---|
| back | ON_CLICK 기본, ON_HOVER/ON_PRESS 가능 | 없음 (handler가 buildBackReaction에 전달 안 함) |
| url | ON_CLICK 기본 | 없음 |
| swap_overlay | overlay와 동일 | overlay와 동일 (NODE.transition) |

connection 입력 레벨에서 trigger/transition은 v1.1부터 모든 액션에 일관적으로 받는다. handler가 액션 타입별로 빌더 호출 시 필요한 것만 전달.

### 5.3 응답

변경 없음 — 기존 `results[].status` / `warning?` / aggregate counts 그대로. 새 액션 중 warning을 발생시키는 케이스는 없음 (warning은 v1.1 scroll 전용 유지).

## 6. 빌더 함수

```ts
// 새 타입 추가
export interface BackBuildInput {
  trigger: TriggerName;
}

export interface UrlBuildInput {
  trigger: TriggerName;
  url: string;
}

export interface SwapOverlayBuildInput {
  trigger: TriggerName;
  transition: TransitionName;
  targetFrameId: string;
}

export function buildBackReaction(input: BackBuildInput): BuiltReaction {
  return {
    trigger: { type: input.trigger },
    actions: [{ type: "BACK" }],
  };
}

export function buildUrlReaction(input: UrlBuildInput): BuiltReaction {
  return {
    trigger: { type: input.trigger },
    actions: [{ type: "URL", url: input.url }],
  };
}

export function buildSwapOverlayReaction(input: SwapOverlayBuildInput): BuiltReaction {
  return {
    trigger: { type: input.trigger },
    actions: [
      {
        type: "NODE",
        destinationId: input.targetFrameId,
        navigation: "SWAP",
        transition: buildTransition(input.transition),
        preserveScrollPosition: false,
      },
    ],
  };
}
```

### `BuiltAction` union 확장

```ts
export type BuiltAction =
  | {
      type: "NODE";
      destinationId: string;
      navigation: "NAVIGATE" | "SCROLL_TO" | "OVERLAY" | "SWAP";  // ← SWAP 추가
      transition: TransitionShape;
      preserveScrollPosition: false;
    }
  | { type: "CLOSE" }
  | { type: "BACK" }                     // NEW
  | { type: "URL"; url: string };        // NEW
```

(URL 액션의 `openInNewTab?`는 일단 emit하지 않음 — v1.5 minimal scope.)

## 7. Plugin handler 분기

`handleCreateReactions`의 if/else chain에 세 새 케이스 추가:

```ts
} else if (conn.action.type === "back") {
  newReaction = buildBackReaction({ trigger: conn.trigger });
} else if (conn.action.type === "url") {
  newReaction = buildUrlReaction({ trigger: conn.trigger, url: conn.action.url });
} else if (conn.action.type === "swap_overlay") {
  const target = figma.getNodeById(conn.action.targetFrameId);
  if (!target) throw new Error(`Swap overlay target frame not found: ${conn.action.targetFrameId}`);
  if (target.type !== "FRAME") {
    throw new Error(`Swap overlay target must be a frame: ${conn.action.targetFrameId} (got ${target.type})`);
  }
  newReaction = buildSwapOverlayReaction({
    trigger: conn.trigger,
    transition: conn.transition,
    targetFrameId: conn.action.targetFrameId,
  });
}
```

검증:
- **back / url**: 추가 검증 없음 (URL의 url 문자열은 Zod에서 `z.string()`으로 잡음)
- **swap_overlay**: navigate / overlay와 동일 — target은 FRAME이어야

  button `dismissThis` (back).

## 10. 라이브 API 리스크

- **`url` 문자열 형식**: Figma가 fully-qualified URL을 요구할지 (`https://...`) — 보통 그렇지만 path-only는 거부 가능. 시나리오 9(a)에서 명시적으로 `https://figma.com` 사용.
- **`SWAP` navigation의 컨텍스트 의존성**: SWAP은 "현재 overlay를 교체"라서 source가 이미 overlay 안에 있을 때만 의미. screen 본문에서 SWAP을 호출하면 Figma 응답이 어떨지 (에러? open으로 폴백?) 라이브에서 확인.
- **BACK 동일 컨텍스트 이슈**: BACK은 navigation history가 있을 때 의미. 첫 화면에서 BACK 호출 시 동작 미정 — 라이브에서 확인.
- 위 세 가지는 v1.1/v1.3 패턴 — unit test green ≠ 라이브 green. 시나리오 9에서 발견 시 정정 커밋.
