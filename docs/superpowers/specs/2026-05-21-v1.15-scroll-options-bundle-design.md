# figma-prototype-mcp v1.15 — Scroll options bundle (B + C) 설계 문서

- 작성일: 2026-05-21
- 상태: 초안 (사용자 승인 완료)
- 기반 버전: v0.13.0
- 시리즈: scroll 영역 2차 — `numberOfFixedChildren` setter + `resetScrollPosition` action 옵션. (v1.14 spec에서 "다음 release 후보"로 명시된 두 항목을 한 release로 번들.)
- 참조:
  - `node_modules/@figma/plugin-typings/plugin-api.d.ts:7725` (`numberOfFixedChildren: number`, mutable)
  - `node_modules/@figma/plugin-typings/plugin-api.d.ts:5165-5173` (NODE Action의 `resetScrollPosition?: boolean` 외 sibling 옵션들; `preserveScrollPosition`은 `@deprecated`)

## 1. 목적

v0.13.0에서 `set_frame_scroll`로 `overflowDirection`을 노출했지만, scroll 영역에서 자주 같이 등장하는 두 개념이 아직 빠져 있음:

1. **Sticky header / fixed children** — 스크롤 영역에 고정 헤더/푸터를 두는 패턴은 거의 모든 앱 prototype에서 등장. 현재 사용자가 Figma UI에서 "Fix position when scrolling" 체크박스로 직접 설정해야 함.
2. **Reset scroll position on navigate** — 화면 전환 시 도착 frame의 스크롤을 top으로 reset할지, 이전 위치를 기억할지. 앱 prototype의 탭/홈/돌아가기 흐름에서 자주 필요.

v1.15는 이 두 항목을 한 release로 묶음:
- **B**: 기존 `set_frame_scroll` tool에 entry-level optional `fixedChildren?: number` 추가
- **C**: 4 NODE action (navigate / scroll / overlay / swap_overlay)에 optional `resetScrollPosition?: boolean` 추가

## 2. Spec-단계 readonly/setter 체크

### 2.1 numberOfFixedChildren

Figma typings line 7725:

```ts
interface DefaultFrameMixin {
  ...
  /**
   * Determines which children of the frame are fixed children in a scrolling frame.
   *
   * In Figma, fixed children are always on top of scrolling (non-fixed) children.
   * ... fixed layers are not represented as a boolean property on individual layers.
   * Instead, what we really have are two sections of children inside each frame.
   */
  numberOfFixedChildren: number
  ...
}
```

`readonly` 키워드 **없음** → setter 존재. ✅ 차단 없음.

**중요 모델 quirk:** Figma는 "어느 child가 sticky인지"를 boolean으로 표시하지 않고, **frame의 상위 N개 children을 sticky로 간주**. 즉 사용자가 sticky로 만들 child를 layer panel 상단에 미리 배치하고, `numberOfFixedChildren: N`을 보내야 함. 우리 tool은 children 순서를 안 건드림.

### 2.2 resetScrollPosition

Figma typings line 5165-5173:

```ts
{
  readonly type: 'NODE'
  readonly destinationId: string | null
  readonly navigation: Navigation
  readonly transition: Transition | null
  /** @deprecated Use `resetScrollPosition` instead. */
  readonly preserveScrollPosition?: boolean
  readonly overlayRelativePosition?: Vector
  readonly resetVideoPosition?: boolean
  readonly resetScrollPosition?: boolean
  readonly resetInteractiveComponents?: boolean
}
```

`readonly`는 typings 레벨의 const-ness — 우리는 plain object를 `setReactionsAsync()`에 통째로 넘기는 방식이라 `readonly`가 차단하지 않음 (v0.1.0부터 모든 reaction 필드가 readonly인 채로 정상 작동해온 전례). ✅ 차단 없음.

**함정:** `preserveScrollPosition`은 `@deprecated`. v1.13 `deprecatedVersion` 패턴 재발 위험 — Zod 스키마에서 명시적으로 제외하고, 사용자가 보내면 reject.

**Runtime 거부 위험:** 단순 boolean optional이라 v1.12 `initialVelocity`처럼 구조적 reject 가능성 낮음. 그래도 시나리오 21 첫 호출에서 1차 확인.

## 3. 스코프

### v1.15 포함
- **B**: `SetFrameScrollEntry.fixedChildren?: number` (int ≥ 0). entry-level Zod refine: `direction`과 `fixedChildren` 중 최소 1개 필수.
- **B**: `handleSetFrameScroll` 분기 — 각 entry에서 둘 다 또는 한쪽만 적용. per-frame result에 `applied: string[]` echo.
- **C**: 4 NODE action schema (navigate / scroll / overlay / swap_overlay)에 `resetScrollPosition?: z.boolean().optional()` 추가.
- **C**: 4 builder (Navigate / Scroll / Overlay / SwapOverlay) input 타입 + emit에 conditional spread (필드 있을 때만 emit).
- **C**: Zod 스키마에서 `preserveScrollPosition` 명시적 reject (deprecated).
- 새 시나리오 20 (sticky header), 시나리오 21 (resetScrollPosition).
- Tools 개수 변화 없음 (확장만, 신규 tool 없음).

### v1.15 비-목표
- `resetVideoPosition` / `resetInteractiveComponents` — fixture(video node / interactive component) 별도 필요 + 실제 수요 미확인. 다음 release 후보.
- `overlayRelativePosition: Vector` — overlay의 trigger-relative 좌표. UX 복잡, 별개 spec 필요.
- ON_KEY_DOWN / ON_MEDIA_* live verify (v1.13 deferred 그대로 유지).

## 4. 사용자 API

### 4.1 B 입력 shape

```ts
set_frame_scroll({
  frames: [
    { frameId, direction?: "NONE"|"HORIZONTAL"|"VERTICAL"|"BOTH", fixedChildren?: number },
    ...
  ]
})
```

- `direction`과 `fixedChildren` 중 최소 1개 필수 (entry-level refine).
- 둘 다 보내면 같은 frame에 둘 다 적용.

**사용 예시:**

```ts
// vTall에 세로 스크롤 + 상단 1개 child sticky
set_frame_scroll({ frames: [{ frameId: "995:3", direction: "VERTICAL", fixedChildren: 1 }] });

// 다른 frame은 sticky만 변경 (direction 유지)
set_frame_scroll({ frames: [{ frameId: "995:3", fixedChildren: 0 }] });

// 기존 (direction-only) 호출은 그대로 작동 — backward-compat
set_frame_scroll({ frames: [{ frameId: "995:12", direction: "HORIZONTAL" }] });
```

### 4.2 B 응답 shape

```ts
{
  results: [
    { frameId, status: "success"|"error", applied?: string[], error?: string },
    // applied: 실제로 적용된 필드명 — ["direction"], ["fixedChildren"], or ["direction","fixedChildren"]
    ...
  ],
  successCount: number,
  errorCount: number
}
```

### 4.3 C 입력 shape

4 action schema 각각:

```ts
// navigate
{ type: "navigate", targetFrameId: string, resetScrollPosition?: boolean }
// scroll
{ type: "scroll", targetNodeId: string, resetScrollPosition?: boolean }
// overlay
{ type: "overlay", targetFrameId: string, resetScrollPosition?: boolean }
// swap_overlay
{ type: "swap_overlay", targetFrameId: string, resetScrollPosition?: boolean }
```

**Default 전략:** 우리 schema에 default 안 둠. 사용자 omit → emit에서도 키 omit → Figma runtime default (보통 `true` = top 초기화) 적용. 명시적 `false` → "이전 위치 유지".

### 4.4 C 사용 예시

```ts
// 명시적 false: 이전 위치 기억 ("새로 가도 스크롤 그대로")
create_reactions({
  connections: [{
    sourceNodeId: "btn:1",
    action: { type: "navigate", targetFrameId: "frm:1", resetScrollPosition: false },
  }],
});

// overlay에 적용
create_reactions({
  connections: [{
    sourceNodeId: "btn:2",
    action: { type: "overlay", targetFrameId: "ovl:1", resetScrollPosition: true },
  }],
});
```

## 5. Zod 스키마 변경

### 5.1 SetFrameScrollEntry

```ts
const SetFrameScrollEntry = z
  .object({
    frameId: z.string().min(1),
    direction: OverflowDirectionEnum.optional(),
    fixedChildren: z.number().int().min(0).optional(),
  })
  .refine(
    (v) => v.direction !== undefined || v.fixedChildren !== undefined,
    { message: "Each entry must include at least one of `direction` or `fixedChildren`" }
  );
```

`SetFrameScrollInput`은 그대로 (`z.object({ frames: z.array(SetFrameScrollEntry).min(1) })`).

### 5.2 4 NODE action schemas

```ts
const NavigateActionInput = z.object({
  type: z.literal("navigate"),
  targetFrameId: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
});
const ScrollActionInput = z.object({
  type: z.literal("scroll"),
  targetNodeId: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
});
const OverlayActionInput = z.object({
  type: z.literal("overlay"),
  targetFrameId: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
});
const SwapOverlayActionInput = z.object({
  type: z.literal("swap_overlay"),
  targetFrameId: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
});
```

`back` / `close` / `url`은 NODE action이 아니라 미적용.

### 5.3 preserveScrollPosition 차단

각 NODE action schema가 `z.object`로 정의돼 있고 우리 코드는 `z.discriminatedUnion`을 쓰지만 strict 모드는 아니라, 모르는 키는 silently dropped. 함정 차단을 위해 **Zod 스키마는 그대로 두고 builder 단계에서 명시적 무시** — input에 `preserveScrollPosition`이 들어와도 우리 emit object에는 포함하지 않음. (Zod에 deprecate 거부 로직 넣으면 schema가 비대해짐.)

## 6. Plugin handler 로직 변경

### 6.1 handleSetFrameScroll (B)

```ts
async function handleSetFrameScroll(params: {
  frames: Array<{
    frameId: string;
    direction?: "NONE" | "HORIZONTAL" | "VERTICAL" | "BOTH";
    fixedChildren?: number;
  }>;
}) {
  await figma.loadAllPagesAsync();
  const results: Array<{
    frameId: string;
    status: "success" | "error";
    applied?: string[];
    error?: string;
  }> = [];
  let successCount = 0;
  let errorCount = 0;
  for (const { frameId, direction, fixedChildren } of params.frames) {
    try {
      const node = figma.getNodeById(frameId);
      if (!node) throw new Error(`Frame not found: ${frameId}`);
      if (node.type !== "FRAME") {
        throw new Error(`Node is not a FRAME: ${node.name} (type: ${node.type})`);
      }
      const applied: string[] = [];
      if (direction !== undefined) {
        (node as FrameNode).overflowDirection = direction;
        applied.push("direction");
      }
      if (fixedChildren !== undefined) {
        (node as FrameNode).numberOfFixedChildren = fixedChildren;
        applied.push("fixedChildren");
      }
      results.push({ frameId, status: "success", applied });
      successCount++;
    } catch (e: any) {
      results.push({ frameId, status: "error", error: e?.message ?? String(e) });
      errorCount++;
    }
  }
  return { results, successCount, errorCount };
}
```

Command union의 `SET_FRAME_SCROLL.params.frames[]` 타입에 `direction?`, `fixedChildren?` 둘 다 optional로 갱신.

### 6.2 4 builder emit (C)

각 `buildNavigateReaction` / `buildScrollReaction` / `buildOverlayReaction` / `buildSwapOverlayReaction`:

- Input 타입에 `resetScrollPosition?: boolean` 추가
- emit object 만들 때 conditional spread:

```ts
const action = {
  type: "NODE",
  destinationId: input.targetFrameId,
  navigation: "NAVIGATE", // or SCROLL_TO / OVERLAY / SWAP
  transition: ...,
  ...(input.resetScrollPosition !== undefined && { resetScrollPosition: input.resetScrollPosition }),
};
```

이렇게 하면 사용자가 안 보낸 경우 키 자체가 빠짐 → Figma default 적용.

### 6.3 Plugin Command union (C 쪽 영향)

`CREATE_REACTIONS.params.connections[].action` union에서 4 NODE action 타입에 `resetScrollPosition?: boolean` 추가. (typings 정합용.)

## 7. list_reactions echo

v1.13에서 `handleListReactions`가 raw passthrough로 변경됐기 때문에 `resetScrollPosition` / `numberOfFixedChildren` 둘 다 자동으로 응답에 포함됨 (Figma가 반환한 reaction.actions[0]에 그대로 있음). 추가 코드 변경 없음.

다만 `numberOfFixedChildren`은 reaction이 아니라 frame property라 list_reactions로는 안 보임. read-side로 노출하려면 별도 tool (get_frame_overview 같은) 필요한데 v1.15 비-목표.

## 8. 새 E2E 시나리오

### 시나리오 20 — sticky header (B)

**Fixture:** 기존 `vTall` (995:3) 재사용. (현재 7개 color block — 첫 블록을 "Header" 역할 시각 단서로 활용.)

(a) Ask: "vTall 세로 스크롤로 켜고 맨 위 블록 고정해줘".
Expected: `set_frame_scroll({ frames: [{ frameId: "995:3", direction: "VERTICAL", fixedChildren: 1 }] })`. 응답: `applied: ["direction","fixedChildren"]`. Figma → Prototype Play 모드: vTall 세로 스크롤되되 v-block-1 화면 상단 고정.

(b) Ask: "vTall 고정 해제 (direction은 유지)".
Expected: `{ frameId: "995:3", fixedChildren: 0 }` (direction omit). 응답: `applied: ["fixedChildren"]`. Figma read-back: `overflowDirection`은 여전히 "VERTICAL", `numberOfFixedChildren`만 0으로.

(c) Error case (Zod): `fixedChildren: -1`. Expected: Zod parse fail → tool 응답에 "Invalid input".

(d) Edge case (runtime): `fixedChildren: 999` (children 수 초과). Expected: Figma 반응 1차 확인 — runtime이 reject할지 silently clamp할지 미지. 결과대로 README에 기록.

(e) Refine: 빈 entry (`{ frameId }`만). Expected: Zod refine fail → "at least one of `direction` or `fixedChildren`".

### 시나리오 21 — resetScrollPosition (C)

**Fixture:**
- 기존 `vTall` (995:3) — scrollable frame, 세로 스크롤로 설정 (20(a) 후속이라 이미 VERTICAL + sticky 상태).
- 새 frame `home` (큰 frame, vTall로 navigate하는 button `goVTall` 포함).
- vTall 안에 button `goHome` (home으로 navigate).
- (필요시 사용자에게 새 fixture 요청.)

(a) Ask: "goHome → home으로 가되 vTall 스크롤 위치 기억해줘". Expected: action `{ type: "navigate", targetFrameId: home, resetScrollPosition: false }`. Play: vTall 스크롤 → goHome 클릭 → home → 다시 vTall 들어오면 이전 스크롤 위치 유지.

(b) Ask: "goHome 같은 기능인데 이번엔 스크롤 top으로 초기화". Expected: `resetScrollPosition: true`. Play: 다시 vTall 들어오면 top.

(c) `list_reactions` on `goHome`. Expected: action.resetScrollPosition 필드 응답에 포함 (raw passthrough).

(d) Overlay variant: 별도 button + overlay frame fixture. Ask: "이 overlay 열 때 스크롤 초기화 안 함". Expected: `{ type: "overlay", ..., resetScrollPosition: false }`. (단, fixture 사용자 의존 — 기존 시나리오 7 fixture 활용 가능.)

(e) Deprecate 함정 차단 검증: 사용자가 일부러 `preserveScrollPosition: true`를 input에 넣을 경우 → 우리 emit object에는 없어야 함. Figma runtime에 도달하지 않음. (스크립트로 직접 확인.)

## 10. 라이브 리스크

1. **Sticky N-from-top 모델 혼동** — README에 굵게 명시 + 시나리오 20 setup에서 "header가 layer panel 최상단" 표기.
2. **fixedChildren OOR 동작 미지** — 시나리오 20(d)에서 1차 확인. runtime reject면 try/catch에서 자연스럽게 error path로 떨어짐.
3. **preserveScrollPosition deprecated 함정** — Zod 차단 안 하고 builder 명시적 무시 (passive defense). 시나리오 21(e)로 검증.
4. **resetScrollPosition runtime 거부 가능성** — 낮음 (단순 boolean). 시나리오 21(a) 첫 호출로 1차 확인.
5. **direction-only call backward-compat** — Zod refine으로 "최소 1개" 조건 추가하면 기존 direction-only 호출은 여전히 통과 (direction이 있음). ✅
6. **list_reactions raw passthrough 보장** — v1.13 이후 fix됐고 v1.14에서도 이상 없었음. 시나리오 21(c)로 1회 재확인.
