- **Release shape:** single v0.20.1 patch bump, all four items in one release. No separate patches.
- **No new functionality**: no new tools, no behavior changes for callers. Pure refactor + docs.

## Section 2 — `recordedHandler` helper

Add to `src/server/tools.ts` near the top of the file (after the imports, alongside type aliases):

```ts
async function recordedHandler<S>(
  store: HistoryStore,
  tool: ProtoToolName,
  parsedInput: S,
  send: () => Promise<unknown>,
): Promise<unknown> {
  const result = await send();
  store.record(tool, parsedInput, summarizeResult(result));
  return result;
}
```

Each proto_* handler shrinks from a 4-line body to ~3 lines:

```ts
// before (proto_wire)
handler: async (input, session) => {
  const parsedInput = input as ProtoWireInput;
  const compiled = compileProtoWire(parsedInput);
  const result = await session.sendCommand("CREATE_REACTIONS" as CommandName, compiled);
  historyStore.record("proto_wire", parsedInput, summarizeResult(result));
  return result;
},

// after
handler: async (input, session) => {
  const parsedInput = input as ProtoWireInput;
  return recordedHandler(historyStore, "proto_wire", parsedInput, () =>
    session.sendCommand("CREATE_REACTIONS" as CommandName, compileProtoWire(parsedInput)),
  );
},
```

Same shape applied to proto_overlay and proto_scroll. The closure carries the compile + send sequence; the helper owns the record + return after.

`ProtoToolName` import is added from `./history.js` (already exported there).

No direct unit test for `recordedHandler` — the existing 7 integration tests in `tests/protoTools-history.test.ts` cover its behavior through the real handlers.

## Section 3 — `makeTools(historyStore)` factory

Currently the TOOLS array is created inside `registerToolHandlers` (since v1.21 Task 4). Move that creation into a small factory function:

```ts
export function makeTools(historyStore: HistoryStore): ToolEntry[] {
  return [
    // existing 10 entries — unchanged structure, but each proto_* handler
    // uses `recordedHandler(historyStore, ...)` internally
  ];
}
```

And:

```ts
export function registerToolHandlers(
  mcp: Server,
  session: PluginSession,
  historyStore: HistoryStore,
): void {
  const TOOLS = makeTools(historyStore);
  // dispatcher body unchanged
}
```

`ToolEntry` and `makeTools` get exported so tests can import and exercise the actual handlers. The handler closures still capture `historyStore`, but `session` arrives as a handler parameter at invocation (per existing `ToolEntry` shape).

**Test rewrite** (`tests/protoTools-history.test.ts`):

The new pattern (per describe block):
```ts
import { makeTools } from "../src/server/tools.js";
import { HistoryStore } from "../src/server/history.js";
import { vi } from "vitest";

function findHandler(tools: ReturnType<typeof makeTools>, name: string) {
  const t = tools.find((x) => x.name === name);
  if (!t || !t.handler) throw new Error(`no handler for ${name}`);
  return t.handler;
}

function makeStubSession(response: unknown) {
  return { sendCommand: vi.fn().mockResolvedValue(response) } as unknown as PluginSession;
}

it("records one entry with tool='proto_wire'", async () => {
  const store = new HistoryStore();
  const tools = makeTools(store);
  const handler = findHandler(tools, "proto_wire");
  const session = makeStubSession({ successCount: 1, errorCount: 0, warningCount: 0, results: [] });
  await handler({ wires: [{ from: "1:1", to: "1:2" }], replaceExisting: false }, session);
  expect(store.size()).toBe(1);
  // ...
});
```

The 7 tests carry over with identical assertions, but they now exercise the production handler code path (compile → send → record → return) rather than a parallel inline copy. Test count stays at 246.

## Section 4 — Server version from `package.json`

`src/server/index.ts`:

```ts
import { readFileSync } from "node:fs";

const pkg = JSON.parse(
  readFileSync(new URL("../../package.json", import.meta.url), "utf8"),
) as { version: string };

const mcp = new Server(
  { name: "figma-prototype-mcp", version: pkg.version },
  { capabilities: { tools: {} } },
);
```

ESM-portable (uses `import.meta.url` instead of CJS `__dirname`). Node ≥18 supports `new URL(..., import.meta.url)` and `node:fs` namespace.

`as { version: string }` cast is acceptable — package.json is repo-owned, not external input.

Side effect: server startup now does one synchronous file read at boot. Cost is microseconds, runs once per process. No runtime hot-path impact.

## Section 5 — README Manual E2E checklist updates

The README currently has 25 scenarios for low-level tools. Append four new scenarios covering the v1.20 + v1.21 surface — each already live-verified during their respective releases (see the live verification appendices in the v1.20 and v1.21 design docs):

- **Scenario 26 — proto_wire (basic + M3_EMPHASIZED default)**
  Call `proto_wire` with one `{ from, to }` pair, no motion. Inspect Figma Prototype panel: On Click → Navigate To → Smart Animate → cubic-bezier (0.2, 0, 0, 1) at 500ms. (Live-verified 2026-05-22.)

- **Scenario 27 — proto_overlay (open + HIG_SNAPPY rewrite)**
  Call `proto_overlay` mode:"open" with `motion: "HIG_SNAPPY"`. Inspect Prototype panel: On Click → Open Overlay → Dissolve + named spring QUICK (NOT Smart Animate — Figma constraint). (Live-verified 2026-05-22, surfaced during v1.20 bypass-probe.)

- **Scenario 28 — proto_scroll (raw TransitionInput escape path)**
  Call `proto_scroll` with `motion: { type: "SCROLL_ANIMATE", duration: 0.8, easing: "EASE_IN_AND_OUT" }`. Inspect Prototype panel: On Click → Scroll To → Animate → Ease In and Out → 800ms. (Live-verified 2026-05-22.)

- **Scenario 29 — proto_get_last_history (3-shot: read / order / FIFO)**
  After running scenarios 26-28, call `proto_get_last_history({ count: 3 })`. Expected: 3 entries, oldest first. Then run 11 sequential `proto_wire` calls + `proto_get_last_history({ count: 10 })`: 10 entries, timestamps ascending. (Live-verified 2026-05-22 via v1.21 bypass-probe.)

These additions are documentation-only — the actual behaviors are already verified. The README change just records the new manual coverage points so a contributor knows what to test on a fresh setup.
