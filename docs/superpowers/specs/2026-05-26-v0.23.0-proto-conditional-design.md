# v0.23.0 — `proto_conditional`

Status: design (2026-05-26).
Predecessor: v0.22.0 (proto_set_variable + proto_toggle_variable).
Release type: minor.
Scope: lift the existing `conditional` action type from the `create_reactions` escape hatch into the `proto_*` namespace as one atomic tool. No new low-level surface; wraps the existing v1.16 CONDITIONAL action. This completes the proto_* high-level surface — all v1.x action types are now expressible without falling back to low-level `create_reactions`.

## Goal

Designer-intent natural language like "showMenu이 true면 메뉴 닫고, false면 오버레이로 열어줘" or "loggedIn 변수가 true면 홈으로, 아니면 로그인으로" resolves to a one-call `proto_conditional` without the LLM needing to construct a low-level `create_reactions` payload with nested `then`/`else` action arrays and explicit `operator` strings.

## Decisions (from brainstorm)

- **Tool count**: 1 atomic tool — `proto_conditional`. Conditional is one Figma action type with a coherent input shape; no benefit from splitting.
- **Naming**: matches the low-level v1.16 action type (`conditional`) with the `proto_` prefix only.
- **Single-action branches only**: `then`/`else` each take ONE branch action via the sugar discriminated union. Multi-action branches remain available via low-level `create_reactions` (escape hatch). Rationale: 99% of conditional use cases are single-action; the multi-action surface adds zod union complexity for negligible LLM-ergonomic benefit.
- **`if` shape with default operator `==`**: condition omits `operator` defaulting to equality (the dominant case). Other operators (`!=`, `<`, `<=`, `>`, `>=`) require explicit `operator` field. Same `boolean | number | string` value union as the low-level surface.
- **Field naming**: top-level uses `if`/`then`/`else` (natural conditional vocabulary). Low-level uses `condition`/`then`/`else`; this divergence is deliberate sugar.
- **Motion at entry top-level, not per-branch**: the low-level plugin code already passes a single `conn.transition` to all branch actions ([src/figma-plugin/code.ts:615](../../../src/figma-plugin/code.ts#L615)). The Figma runtime stores transition at the reaction level, not per-block. So `motion` belongs on the conditional entry, not on each branch sugar.
- **Branch sugar covers all 8 non-conditional actions**: navigate, scroll, overlay, swap, close, back, url, set. Same set as the low-level `NonConditionalActionInput`. `toggle_variable` is excluded (consistent with low-level — `toggle_variable` itself desugars to `CONDITIONAL` under the hood, so nesting is meaningless).
- **Overlay/swap → DISSOLVE rewrite**: if either `then` or `else` is `{ overlay: ... }` or `{ swap: ... }`, apply the same SMART_ANIMATE → DISSOLVE rewrite as `proto_overlay` (Figma's overlay transition constraint). Preserves duration/easing.
- **`.strict()` on every branch sugar entry**: prevents LLM from accidentally adding unknown keys like `motion`/`transition` to a branch (those don't apply per-branch — only at entry level).
- **History recording**: mutating tool; recorded via existing `recordedHandler`. `ProtoToolName` extends from 7 to 8 entries.
- **Korean disclaimer prefix**: applied to the description (same convention since v1.21).

## Section 2 — Tool signature

```ts
proto_conditional({
  conditions: [{
    from: string,                                     // source node id
    trigger?: TriggerInput,                           // proto_* 표준 (default ON_CLICK)
    motion?: MotionInput,                             // entry-level transition (proto_* 표준)
    if: {
      variable: string,                               // 이름 — plugin resolves to id (v1.16 path)
      operator?: "==" | "!=" | "<" | "<=" | ">" | ">=",  // default "=="
      value: boolean | number | string,
    },
    then: BranchAction,                               // single branch action
    else?: BranchAction,                              // optional single branch action
  }],
  replaceExisting?: boolean,
})

// BranchAction: discriminated union by single sugar key, each .strict()
type BranchAction =
  | { navigate: string;     resetScrollPosition?: boolean }
  | { scroll: string;       resetScrollPosition?: boolean }
  | { overlay: string }
  | { swap: string }
  | { close: true }
  | { back: true }
  | { url: string; openInNewTab?: boolean }
  | { set: { variable: string; value: boolean | number | string } }
```

### Example calls

```ts
// BOOLEAN if/then/else
proto_conditional({
  conditions: [{
    from: "1073:30",
    if: { variable: "showMenu", value: true },        // operator defaults to "=="
    then: { close: true },
    else: { overlay: "menuFrame:42" },
  }],
})

// FLOAT comparison with explicit operator + motion
proto_conditional({
  conditions: [{
    from: "1073:31",
    motion: "M3_STANDARD",
    if: { variable: "step", operator: ">=", value: 2 },
    then: { navigate: "review:100" },
    else: { navigate: "form:50" },
  }],
})

// STRING equality, then-only (no else)
proto_conditional({
  conditions: [{
    from: "1073:32",
    if: { variable: "userRole", value: "admin" },
    then: { url: "https://admin.example.com", openInNewTab: true },
  }],
})
```

## Section 3 — Compile path

```ts
import { TriggerInput, TransitionInput } from "./tools.js";
import { PRESET_NAMES, resolveMotion } from "../shared/motionPresets.js";
import type { MotionInput } from "../shared/motionPresets.js";
import type { CreateReactionsInput as CreateReactionsInputType } from "./tools.js";

const PresetNameEnum = z.enum(PRESET_NAMES);
const MotionInputSchema = z.union([PresetNameEnum, TransitionInput]);

const ComparisonOperator = z.enum(["==", "!=", "<", "<=", ">", ">="]);

const ProtoConditionIf = z.object({
  variable: z.string().min(1),
  operator: ComparisonOperator.default("=="),
  value: z.union([z.boolean(), z.number(), z.string()]),
}).strict();

const BranchNavigate = z.object({
  navigate: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
}).strict();
const BranchScroll = z.object({
  scroll: z.string().min(1),
  resetScrollPosition: z.boolean().optional(),
}).strict();
const BranchOverlay = z.object({ overlay: z.string().min(1) }).strict();
const BranchSwap = z.object({ swap: z.string().min(1) }).strict();
const BranchClose = z.object({ close: z.literal(true) }).strict();
const BranchBack = z.object({ back: z.literal(true) }).strict();
const BranchUrl = z.object({
  url: z.string().min(1),
  openInNewTab: z.boolean().optional(),
}).strict();
const BranchSet = z.object({
  set: z.object({
    variable: z.string().min(1),
    value: z.union([z.boolean(), z.number(), z.string()]),
  }).strict(),
}).strict();

const BranchAction = z.union([
  BranchNavigate, BranchScroll, BranchOverlay, BranchSwap,
  BranchClose, BranchBack, BranchUrl, BranchSet,
]);

const ProtoConditionalEntry = z.object({
  from: z.string().min(1),
  trigger: TriggerInput.optional(),
  motion: MotionInputSchema.optional(),
  if: ProtoConditionIf,
  then: BranchAction,
  else: BranchAction.optional(),
}).strict();

export const ProtoConditionalInput = z.object({
  conditions: z.array(ProtoConditionalEntry).min(1),
  replaceExisting: z.boolean().default(false),
});

export type ProtoConditionalInput = z.infer<typeof ProtoConditionalInput>;

type Connection = CreateReactionsInputType["connections"][number];
type NonConditionalAction = Extract<Connection["action"],
  { type: "navigate" | "scroll" | "overlay" | "close" | "back" | "url" | "swap_overlay" | "set_variable" }>;

function compileBranchAction(b: z.infer<typeof BranchAction>): NonConditionalAction {
  if ("navigate" in b) return { type: "navigate", targetFrameId: b.navigate,
    ...(b.resetScrollPosition !== undefined && { resetScrollPosition: b.resetScrollPosition }) };
  if ("scroll" in b)   return { type: "scroll", targetNodeId: b.scroll,
    ...(b.resetScrollPosition !== undefined && { resetScrollPosition: b.resetScrollPosition }) };
  if ("overlay" in b)  return { type: "overlay", targetFrameId: b.overlay };
  if ("swap" in b)     return { type: "swap_overlay", targetFrameId: b.swap };
  if ("close" in b)    return { type: "close" };
  if ("back" in b)     return { type: "back" };
  if ("url" in b)      return { type: "url", url: b.url,
    openInNewTab: b.openInNewTab ?? false };
  if ("set" in b)      return { type: "set_variable", variable: b.set.variable, value: b.set.value };
  throw new Error("unreachable: zod parse guarantees coverage");
}

function branchUsesOverlayOrSwap(b: z.infer<typeof BranchAction>): boolean {
  return "overlay" in b || "swap" in b;
}

export function compileProtoConditional(input: ProtoConditionalInput): CreateReactionsInputType {
  const connections: Connection[] = input.conditions.map((c) => {
    const baseTransition = resolveMotion(c.motion);
    const needsOverlayRewrite =
      branchUsesOverlayOrSwap(c.then) ||
      (c.else !== undefined && branchUsesOverlayOrSwap(c.else));
    const transition = needsOverlayRewrite ? rewriteForOverlay(baseTransition) : baseTransition;

    const action: Connection["action"] = {
      type: "conditional",
      condition: {
        variable: c.if.variable,
        operator: c.if.operator,        // zod already applied default "=="
        value: c.if.value,
      },
      then: [compileBranchAction(c.then)],
      ...(c.else !== undefined && { else: [compileBranchAction(c.else)] }),
    };

    return {
      sourceNodeId: c.from,
      trigger: c.trigger ?? DEFAULT_TRIGGER,
      transition,
      action,
    } as Connection;
  });
  return { connections, replaceExisting: input.replaceExisting };
}
```

**Reuses**: `resolveMotion`, `rewriteForOverlay`, `DEFAULT_TRIGGER` from existing protoTools.ts. No new utilities needed.

## Section 4 — Description (LLM steering)

```
디스크-기반 프로토타입 조건부 분기. 변수 비교 결과에 따라 then/else 둘 중 하나의 액션 실행.

Wire a conditional reaction (if/then/else) on a source node based on a 
variable comparison. The variable is referenced by name; the plugin 
resolves it to the local variable id at runtime.

- `if.operator` defaults to "==" if omitted (the most common case).
- `then` and `else` each take exactly ONE branch action (single sugar 
  entry). For multi-action branches, use the low-level `create_reactions` 
  escape hatch.
- Branch sugar keys: navigate / scroll / overlay / swap / close / back / 
  url / set. `toggle_variable` is not available inside conditional 
  (toggle itself desugars to CONDITIONAL — nesting is meaningless).
- Overlay/swap branches: if either branch is { overlay } or { swap }, 
  SMART_ANIMATE auto-rewrites to DISSOLVE (Figma's overlay transition 
  constraint). The motion intent (duration/easing) is preserved.
- Variable type ↔ value must match (BOOLEAN/FLOAT/STRING). COLOR 
  variables are NOT comparable in v1.x — use set_variable with a hex 
  string for COLOR mutation; conditional on COLOR will throw.
- trigger / motion apply at the conditional reaction level (shared 
  across both branches). Branch sugars do NOT accept motion/trigger 
  (.strict() rejects).
```

## Section 7 — Live verification (probes)

Run **after** all unit tests green and TypeScript clean. Single test file with sequential 2-probe verification:

### Probe 1 — BOOLEAN if/then/else

```ts
// Setup: a button + a menu overlay frame + a BOOLEAN variable `showMenu`
proto_conditional({
  conditions: [{
    from: "<buttonId>",
    if: { variable: "showMenu", value: true },
    then: { close: true },
    else: { overlay: "<menuFrameId>" },
  }],
})
```

**Expect:**
- Figma's reaction panel: `On Click → If showMenu == true → Close, Else → Open Overlay <menuFrame>`.
- `list_reactions(<buttonId>)` round-trips to a CONDITIONAL decoded shape with both `then` and `else` arrays populated.
- Prototype mode: clicking the button toggles between closed/open overlay based on current `showMenu` value.

### Probe 2 — FLOAT operator + motion + overlay rewrite

```ts
proto_conditional({
  conditions: [{
    from: "<button2Id>",
    motion: "SMART_ANIMATE",                    // would normally apply...
    if: { variable: "step", operator: ">=", value: 2 },
    then: { navigate: "<reviewFrameId>" },
    else: { overlay: "<formOverlayId>" },       // ...but else has overlay
  }],
})
```

**Expect:**
- Reaction's transition is DISSOLVE (not SMART_ANIMATE) — overlay rewrite applied.
- `if.operator` reads `>=` in Figma panel (not the default `==`).
- `step` is a FLOAT variable; comparison with literal `2` passes type check.

### Bypass-probe decision

**Not needed.** proto_conditional is a pure compile-layer change — it produces the same low-level `create_reactions` payload that v1.16 already live-verified. The plugin code path is identical. The risk surface is in our compile function (covered by unit tests + 2 integration probes), not in any new Figma API call.

Reference: v0.21.0 and v0.22.0 both shipped 0 live-API surprises with this same "compile-only, no bypass-probe" rationale.

## Section 9 — Non-goals (v0.23.0)

- Multi-action branches in `then` / `else`. Use low-level `create_reactions` if needed.
- Per-branch motion / trigger. Low-level plugin path doesn't support it either; would require plugin refactor.
- AND/OR compound conditions. Single comparison only (matches v1.16 low-level).
- Nested conditionals. Schema explicitly rejects (matches v1.16 low-level constraint).
- COLOR variable comparison. Plugin throws at runtime (v1.18 enforcement). Not changing here.

## Section 10 — Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Branch sugar union ambiguity (zod) | Low | All 8 entries `.strict()`; `then: {}` correctly rejects with no discriminator. Unit test covers. |
| Overlay rewrite missed when else has overlay | Med | Compile path checks both branches; unit test covers `else: { overlay }` case explicitly. |
| `if.operator` default not applied | Low | zod `.default("==")` applies at parse, not at compile. Unit test asserts emitted value `==` when omitted. |
| LLM tries to put motion inside `then: { navigate, motion }` | Med | `.strict()` on every branch sugar rejects extra keys. Description Section 4 explicitly states "Branch sugars do NOT accept motion". |
| Live surprise (Figma runtime behavior) | Low | No new low-level surface; pure compile layer over v1.16. 2 consecutive 0-surprise releases at this layer. |
