## 1. 목적

v0.14.0 기준 모든 reaction action은 무조건 실행 (Navigate / Scroll / Overlay / Close / Back / URL / Swap Overlay). Figma는 상태 의존 액션을 위해 `CONDITIONAL` action을 노출하지만, 우리 도구는 미지원. 결과: 디자이너가 "로그인 됐으면 home, 아니면 login" 같은 자연스러운 분기를 자연어로 요청할 수 없음.

v1.16은 새 `conditional` action type을 노출. **MVP 스코프**: 단일 비교 조건 + IF/ELSE 분기 + 분기 안 action은 기존 7가지. AND·OR·NOT·nested conditional·SET_VARIABLE은 후속 release.

## 2. Spec-단계 readonly/runtime 체크

Figma typings:

```ts
type ExpressionFunction =
  | 'ADDITION' | 'SUBTRACTION' | 'MULTIPLICATION' | 'DIVISION'
  | 'EQUALS' | 'NOT_EQUAL'
  | 'LESS_THAN' | 'LESS_THAN_OR_EQUAL' | 'GREATER_THAN' | 'GREATER_THAN_OR_EQUAL'
  | 'AND' | 'OR' | 'VAR_MODE_LOOKUP' | 'NEGATE' | 'NOT'

interface Expression {
  expressionFunction: ExpressionFunction
  expressionArguments: VariableData[]
}

type VariableDataType = 'BOOLEAN' | 'FLOAT' | 'STRING' | 'VARIABLE_ALIAS' | 'COLOR' | 'EXPRESSION'

interface VariableData {
  type?: VariableDataType
  resolvedType?: VariableResolvedDataType
  value?: VariableValue | Expression
}

type ConditionalBlock = {
  condition?: VariableData    // 없으면 ELSE 분기
  actions: Action[]
}

// Action union variant:
| { readonly type: 'CONDITIONAL'; readonly conditionalBlocks: ConditionalBlock[] }
```

`readonly` 키워드는 declaration-level — 우리는 plain object를 `setReactionsAsync`에 통째로 넘기는 패턴이라 차단 없음 (v0.1.0부터 모든 reaction 필드가 readonly인 채로 정상 작동). ✅

**Runtime 거부 위험:** typings에서 명시된 shape이지만 v1.15 `resetScrollPosition` 경우처럼 **wiring 단계에서 silent drop**할 가능성 있음. **Spec-stage bypass-probe 의무**: 첫 task 시작 전 use_figma로 raw setReactionsAsync 호출해서 `CONDITIONAL` shape이 실제로 persist되는지 1차 확인 후 진행.

## 3. 스코프

### v1.16 포함

- 새 action type `"conditional"` (소문자, 우리 API 컨벤션 — 내부에서 Figma의 `'CONDITIONAL'` 대문자로 변환).
- 단일 condition 노출 — `{ variable: string, operator: "==" | "!=" | "<" | "<=" | ">" | ">=", value: boolean | number | string }`.
- `then: Action[]` (필수, 최소 1개) + `else?: Action[]` (옵션).
- 분기 안 action은 기존 7가지: navigate / scroll / overlay / close / back / url / swap_overlay. **Nested `conditional` 금지** (Zod에서 차단).
- 변수 이름 → id 해석을 plugin handler에서 수행 (`figma.variables.getLocalVariablesAsync()`).
- list_reactions echo: conditional action을 우리 친화적 shape으로 변환해서 응답 (raw conditionalBlocks 그대로는 디버그에 도움 안 됨).
- 새 시나리오 22 (conditional navigate).

### v1.16 비-목표

- AND / OR / NOT 조합 (nested expression). 사용자가 명시적으로 "제일 간단한거부터"를 선택. v1.17+ 후보.
- ADDITION/SUBTRACTION/MULTIPLICATION/DIVISION (산술), VAR_MODE_LOOKUP, NEGATE.
- SET_VARIABLE action (토글 패턴의 핵심). 별개 release로.
- Nested conditional (then 안에 또 conditional).
- 라이브러리 변수 (다른 파일에서 import한 variable). 로컬 variable만.
- COLOR type variable 비교.

## 4. 사용자 API

### 4.1 입력 shape

```ts
{
  sourceNodeId: string,
  trigger: ...,
  transition?: ...,  // 무시됨 — conditional은 transition 없음
  action: {
    type: "conditional",
    condition: {
      variable: string,                            // local variable name
      operator: "==" | "!=" | "<" | "<=" | ">" | ">=",
      value: boolean | number | string,            // literal
    },
    then: ActionWithoutConditional[],              // 최소 1개
    else?: ActionWithoutConditional[],             // 옵션
  }
}
```

`ActionWithoutConditional`은 기존 7가지 action shape (navigate / scroll / overlay / close / back / url / swap_overlay) — `conditional`은 제외하여 nesting 차단.

### 4.2 응답 shape (list_reactions echo)

```ts
// list_reactions가 conditional reaction을 echo할 때:
{
  type: "CONDITIONAL",
  condition: {                                     // 우리 친화적 형태로 디코딩
    variable: string,                              // 이름 (read-back 시 id → name 역변환)
    operator: "==" | "!=" | "<" | "<=" | ">" | ">=",
    value: boolean | number | string,
  },
  then: ActionEcho[],                              // 기존 echo shape 재사용
  else?: ActionEcho[],
}
```

읽기 측에서 `ConditionalBlock[]`을 우리 IF/ELSE 모양으로 역변환:
- `conditionalBlocks[0]` (condition 있음) → 그 condition + actions → `condition` + `then`.
- `conditionalBlocks[1]` (condition 없음) → `else`.
- 그 외 경우 (>2 blocks 또는 비표준 패턴): raw `conditionalBlocks` 그대로 응답 + warning.

### 4.3 사용 예시 (자연어 → emit)

```ts
// "로그인 됐으면 home, 아니면 login"
{ type: "conditional",
  condition: { variable: "loggedIn", operator: "==", value: true },
  then: [{ type: "navigate", targetFrameId: "home_id" }],
  else: [{ type: "navigate", targetFrameId: "login_id" }],
}

// "포인트 100 이상이면 VIP 화면" (else 생략 → 조건 false면 아무것도 안 함)
{ type: "conditional",
  condition: { variable: "points", operator: ">=", value: 100 },
  then: [{ type: "navigate", targetFrameId: "vip_id" }],
}

// "tier가 'gold'면 골드 페이지로"
{ type: "conditional",
  condition: { variable: "tier", operator: "==", value: "gold" },
  then: [{ type: "navigate", targetFrameId: "gold_id" }],
}
```

## 6. Plugin handler 변경

### 6.1 변수 이름 → id 해석 + Expression 빌드

새 helper 함수 `buildConditionalAction` (또는 plugin handler 내부 헬퍼):

```ts
async function resolveVariableByName(name: string) {
  const all = await figma.variables.getLocalVariablesAsync();
  const matches = all.filter(v => v.name === name);
  if (matches.length === 0) throw new Error(`Variable not found: ${name}`);
  const picked = matches[0];
  const warning = matches.length > 1
    ? `Multiple variables named "${name}" found (${matches.length}); using the first`
    : undefined;
  return { variable: picked, warning };
}

function literalToVariableData(value: boolean | number | string, resolvedType: VariableResolvedDataType): VariableData {
  if (typeof value === "boolean") {
    return { type: "BOOLEAN", resolvedType: "BOOLEAN", value };
  }
  if (typeof value === "number") {
    return { type: "FLOAT", resolvedType: "FLOAT", value };
  }
  return { type: "STRING", resolvedType: "STRING", value };
}

const OPERATOR_MAP: Record<string, ExpressionFunction> = {
  "==": "EQUALS",
  "!=": "NOT_EQUAL",
  "<":  "LESS_THAN",
  "<=": "LESS_THAN_OR_EQUAL",
  ">":  "GREATER_THAN",
  ">=": "GREATER_THAN_OR_EQUAL",
};

async function buildCondition(input: { variable: string; operator: string; value: boolean|number|string }) {
  const { variable, warning } = await resolveVariableByName(input.variable);
  const literalVD = literalToVariableData(input.value, variable.resolvedType);
  const conditionVD: VariableData = {
    type: "EXPRESSION",
    resolvedType: "BOOLEAN",
    value: {
      expressionFunction: OPERATOR_MAP[input.operator],
      expressionArguments: [
        { type: "VARIABLE_ALIAS", resolvedType: variable.resolvedType,
          value: { type: "VARIABLE_ALIAS", id: variable.id } },
        literalVD,
      ],
    } as any, // typings: VariableValueWithExpression — Expression 분기
  };
  return { conditionVD, warning };
}
```

**타입 호환성 검증:** literal value의 JS 타입 ↔ Figma variable resolvedType.
- `BOOLEAN` 변수 ↔ `boolean` value
- `FLOAT` 변수 ↔ `number` value
- `STRING` 변수 ↔ `string` value
- 그 외 조합 → 명확한 error: `"Variable '${name}' is ${resolvedType}; cannot compare against ${typeof value}"`.

### 6.2 handleCreateReactions의 conditional 분기

기존 if/else 체인 (navigate/scroll/overlay/close/back/url/swap_overlay) 끝에 추가:

```ts
} else if (conn.action.type === "conditional") {
  const { conditionVD, warning: condWarning } = await buildCondition(conn.action.condition);
  if (condWarning) warning = condWarning;

  // 변수 type vs literal type 호환성 검증
  // (위 buildCondition 안에 통합 가능; 명확한 별도 함수로 유지)

  const thenActions = await Promise.all(
    conn.action.then.map(a => buildNonConditionalActionForBlock(a))
  );
  const elseActions = conn.action.else
    ? await Promise.all(conn.action.else.map(a => buildNonConditionalActionForBlock(a)))
    : undefined;

  newReaction = buildConditionalReaction({
    trigger: conn.trigger,
    afterTimeoutSeconds: conn.afterTimeoutSeconds,
    condition: conditionVD,
    thenActions,
    elseActions,
  });
}
```

여기서 `buildNonConditionalActionForBlock`은 기존 navigate/scroll/overlay/swap_overlay branch들에서 target frame 존재 검증을 수행하고 BuiltAction 1개를 반환하는 추출 헬퍼. 기존 코드의 중복을 줄임.

### 6.3 builder — buildConditionalReaction

```ts
export interface ConditionalBuildInput {
  trigger: TriggerInput;
  afterTimeoutSeconds?: number;
  condition: VariableData;          // 이미 plugin에서 변수 해석 완료
  thenActions: BuiltAction[];
  elseActions?: BuiltAction[];
}

export function buildConditionalReaction(input: ConditionalBuildInput): BuiltReaction {
  const blocks: ConditionalBlockShape[] = [
    { condition: input.condition, actions: input.thenActions },
  ];
  if (input.elseActions && input.elseActions.length > 0) {
    blocks.push({ actions: input.elseActions });  // else block: no condition
  }
  return {
    trigger: buildTrigger(input.trigger, input.afterTimeoutSeconds),
    actions: [{ type: "CONDITIONAL", conditionalBlocks: blocks }],
  };
}
```

### 6.4 BuiltAction union 확장

```ts
export type BuiltAction =
  | { type: "NODE"; destinationId: string; navigation: ...; transition: ...; resetScrollPosition?: boolean }
  | { type: "CLOSE" }
  | { type: "BACK" }
  | { type: "URL"; url: string; openInNewTab: boolean }
  | { type: "CONDITIONAL"; conditionalBlocks: ConditionalBlockShape[] };
```

## 7. list_reactions echo 변경

`handleListReactions`의 action enumeration 분기:

```ts
if (action.type === "CONDITIONAL") {
  return {
    type: "CONDITIONAL",
    condition: decodeCondition(action.conditionalBlocks[0]?.condition),
    then: action.conditionalBlocks[0]?.actions.map(decodeAction) ?? [],
    else: action.conditionalBlocks.length > 1
      ? action.conditionalBlocks[1].actions.map(decodeAction)
      : undefined,
    raw: action.conditionalBlocks.length > 2 ? action.conditionalBlocks : undefined, // 표준 패턴 벗어나면 raw로

}
```

`decodeCondition`은 Expression VariableData → 우리 친화적 shape으로 역변환. 다중 매칭 변수 케이스는 id를 그대로 노출하는 fallback (read 시점에 변수가 사라졌을 수도 있음).

## 8. 새 E2E 시나리오 22

### Fixture (자동 생성)

- BOOLEAN local variable `loggedIn` (default: false)
- 2 frames: `home`, `login` (또는 기존 frame 재사용)
- Button `checkLogin` (어느 frame 안 또는 별도) — 작은 frame node

### Sub-cases

(a) **Wire conditional**: `loggedIn == true ? home : login`. Expected: wire 성공, list_reactions echo가 우리 친화적 shape으로 응답.

(b) **Manual Play-mode**: `loggedIn` 값을 false → 클릭 → login으로 이동. 값을 true → 클릭 → home으로 이동. (수동 visual.)

(c) **Variable not found**: `variable: "doesNotExist"`. Expected: status=error, message `"Variable not found: doesNotExist"`.

(d) **Type mismatch**: BOOLEAN 변수에 `value: "yes"` (string). Expected: status=error, message에 variable type vs value type 불일치 명시.

(e) **else 생략 + 조건 false**: `then`만 있고 `else` 없음, 조건 false인 상태로 클릭 → 아무것도 안 일어남 (Figma 기본 동작 확인).

(f) **Defense — nested conditional**: 사용자가 `then: [{ type: "conditional", ... }]`를 보내면 Zod에서 reject (NonConditionalActionInput union이 conditional 제외이므로).

## 10. 라이브 리스크

1. **Bypass-probe 의무** — v1.15 resetScrollPosition wiring 누락 패턴 재발 방지. 첫 task 시작 전에 use_figma로 직접 setReactionsAsync 호출 (CONDITIONAL shape + 가짜 변수 id) → Figma가 persist하는지 확인. persist OK면 진행, drop이면 v1.4 overlay처럼 spec/plan 보관하고 release 보류.

2. **VariableData wrapping의 nullable이 까다로움** — typings에서 `condition?` (옵션), `type?`, `resolvedType?`, `value?` 다 옵션. Figma는 어떤 조합을 받아들이는지 정확히 모름. doc 예시 (line 7850+)의 shape를 그대로 따라 emit해서 안정성 확보.

3. **변수 이름 매칭의 i18n / 공백** — 정확한 문자열 비교. 부분 매칭 X. `figma.variables.getLocalVariablesAsync()`가 deleted variable를 포함하는지 미확인.

4. **`else` 분기 단독 사용 (then 없이)**: Figma는 condition 없는 block을 받지만 우리는 항상 first block을 `condition + then`, second block을 `else`로 emit. 우리 API에서는 `then` 필수라 이 경우 자체가 없음. ✅

5. **list_reactions raw fallback** — conditionalBlocks가 우리 expected 패턴 (1~2 blocks) 벗어나면 raw로 응답. 그러면 우리 도구 외 다른 도구로 만든 reaction도 inspect 가능.

6. **시나리오 22(d) Zod-level type check 위치** — Zod는 union literal을 받지만 "변수 BOOLEAN인데 value가 string" 같은 cross-field 검증은 Zod 차원에서 불가능 (변수 type을 모름). plugin handler에서 검증해서 명확한 error 메시지로 반환.

7. **현재 `handleCreateReactions`가 200+ 줄**. conditional 분기 추가 + helper 추출 (`buildNonConditionalActionForBlock`) 작업이 같이 가야 함. 분리 안 하면 함수가 300줄 넘어가서 다음 release에서 곤란.
