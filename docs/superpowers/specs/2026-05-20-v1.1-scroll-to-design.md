# figma-prototype-mcp v1.1 — Scroll To 액션 추가 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 리뷰 대기)
- 기반 버전: v1 (`docs/superpowers/specs/2026-05-19-figma-prototype-mcp-design.md`)

## 1. 목적

v1은 Navigate To 한 종류의 prototype action만 다뤘다. 실제 디자인 작업에서는 한 프레임 안에서 섹션 사이 스크롤이 매우 흔한 패턴(랜딩 페이지의 TOC → 섹션, FAQ 항목 클릭 → 항목 위치로 이동 등)이라, 다음 우선 능력 확장으로 **Scroll To** 액션을 추가한다.

동시에 툴 surface를 향후 액션 추가에 자연스럽게 열어두기 위해 핵심 쓰기 툴의 이름과 입력 모양을 일반화한다(`create_navigate_reactions` → `create_reactions`).

## 2. 스코프

### v1.1 포함

- Reaction action 타입 추가: **SCROLL_TO**
- 핵심 쓰기 툴 일반화: `create_reactions`로 rename + `action` 디스크리미네이티드 union
- 소프트 검증: scroll 타깃이 scrollable 조상 프레임 안에 없으면 응답에 `warning` 필드 첨부 (reaction은 그래도 생성)
- 응답 스키마에 `warningCount`, 각 result에 `warning?` 추가

### v1.1 비-목표 (의도적으로 제외)

- 그 외 action 타입(OPEN_OVERLAY / CLOSE / SWAP_OVERLAY / BACK / OPEN_URL / SET_VARIABLE / SET_VARIABLE_MODE / CONDITIONAL)
- Scroll의 `preserveScrollPosition` 옵션 노출
- Transition duration / easing 커스터마이즈 (계속 0.3초 + EASE_OUT 고정)
- Plugin 자동 실행 / `figma.clientStorage`로 채널 영속화 (별도 차후)
- Variable·Conditional 기반 분기
- 다중 페이지 / 다중 사용자 / 클라우드 호스팅 (v1과 동일)

## 3. 변경 요약

| 영역 | v1 | v1.1 |
|---|---|---|
| 핵심 쓰기 툴 이름 | `create_navigate_reactions` | `create_reactions` |
| connection 입력 모양 | `{ sourceNodeId, targetFrameId, trigger?, transition? }` | `{ sourceNodeId, trigger?, transition?, action: navigate \| scroll }` |
| 응답 result 필드 | `sourceNodeId, status, error?, reactionIndex?` | `+ warning?` |
| 응답 집계 필드 | `successCount, errorCount` | `+ warningCount` |
| Wire-level command 이름 | `CREATE_NAVIGATE_REACTIONS` | `CREATE_REACTIONS` |
| 지원 action 타입 | Navigate (NODE) | Navigate, Scroll (SCROLL_TO) |

`get_canvas_overview`, `find_nodes`, `list_reactions`, `clear_reactions`는 v1 그대로(읽기 + 비-type-specific 삭제). `list_reactions`는 SCROLL_TO 결과도 그대로 노출(action.type 값만 다름).

## 4. 툴 API

### 4.1 `create_reactions` (입력)

```ts
{
  connections: [
    {
      sourceNodeId: string,
      trigger?: "ON_CLICK" | "ON_HOVER" | "ON_PRESS",       // default ON_CLICK
      transition?: "INSTANT" | "DISSOLVE" | "SMART_ANIMATE", // default INSTANT
      action:
        | { type: "navigate", targetFrameId: string }
        | { type: "scroll",   targetNodeId: string }
    }
  ],
  replaceExisting?: boolean   // default false
}
```

Zod 표현은 `action`을 `z.discriminatedUnion("type", [...])`로 정의.

**`replaceExisting` 의미** (action 타입 섞이는 배치에서 명시): `true`이면 각 connection의 `sourceNodeId`에 대해 **action 타입을 가리지 않고** 기존 reactions 전체를 제거한 뒤 새 reaction을 추가한다. 같은 source가 배치에 중복으로 등장하면 마지막 connection의 결과가 최종 상태.

**`transition` per action**: SCROLL_TO에서 SMART_ANIMATE는 Figma의 의도된 사용처(매칭 레이어 간 트윈)와 어긋날 수 있다. v1.1은 그대로 pass-through하며, Figma가 거부하면 result.error로 노출된다. 향후 검증 강화는 v1.2 후보.

### 4.2 `create_reactions` (응답)

```ts
{
  results: [
    {
      sourceNodeId: string,
      status: "success" | "error",
      error?: string,            // status === "error"일 때
      reactionIndex?: number,    // status === "success"일 때
      warning?: string           // 적용은 됐지만 주의가 필요한 경우
    }
  ],
  successCount: number,
  errorCount: number,
  warningCount: number
}
```

`warning`은 현재 단 한 종류:
> `"Target {targetNodeId} ({nodeName}) has no scrollable ancestor frame; scroll-to will not animate at runtime"`

향후 다른 경고가 생길 수 있으므로 필드는 단일 문자열이지만, 메시지 prefix를 안정적으로 유지하지 않는다(LLM이 해석하므로 사람-친화 메시지로 자유롭게 작성).

## 5. 검증 로직

Plugin 핸들러는 각 `connection.action.type === "scroll"`에 대해 다음 헬퍼를 호출한다:

```ts
function findScrollableAncestor(node: BaseNode): SceneNode | null {
  let cur: BaseNode | null = node.parent ?? null;
  while (cur) {
    if (
      "overflowDirection" in cur &&
      (cur as any).overflowDirection !== "NONE"
    ) {
      return cur as SceneNode;
    }
    cur = cur.parent ?? null;
  }
  return null;
}
```

`null`이면 result에 `warning` 첨부. reaction 생성은 그대로 진행. `warningCount`는 집계 단계에서 계산.

검증 순서:
1. `sourceNodeId` 존재 / `setReactionsAsync` 가능 노드 확인 → 실패 시 `status: "error"` (warning과 무관, v1과 동일)
2. `action.type === "scroll"`이면 `targetNodeId` 존재 확인 → 실패 시 `status: "error"`
3. scroll일 때 `findScrollableAncestor` 호출 → null이면 `warning` 첨부 (status는 success 유지)
4. `buildScrollReaction` / `buildNavigateReaction`으로 reaction 생성 후 `setReactionsAsync`

Navigate에는 검증 없음(v1 동작 유지). 즉 warning은 SCROLL_TO 전용.

## 6. 빌더 함수

`src/figma-plugin/reaction-builder.ts`를 다음 모양으로 일반화한다.

```ts
type TransitionName = "INSTANT" | "DISSOLVE" | "SMART_ANIMATE";
type Transition =
  | null
  | { type: "DISSOLVE";       duration: number; easing: { type: string } }
  | { type: "SMART_ANIMATE";  duration: number; easing: { type: string } };

function buildTransition(name: TransitionName): Transition {
  if (name === "INSTANT") return null;
  return { type: name, duration: 0.3, easing: { type: "EASE_OUT" } };
}

export function buildNavigateReaction(input: {
  sourceNodeId: string;
  targetFrameId: string;
  trigger: TriggerName;
  transition: TransitionName;
}): BuiltReaction { /* action.type = "NODE", navigation = "NAVIGATE", destinationId = targetFrameId */ }

export function buildScrollReaction(input: {
  sourceNodeId: string;
  targetNodeId: string;
  trigger: TriggerName;
  transition: TransitionName;
}): BuiltReaction { /* action.type = "NODE", navigation = "SCROLL_TO", destinationId = targetNodeId, preserveScrollPosition = false */ }
```

두 함수는 `buildTransition`을 공유. 응답 reaction의 trigger / preserveScrollPosition 등 공통 필드는 양쪽 동일.

> **라이브 API 검증으로 수정됨 (commit `486d69c`):** 초안에서는 Scroll To를 별도 top-level action(`type: "SCROLL_TO"`)으로 모델링했지만, Figma의 `setReactionsAsync`는 `type: "NODE", navigation: "SCROLL_TO"` 형식만 받는다. 사용자-노출 툴 입력(`action: { type: "scroll", targetNodeId }`)은 그대로 두고, 내부 `buildScrollReaction`이 반환하는 wire shape만 `{ type: "NODE", destinationId, navigation: "SCROLL_TO", transition, preserveScrollPosition: false }`로 정정. v1의 INSTANT/DISSOLVE 사고(5ff161b)와 동일한 패턴 — 단위 테스트는 통과해도 라이브 API 검증을 따로 해야 한다.

## 9. 마이그레이션 / 호환성

v1 외부 사용자가 없으므로 (있다면 본 세션 사용자뿐) breaking change는 안전하게 진행. 다만 본인의 MCP 클라이언트 설정에서 `create_navigate_reactions` 호출이 캐시되어 있을 수 있어 Claude Code 재시작 권장. 다른 호환성 계층은 두지 않는다.

## 10. 구현 순서 (TDD 큰 그림)

1. `buildTransition` 헬퍼: 실패 테스트 → 구현 (refactor 단계에서 추출이라 빨강은 작음).
2. `buildScrollReaction`: 실패 테스트 → 구현.
3. `create_reactions` Zod 스키마(discriminated union): 실패 테스트 → 구현 (navigate / scroll / 잘못된 union 각각).
4. 툴 이름 + wire command 이름 rename: 테스트 통과 유지하면서 단일 변경.
5. 플러그인 핸들러 `action.type` 분기 + `findScrollableAncestor` + warning 첨부. 단위 테스트 어려운 영역이라 E2E로 커버.
6. 빌드 → 시나리오 1·2·3·4·5 회귀 → 시나리오 6 수행.

## 11. 외부 영향 / 위험

- **Plugin 동시성 가드 (v1.0.1, commit d598703)는 그대로 유효**: 핸들러 진입점이 동일한 `commandQueue.enqueue` 경로를 통과하므로 새 `findScrollableAncestor` 호출도 자동으로 직렬화된다.
- **Figma SCROLL_TO 동작 차이**: `setReactionsAsync`가 SCROLL_TO 액션 객체의 shape에 대해 NAVIGATE와 같은 검증을 한다는 보장 없음. v1에서 INSTANT/DISSOLVE 모양이 거부됐던 전례가 있으므로(commit 5ff161b) 구현 직후 라이브 API 한 번 검증 필요.
- **transition shape 일관성**: 본 설계는 INSTANT를 `null`로, 그 외를 `{ type, duration, easing }`로 통일. v1에서 확정된 형식과 동일.
