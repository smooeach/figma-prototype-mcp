# figma-prototype-mcp v1.14 — set_frame_scroll (overflowDirection setter) 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 승인 완료)
- 기반 버전: v0.12.0
- 시리즈: scroll 영역 1차 — `overflowDirection` setter 노출. action 옵션 (`resetScrollPosition`) + sticky/fixed-children은 다음 release 후보.
- 참조: `node_modules/@figma/plugin-typings/plugin-api.d.ts` line 7717 (`overflowDirection: OverflowDirection`), line 5265 (`type OverflowDirection`)

## 1. 목적

v0.12.0 기준 프로토타입 reaction은 다 생성할 수 있지만 frame 자체의 **scrollable 설정**은 사용자가 Figma UI에서 수동으로 Prototype 탭 → Overflow Behavior 토글해야 함. 시나리오 6 ("Pricing 섹션으로 스크롤되게 해줘") setup이 "tall frame에 Vertical scrolling 켜기"가 prerequisite라 CLAUDE가 단독 수행 불가.

v1.14는 새 MCP tool `set_frame_scroll`을 노출해서 `FrameNode.overflowDirection` 값을 plugin에서 직접 설정. 결과적으로 "이 frame 세로 스크롤되게 만들고 Pricing 섹션으로 가는 버튼 만들어줘"가 1개의 자연어 요청 → 두 개의 tool 호출 (`set_frame_scroll` + `create_reactions`)로 완결됨.

## 2. Spec-단계 readonly/setter 체크

Figma typings line 7717:

```ts
interface DefaultFrameMixin {
  /**
   * Determines whether a frame will scroll in presentation mode...
   * Reflects the value shown in "Overflow Behavior" in the Prototype tab.
   */
  overflowDirection: OverflowDirection
  ...
}
```

`readonly` 키워드 **없음** → setter 존재. v1.4 overlay 속성 (`readonly overlayPositionType` 등) 차단 사례와 다름. ✅ 차단 없음.

Line 5265:
```ts
type OverflowDirection = 'NONE' | 'HORIZONTAL' | 'VERTICAL' | 'BOTH'
```

4 enum 값 모두 노출 대상. write 검증은 v1.12 `initialVelocity` / v1.13 `deprecatedVersion` 패턴과 달리 typings의 단순 mutable property라 runtime 거부 위험은 낮지만, 시나리오 19 라이브에서 1차 확인.

## 4. 사용자 API

### 4.1 입력 shape
```ts
set_frame_scroll({
  frames: [
    { frameId: string, direction: "NONE" | "HORIZONTAL" | "VERTICAL" | "BOTTOM" | "BOTH" },
    ...
  ]
});
```

**오타 정정:** direction enum은 정확히 `"NONE" | "HORIZONTAL" | "VERTICAL" | "BOTH"` (TOP/BOTTOM 같은 방향은 directional transition 영역).

### 4.2 응답 shape
```ts
{
  results: [
    { frameId: string, status: "success" | "error", error?: string },
    ...
  ],
  successCount: number,
  errorCount: number
}
```

### 4.3 사용 예시
```ts
// 한 frame 세로 스크롤
set_frame_scroll({ frames: [{ frameId: "962:22094", direction: "VERTICAL" }] })

// 여러 frame 동시 설정
set_frame_scroll({
  frames: [
    { frameId: "962:22094", direction: "VERTICAL" },
    { frameId: "962:22098", direction: "HORIZONTAL" },
    { frameId: "962:22108", direction: "NONE" },  // 기존 스크롤 끄기
  ]
})

// 다른 자연어 시나리오:
// "이 frame 가로 스크롤로 바꿔줘" → direction: "HORIZONTAL"
// "스크롤 꺼줘" → direction: "NONE"
// "이 화면 양방향 스크롤되게" → direction: "BOTH"
```

## 5. Zod 스키마

```ts
const OverflowDirectionEnum = z.enum(["NONE", "HORIZONTAL", "VERTICAL", "BOTH"]);

const SetFrameScrollEntry = z.object({
  frameId: z.string().min(1),
  direction: OverflowDirectionEnum,
});

export const SetFrameScrollInput = z.object({
  frames: z.array(SetFrameScrollEntry).min(1),
});
```

## 6. Plugin handler 로직

```ts
async function handleSetFrameScroll(params: {
  frames: Array<{ frameId: string; direction: OverflowDirection }>;
}) {
  await figma.loadAllPagesAsync();
  const results = [];
  let successCount = 0;
  let errorCount = 0;
  for (const { frameId, direction } of params.frames) {
    try {
      const node = figma.getNodeById(frameId);
      if (!node) throw new Error(`Frame not found: ${frameId}`);
      if (node.type !== "FRAME") {
        throw new Error(`Node is not a FRAME: ${node.name} (type: ${node.type})`);
      }
      (node as FrameNode).overflowDirection = direction;
      results.push({ frameId, status: "success" });
      successCount++;
    } catch (e: any) {
      results.push({ frameId, status: "error", error: e?.message ?? String(e) });
      errorCount++;
    }
  }
  return { results, successCount, errorCount };
}
```

플러그인 `Command` union에 새 variant 추가:
```ts
| { type: "SET_FRAME_SCROLL";
    params: { frames: Array<{ frameId: string; direction: "NONE" | "HORIZONTAL" | "VERTICAL" | "BOTH" }> };
  }
```

dispatch에 `case "SET_FRAME_SCROLL": return { status: "ok", result: await handleSetFrameScroll(params) };`.

## 8. 새 E2E 시나리오 19

**Setup (새 fixture):**
- MCP_Test_09 section 안에 3 frame: `vTall` (예: 400×1200, 내부에 큰 콘텐츠), `hWide` (예: 1200×400), `noScroll` (예: 400×400 작은 콘텐츠)
- 각 frame 안에는 frame bounds 초과하는 child가 있어야 prototype play에서 실제 스크롤 확인 가능 (단, wire 검증 자체는 그것과 무관)

**Sub-cases:**

(a) **vTall direction=VERTICAL**: `set_frame_scroll({ frames: [{ frameId: <vTall>, direction: "VERTICAL" }] })`. Expected: successCount=1. Figma Prototype 탭 열어서 Overflow Behavior가 "Vertical scrolling"으로 표시. Play 모드에서 세로 스크롤 작동.

(b) **vTall direction=NONE**: 같은 frame에 `direction: "NONE"`. Expected: 토글 해제. Play 모드에서 overflow 잘림.

(c) **hWide direction=HORIZONTAL**: 가로 frame에 `direction: "HORIZONTAL"`. Expected: 가로 스크롤 작동.

**에러 케이스 (별도 검증):**
- 존재하지 않는 frameId → status=error, errorCount=1
- FRAME이 아닌 노드 (예: RECTANGLE button id) → status=error, "is not a FRAME"

## 10. 라이브 리스크

- **Content overflow 의존성**: overflowDirection을 VERTICAL로 켜도 child가 frame bounds 안 넘으면 prototype play에서 스크롤바 안 보임. wire 자체는 성공이라 시나리오 19(a)/(c) fixture 만들 때 child 크기 신경.
- **Page 미로드 시 frame 못 찾음**: handler에서 `figma.loadAllPagesAsync()` 먼저 호출 — 다른 핸들러와 동일 패턴.
- **Runtime mismatch 위험**: 낮음 — 단순 mutable property라 v1.12 initialVelocity / v1.13 deprecatedVersion 같은 거부 패턴 가능성 낮음. 그래도 19(a) 첫 호출에서 1차 확인 후 진행.
- **Plugin reload 필요**: 새 Command variant라 plugin 번들 다시 만들고 사용자 reload 필요.
