## 1. 목적

v0.16.0의 `set_variable`은 BOOLEAN/FLOAT/STRING만 지원. 디자이너가 v1.17 테스트에서 "오른쪽 박스 색깔 토글"을 요청했을 때 COLOR 변수 직접 변경이 불가능해서 "회색 박스를 밑에 깔고 빨강 박스 visibility 토글하는 트릭"으로 우회. v1.18은 COLOR variable 직접 할당을 노출해서 자연스러운 색상 토글/전환 가능하게 함.

## 2. Spec-단계 readonly/runtime 체크

Figma typings:

```ts
type VariableDataType = 'BOOLEAN' | 'FLOAT' | 'STRING' | 'VARIABLE_ALIAS' | 'COLOR' | 'EXPRESSION'

interface RGB { readonly r: number; readonly g: number; readonly b: number }
interface RGBA { readonly r: number; readonly g: number; readonly b: number; readonly a: number }
```

COLOR VariableData shape (Figma plugin API 표준):

```ts
{ type: "COLOR", resolvedType: "COLOR", value: { r: 0.95, g: 0.25, b: 0.25 } }         // RGB
{ type: "COLOR", resolvedType: "COLOR", value: { r: 0.95, g: 0.25, b: 0.25, a: 0.8 } }  // RGBA
```

`readonly`는 typings declaration-level — 우리는 plain object를 `setReactionsAsync`에 통째로 넘겨서 차단 없음. ✅

**Runtime 거부 위험**: 낮음 — COLOR는 Figma 표준 타입이고 v1.17에서 BOOLEAN/FLOAT/STRING SET_VARIABLE이 모두 persist된 패턴 따라감. 그래도 **bypass-probe (Task 1) 의무** — COLOR SET_VARIABLE이 우리 보내는 shape 그대로 persist하는지 코드 한 줄 쓰기 전에 확인.

## 3. 스코프

### v1.18 포함

- `set_variable`의 `value`가 hex 문자열 형식도 수용 (`#RRGGBB` 또는 `#RRGGBBAA`, 대소문자 모두 OK).
- COLOR 변수에 hex 문자열 할당 → 내부적으로 `{ r, g, b, a? }` float 0-1 객체로 변환해서 Figma에 전송.
- hex 파싱 헬퍼 `hexToRgb`, hex 직렬화 헬퍼 `rgbToHex`.
- `validateVariableLiteralCompat`에 COLOR 분기 추가 — 변수 type vs 입력 value 검증 + 명확한 에러 메시지.
- `encodeActionForListEcho`의 SET_VARIABLE 분기에서 COLOR VariableData → hex 문자열로 디코드해서 응답.
- 새 시나리오 24 (color set + RGB + RGBA + echo round-trip + 4가지 에러 경로).

### v1.18 비-목표

- Conditional의 `condition.value`에 COLOR 비교 (`backgroundColor == "#FF4040"`) — 프로토타입에서 거의 쓰이지 않음. condition value union은 boolean/number/string 그대로.
- toggle_variable의 COLOR 지원 — toggle 의미는 BOOLEAN 한정. 변경 없음.
- 3-char hex 단축형식 (`#FFF`) — YAGNI. 6자 또는 8자만.
- CSS named colors (`"red"`, `"darkgray"`) — hex만으로 통일.
- COLOR variable의 VARIABLE_ALIAS / EXPRESSION 값 (변수=변수, 산술) — 별도 release.
- SET_VARIABLE_MODE action (Light/Dark 모드 전환) — 별도 release.

## 4. 사용자 API

기존 `set_variable` 형식 변화 없음 — `value`만 hex 문자열을 추가로 수용:

```ts
// COLOR variable 할당 (RGB)
{ type: "set_variable", variable: "bgColor", value: "#FF4040" }

// COLOR variable 할당 (RGBA, alpha 16진)
{ type: "set_variable", variable: "tint", value: "#FF4040CC" }

// 기존 boolean/number/string도 그대로 동작
{ type: "set_variable", variable: "showMenu", value: true }
{ type: "set_variable", variable: "score",    value: 10 }
{ type: "set_variable", variable: "tier",     value: "gold" }
```

### 4.1 입력 규칙

- 변수가 COLOR 타입 → value는 반드시 hex 문자열 (`^#[0-9a-fA-F]{6}([0-9a-fA-F]{2})?$`)
- 변수가 STRING 타입 → 문자열이면 그냥 STRING으로 저장 (hex 모양이어도 의도적으로 STRING). 변수 타입 기준으로 판단.
- 변수가 BOOLEAN/FLOAT → 기존 그대로.

### 4.2 응답 shape (list_reactions echo)

SET_VARIABLE 액션 디코드 시:
- COLOR variable → `{ type: "set_variable", variable: <name>, value: "#RRGGBB" or "#RRGGBBAA" }`
- 그 외 → 기존 (boolean/number/string 그대로).

Alpha 처리: RGBA에서 `a ≈ 1.0` (round 후 255)인 경우 RGB-only로 응답 (`#RRGGBB`). `a < 1.0` 인 경우 `#RRGGBBAA`로 응답.

### 4.3 자연어 매핑 예시

| 디자이너 입력 | 결과 |
|---|---|
| "버튼 누르면 배경색 빨강으로" | `{ type: "set_variable", variable: "bg", value: "#FF4040" }` |
| "버튼 누르면 반투명 빨강 (80%)" | `{ type: "set_variable", variable: "bg", value: "#FF4040CC" }` |
| "테마를 다크모드 색으로" | `{ type: "set_variable", variable: "themeBg", value: "#1A1A1F" }` |

## 5. Zod 스키마 변경

**없음.** 기존 `SetVariableActionInput.value`는 이미 `z.union([z.boolean(), z.number(), z.string()])`. Hex 문자열은 string으로 통과. 변수 type vs literal 호환성은 plugin runtime 검증.

(이론적으로 Zod에서 hex regex로 string 정합성을 미리 잡을 수 있지만 — Zod는 변수 type을 모르므로 hex여야 하는지 일반 string이어야 하는지 판단 불가. Plugin runtime에서 변수 type 확인 후 판단.)

## 6. Plugin handler 변경

### 6.1 hex parsing 헬퍼

`src/figma-plugin/code.ts`에 추가:

```ts
const HEX_REGEX = /^#[0-9a-fA-F]{6}([0-9a-fA-F]{2})?$/;

function hexToRgb(hex: string): { r: number; g: number; b: number; a?: number } {
  // hex must match HEX_REGEX before calling
  const clean = hex.slice(1);
  const r = parseInt(clean.slice(0, 2), 16) / 255;
  const g = parseInt(clean.slice(2, 4), 16) / 255;
  const b = parseInt(clean.slice(4, 6), 16) / 255;
  if (clean.length === 8) {
    const a = parseInt(clean.slice(6, 8), 16) / 255;
    return { r, g, b, a };
  }
  return { r, g, b };
}

function rgbToHex(rgb: { r: number; g: number; b: number; a?: number }): string {
  const toHex = (n: number) =>
    Math.max(0, Math.min(255, Math.round(n * 255))).toString(16).padStart(2, "0").toUpperCase();
  const base = `#${toHex(rgb.r)}${toHex(rgb.g)}${toHex(rgb.b)}`;
  if (rgb.a === undefined || Math.round(rgb.a * 255) === 255) return base;
  return base + toHex(rgb.a);
}
```

### 6.2 validateVariableLiteralCompat에 COLOR 분기

기존 함수 본체:

```ts
const expected =
  variable.resolvedType === "BOOLEAN" ? "boolean"
  : variable.resolvedType === "FLOAT" ? "number"
  : variable.resolvedType === "STRING" ? "string"
  : "unknown";
```

COLOR 분기 추가. 시그니처 그대로 두고 내부에서 type-specific 처리:

```ts
function validateVariableLiteralCompat(
  variable: Variable,
  value: boolean | number | string,
  context: string,
): unknown {  // 반환 타입을 더 넓게 (COLOR도 가능)
  const valueType = typeof value;

  if (variable.resolvedType === "COLOR") {
    if (valueType !== "string") {
      const action = context === "comparison" ? "compare against" : "assign";
      throw new Error(`Variable "${variable.name}" is COLOR; cannot ${action} ${valueType} literal (expected hex string)`);
    }
    if (!HEX_REGEX.test(value as string)) {
      const action = context === "comparison" ? "compare against" : "assign";
      throw new Error(`Variable "${variable.name}" is COLOR; value must be a hex string like #RRGGBB or #RRGGBBAA (got "${value}")`);
    }
    const rgb = hexToRgb(value as string);
    return { type: "COLOR", resolvedType: "COLOR", value: rgb };
  }

  // 기존 BOOLEAN / FLOAT / STRING 분기
  const expected =
    variable.resolvedType === "BOOLEAN" ? "boolean"
    : variable.resolvedType === "FLOAT" ? "number"
    : variable.resolvedType === "STRING" ? "string"
    : "unknown";
  if (expected === "unknown") {
    throw new Error(`Variable "${variable.name}" has unsupported type ${variable.resolvedType} for ${context} (supported: BOOLEAN, FLOAT, STRING, COLOR)`);
  }
  if (valueType !== expected) {
    const action = context === "comparison" ? "compare against" : "assign";
    throw new Error(`Variable "${variable.name}" is ${variable.resolvedType}; cannot ${action} ${valueType} literal (expected ${expected})`);
  }
  return { type: variable.resolvedType, resolvedType: variable.resolvedType, value };
}
```

**중요**: 함수 반환 타입을 좁은 `{ type: BOOLEAN|FLOAT|STRING; ... }`에서 `unknown` (또는 opaque VariableData)으로 넓힘. 호출자 (`buildCondition`, `buildSetVariableData`) 둘 다 이 반환을 opaque VariableData로 다룸 (이미 그렇게 함). 영향 없음.

**Conditional comparison 부수효과**: 사용자가 conditional에서 COLOR 변수를 비교하려고 시도하면, 위 같은 분기에 의해 `Variable "..." is COLOR; cannot compare against ... literal (expected hex string)` 에러로 막힘 — 약간 어색한 에러지만 비-목표라 일단 동일 분기로 흐름. 이상하면 별도 conditional용 COLOR 차단 로직 추가 (또는 `context === "comparison"`일 때 COLOR 자체를 unsupported로 즉시 throw).

→ 더 명확한 메시지를 위해 `context === "comparison"` + COLOR 케이스에서는 처음부터 별도 throw:

```ts
if (variable.resolvedType === "COLOR" && context === "comparison") {
  throw new Error(`Variable "${variable.name}" is COLOR; conditional comparison against COLOR variables is not supported in v1.18 (use BOOLEAN/FLOAT/STRING for condition.value)`);
}
```

이걸 COLOR 분기 맨 위에 둠. assignment context는 정상 처리.

### 6.3 encodeActionForListEcho — SET_VARIABLE의 COLOR 디코드

기존 SET_VARIABLE 분기:

```ts
return {
  type: "set_variable",
  variable: varName ?? `<id:${action.variableId}>`,
  value: action.variableValue?.value,

```

COLOR 디코드 추가:

```ts
if (action.type === "SET_VARIABLE") {
  let varName: string | undefined;
  if (action.variableId) {
    try {
      const v = await figma.variables.getVariableByIdAsync(action.variableId);
      varName = v?.name;
    } catch { /* deleted variable */ }
  }
  const vd = action.variableValue;
  let value: unknown;
  if (vd?.type === "COLOR" && vd?.value && typeof vd.value === "object" && "r" in vd.value) {
    value = rgbToHex(vd.value as { r: number; g: number; b: number; a?: number });
  } else {
    value = vd?.value;
  }
  return {
    type: "set_variable",
    variable: varName ?? `<id:${action.variableId}>`,
    value,
  };
}
```

## 9. 라이브 리스크

1. **Bypass-probe 의무** (Task 1): COLOR SET_VARIABLE shape이 우리가 보내는 그대로 Figma persist 확인. v1.16/v1.17 동일 패턴.

2. **Float 양자화 round-trip 정확성**: hex → float → hex 변환에서 보통 hex 값 (`#FF4040` 등 가시광선 표현)은 무손실. 일부 중간 값에서 ±1 bit 차이 가능 (0-1 float 정밀도 한계). 디자이너 입장에서 무시 가능.

3. **STRING vs COLOR 구분 일관성**: 사용자가 STRING 변수에 `"#FF4040"` 입력해도 변수 type 기준으로 STRING으로 저장. 변수 type 잘못 잡으면 어색한 결과 — 하지만 plugin 에러 메시지가 명확하니 사용자가 알아차림.

4. **Conditional COLOR 차단의 친절함**: 별도 분기로 명확한 에러. context-aware 분기.

5. **validateVariableLiteralCompat 반환 타입 넓힘**: 함수 시그니처가 `{ type: BOOLEAN|FLOAT|STRING; ... }` → `unknown`이 됨. 호출자 둘 다 opaque로 처리하므로 영향 없지만 typecheck에서 좁힌 narrowing이 사라짐. 호출자 코드에서 type assertion이 필요한 곳 없는지 확인 필요.

## 12. Probe outcome (recorded post-Task 1)

**Date:** 2026-05-21
**Result:** ✅ Pass with critical finding — proceed to Task 2 with helper adjustment.

**Critical finding (runtime stricter than typings):** Figma's runtime validation **rejects** `{ type: "COLOR", value: { r, g, b } }` (RGB without alpha) with `"Required value missing at variableValue.value.a"`. Even though Figma's TS typings declare RGB and RGBA as separate interfaces (alpha optional), the runtime requires `a` to be present on COLOR VariableData. Same pattern family as v1.12 `initialVelocity` (where runtime rejected a field typings declared) — this is the opposite direction (runtime requires a field typings made optional).

**Implementation impact on Task 2:**
- `hexToRgb` MUST always include `a` in its return (default to `1` when hex is 6-char `#RRGGBB`). Return type is `{ r, g, b, a }` always — no optional alpha.
- `validateVariableLiteralCompat` COLOR branch passes through the `a:1` to Figma.
- `rgbToHex` (Task 3) still emits `#RRGGBB` (hides alpha) when `a` rounds to 255 — designer-facing echo stays clean.

**Float precision verified:** `0.8` sent → `0.800000011920929` stored (32-bit float quantization). `Math.round(0.800000011920929 * 255) = 204 → "CC"` — round-trip safe for common alpha values.

**Stored shape verified for RGBA:**
```json
{ "type": "SET_VARIABLE", "variableId": "VariableID:413:1375",
  "variableValue": { "type": "COLOR", "resolvedType": "COLOR",
    "value": { "r": 1, "g": 0.25, "b": 0.25, "a": 0.800000011920929 } } }
```

No surprises beyond the alpha-required finding; spec §6.1 hexToRgb signature updated implicitly via Task 2 prompt.
