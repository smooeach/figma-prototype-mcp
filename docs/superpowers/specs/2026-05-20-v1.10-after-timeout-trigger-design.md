# figma-prototype-mcp v1.10 — AFTER_TIMEOUT trigger 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 리뷰 대기)
- 기반 버전: v0.8.0
- 영향 범위: trigger 추가 — 모든 reaction action 타입과 자동 결합

## 1. 목적

지금까지 trigger는 클릭/호버/프레스 3종이라 "사용자 액션 → 다음 화면" 패턴만 가능. 자동 진행 패턴 (splash → home, 토스트 자동 닫기, 카운트다운) 은 표현 불가. AFTER_TIMEOUT trigger를 추가하면 한 줄로 가능.

## 2. spec-단계 readonly/setter 체크

```ts
type Trigger =
  | { readonly type: 'ON_CLICK' | 'ON_HOVER' | 'ON_PRESS' | 'ON_DRAG' }
  | { readonly type: 'AFTER_TIMEOUT'; readonly timeout: number }
  | ...
```

`readonly`는 read shape 마커. 우리는 객체를 wire로 보내는 쪽이라 mutation 차단 무관 — 기존 action 객체의 `type: "NODE"` 같은 readonly와 동일 패턴. ✅ 차단 없음.

setReactionsAsync는 동일 mutation 경로 — 새 method 호출 없음.

## 3. 스코프

### v1.10 포함
- 사용자-노출 trigger enum에 `"AFTER_TIMEOUT"` 추가
- 새 옵셔널 입력 `afterTimeoutSeconds: number`
- Zod refine: trigger=AFTER_TIMEOUT이면 afterTimeoutSeconds 필수
- 모든 7개 builder (`buildNavigateReaction`, `buildScrollReaction`, `buildOverlayReaction`, `buildCloseReaction`, `buildBackReaction`, `buildUrlReaction`, `buildSwapOverlayReaction`)가 새 trigger를 emit 가능
- 공통 helper `buildTrigger(name, afterTimeoutSeconds?)` 추출
- Plugin handler에서 builder들에 `afterTimeoutSeconds` 전달
- 새 E2E 시나리오 13 (splash 2초 후 자동 navigate)

### v1.10 비-목표
- 다른 새 trigger 타입 (ON_DRAG / MOUSE_UP/DOWN/ENTER/LEAVE / ON_KEY_DOWN / ON_MEDIA_HIT / ON_VARIABLE_CHANGE)
- Trigger를 full discriminated union으로 refactor (지금은 flat trigger enum + optional partner field)
- 타임아웃 분 단위 / ms 단위 옵션 (sec 단위 고정)

## 4. 사용자 API

### 4.1 ConnectionInput (입력)

```ts
{
  sourceNodeId: string,
  trigger?: "ON_CLICK" | "ON_HOVER" | "ON_PRESS" | "AFTER_TIMEOUT",     // default ON_CLICK
  afterTimeoutSeconds?: number,                                          // required iff trigger="AFTER_TIMEOUT"
  transition?: "INSTANT" | "DISSOLVE" | "SMART_ANIMATE",                 // default INSTANT
  action: navigate | scroll | overlay | close | back | url | swap_overlay
}
```

Zod `.refine`로 강제:

```ts
ConnectionInput.refine(
  (v) => v.trigger !== "AFTER_TIMEOUT" || typeof v.afterTimeoutSeconds === "number",
  { message: "afterTimeoutSeconds is required when trigger is AFTER_TIMEOUT" }
);
```

afterTimeoutSeconds는 양수여야 함: `z.number().positive()`.

### 4.2 응답

변화 없음. `list_reactions`는 자동으로 새 trigger 노출 (handler가 이미 `r.trigger?.type`을 echo). 추가로 `timeout` 필드도 노출하면 더 좋음 (시나리오 3 inspection에서 "2초 후 navigate"임을 알 수 있도록).

## 5. 빌더 함수 변화

### `buildTrigger` helper (NEW)

```ts
type TriggerObject = { type: string; timeout?: number };

export function buildTrigger(name: TriggerName, afterTimeoutSeconds?: number): TriggerObject {
  if (name === "AFTER_TIMEOUT") {
    if (afterTimeoutSeconds === undefined) {
      throw new Error("AFTER_TIMEOUT trigger requires afterTimeoutSeconds");
    }
    return { type: "AFTER_TIMEOUT", timeout: afterTimeoutSeconds };
  }
  return { type: name };
}

export type TriggerName = "ON_CLICK" | "ON_HOVER" | "ON_PRESS" | "AFTER_TIMEOUT";
```

### 7개 builder 모두

기존:
```ts
trigger: { type: input.trigger }
```

→

```ts
trigger: buildTrigger(input.trigger, input.afterTimeoutSeconds)
```

각 BuildInput interface에 `afterTimeoutSeconds?: number` 추가 (NavigateBuildInput / ScrollBuildInput / OverlayBuildInput / CloseBuildInput / BackBuildInput / UrlBuildInput / SwapOverlayBuildInput).

### BuiltReaction 타입

```ts
export interface BuiltReaction {
  trigger: TriggerObject;           // type 문자열 + optional timeout
  actions: BuiltAction[];
}
```

## 6. Plugin handler 변화

`handleCreateReactions`의 7개 branch 각각에서 builder 호출 시 `afterTimeoutSeconds: conn.afterTimeoutSeconds` 전달.

```ts
newReaction = buildNavigateReaction({
  targetFrameId: conn.action.targetFrameId,
  trigger: conn.trigger,
  transition: conn.transition,
  afterTimeoutSeconds: conn.afterTimeoutSeconds,   // ← NEW
});
```

7개 branch 동일 패턴.

## 7. list_reactions 보강

`handleListReactions` mapping에 trigger의 `timeout` 노출:

```ts
trigger: {
  type: r.trigger?.type ?? "UNKNOWN",
  timeout: r.trigger?.timeout,        // ← NEW (AFTER_TIMEOUT일 때만 의미)
},
```

`ReactionSummary.trigger` 타입에 `timeout?: number` 추가.

## 10. 라이브 리스크
- timeout 단위 (seconds vs ms) 가 typings엔 명시 안 됨. Figma UI는 sec. 첫 시도 실패하면 ms로 정정.
- 일부 action이 AFTER_TIMEOUT과 호환 안 될 수도 있음 (e.g. CLOSE는 overlay 안에서만 의미, BACK은 navigation history 의존). 시나리오 13(a) navigate부터 검증.
