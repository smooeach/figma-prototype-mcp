# v0.24.0 — conditional `else-if` chain

Status: design (2026-06-01).
Predecessor: v0.23.0 (`proto_conditional`).
Release type: minor.
Scope: extend the existing `conditional` action from a single if/else into an **if / else-if* / else** chain, exposed on BOTH the low-level `create_reactions` conditional action and the high-level `proto_conditional` tool. Backward-compatible (`elseIf` is a new optional field; existing `then`/`else` unchanged). Each block remains a **single comparison** — compound AND/OR/NOT conditions are an explicit non-goal (separate future candidate).

## Goal

Designer-intent natural language like "step이 3 이상이면 A로, 2 이상이면 B로, 아니면 C로" resolves to a single conditional reaction with an ordered chain of branches, instead of forcing the LLM to either nest conditionals (rejected) or chain multiple reactions. Figma's native `conditionalBlocks` is already an ordered array — this feature surfaces that capability that the v1.16 / v0.23.0 implementation artificially capped at two blocks.

## Decisions (from brainstorm)

- **Surface**: both low-level `create_reactions` conditional action AND high-level `proto_conditional`. Mirrors the v1.16 → v0.23.0 pattern (low-level capability first, proto_* sugar on top). The builder must support N blocks regardless, so exposing it on both costs little extra.
- **API shape — additive `elseIf` array (backward-compatible)**, chosen over a breaking `branches: [{condition?, actions}]` restructure. Existing `then`/`else` and all 335 tests + the shipped v0.23.0 `proto_conditional` stay unchanged. `elseIf` is an optional array inserted semantically between `then` and `else`.
- **Single comparison per block**: each `elseIf` entry carries exactly one `condition` (low-level) / `if` (proto), same as the primary block. No compound conditions — that was the rejected sibling candidate.
- **Field naming per surface**: low-level `elseIf: [{ condition, then }]` (mirrors low-level `condition`/`then`); proto `elseIf: [{ if, then }]` (mirrors proto `if`/`then` sugar). Deliberate divergence, consistent with the existing per-surface naming.
- **Ordered evaluation**: compiled `conditionalBlocks = [then-block(condition), ...elseIf-blocks(condition), else-block(no condition)]`. Figma evaluates in order, first match wins, the trailing condition-less block is the else.
- **`elseIf` minimum 1 when present**: an empty `elseIf: []` is rejected (`.min(1)`) — it's meaningless and signals an LLM mistake.
- **Nesting still rejected**: `elseIf[*].then` uses the same `NonConditionalActionInput` union (no `conditional`, no `toggle_variable`) as `then`/`else`. The no-nesting + no-toggle-inside-conditional rules hold for every block.
- **Overlay/swap → DISSOLVE rewrite scans the whole chain**: if `then`, ANY `elseIf[*].then`, or `else` is `{ overlay }` / `{ swap }`, apply the SMART_ANIMATE → DISSOLVE rewrite (transition is shared across the reaction). `branchUsesOverlayOrSwap` is extended to scan `elseIf`.
- **list_reactions round-trip extended to N blocks**: the decoder's `standardPattern` (currently capped at 1–2 blocks) is generalized — `blocks[0]` → `then`, intermediate condition-bearing blocks → `elseIf`, a trailing condition-less block → `else`. `detectTogglePattern` keeps precedence (its 5-strict-condition match cannot collide with an else-if chain).
- **History**: `proto_conditional` already captures the full parsed input via `recordedHandler`; `elseIf` is recorded automatically. No history code change.

## Section 2 — Low-level schema (`create_reactions`)

```ts
ConditionalActionInput = z.object({
  type: z.literal("conditional"),
  condition: ConditionInput,                       // 기존 (primary if)
  then: z.array(NonConditionalActionInput).min(1),  // 기존
  elseIf: z.array(z.object({                        // NEW, optional
    condition: ConditionInput,
    then: z.array(NonConditionalActionInput).min(1),
  })).min(1).optional(),
  else: z.array(NonConditionalActionInput).min(1).optional(),  // 기존
});
```

Validation: `elseIf` optional; if present ≥1 entry; each entry has a `condition` + non-empty `then`. `else` independent of `elseIf` (chain may end with or without a fallback). Existing nesting rejection (branches use `NonConditionalActionInput`, which excludes `conditional` + `toggle_variable`) applies unchanged to every `elseIf[*].then`.

## Section 3 — Plugin builder + condition construction

`reaction-builder.ts`:

```ts
export interface ConditionalBuildInput {
  trigger: TriggerInput;
  afterTimeoutSeconds?: number;
  condition: unknown;                                       // primary if (built by plugin)
  thenActions: BuiltAction[];
  elseIfBlocks?: { condition: unknown; actions: BuiltAction[] }[];  // NEW
  elseActions?: BuiltAction[];
}

export function buildConditionalReaction(input: ConditionalBuildInput): BuiltReaction {
  const blocks: ConditionalBlockShape[] = [
    { condition: input.condition, actions: input.thenActions },
  ];
  for (const b of input.elseIfBlocks ?? []) {
    blocks.push({ condition: b.condition, actions: b.actions });
  }
  if (input.elseActions && input.elseActions.length > 0) {
    blocks.push({ actions: input.elseActions });            // no condition = else
  }
  return { trigger: buildTrigger(input.trigger, input.afterTimeoutSeconds),
           actions: [{ type: "CONDITIONAL", conditionalBlocks: blocks }] };
}
```

`code.ts` conditional handler: for each `elseIf` entry, call the existing `buildCondition({ variable, operator, value })` for the condition and `buildNonConditionalAction` for each then-action — exactly the paths already used for the primary `then`/`else`. No new validation logic; type-mismatch / variable-not-found errors propagate per-reaction as today.

## Section 4 — list_reactions round-trip decoder (`encodeActionForListEcho`)

Generalize the `standardPattern` (currently `blocks.length <= 2`):

```
- detectTogglePattern(blocks) first (unchanged precedence).
- standardPattern: blocks.length >= 1, blocks[0].condition defined,
  every block EXCEPT possibly the last has a condition.
  (i.e. only the final block may be condition-less = else)
- Decode:
    condition = decode(blocks[0].condition); then = decode(blocks[0].actions)
    elseIf = blocks[1..] that HAVE a condition → [{ condition, then }]
    else   = trailing block WITHOUT condition (if any) → actions
- Non-conforming shapes (e.g. a condition-less block in the middle) → { type: "CONDITIONAL", raw: blocks } fallback (unchanged).
```

Echo shape: `{ type: "CONDITIONAL", condition, then, elseIf?: [{ condition, then }], else? }`. `elseIf` omitted when empty (keeps the existing 2-block echo byte-identical → no regression for v0.23.0 reactions).

## Section 5 — `proto_conditional` sugar (`protoTools.ts`)

```ts
ProtoConditionalEntry = {
  from, trigger?, motion?,
  if: { variable, operator?, value },
  then: BranchAction,                                  // single branch sugar (기존)
  elseIf?: [{ if: { variable, operator?, value }, then: BranchAction }].min(1),  // NEW
  else?: BranchAction,
}
```

`compileProtoConditional` maps each `elseIf[i]` to a low-level `elseIf` entry: `condition` from `if` (operator default `==` applied by zod), `then: [compileBranchAction(entry.then)]`. Reuses `compileBranchAction` unchanged.

Overlay rewrite: `branchUsesOverlayOrSwap` is called on `then`, every `elseIf[*].then`, and `else`; if any returns true, `rewriteForOverlay` runs on the shared transition (same logic as v0.23.0, extended to scan the chain). Each branch sugar stays `.strict()` (no per-branch motion/trigger).

## Section 6 — Test plan

- **Low-level zod** (`tests/tools.test.ts`): accept conditional with 1 elseIf; accept 2+ elseIf; accept elseIf + else; accept elseIf without else; reject empty `elseIf: []`; reject elseIf entry missing `condition`; reject elseIf entry with empty `then`; reject `conditional`/`toggle_variable` inside an elseIf then (nesting guard).
- **Builder** (`tests/reaction-builder.test.ts`): `buildConditionalReaction` with elseIfBlocks produces `conditionalBlocks` in order [then(cond), elseIf1(cond), elseIf2(cond), else(no cond)]; without elseIf the output is byte-identical to today (regression guard).
- **proto zod** (`tests/protoTools.test.ts`): accept entry with elseIf chain; reject empty elseIf; reject unknown key inside an elseIf.if / elseIf.then (.strict).
- **proto compile** (`tests/protoTools-compile.test.ts`): elseIf maps to low-level elseIf with correct operator/value/branch; overlay rewrite fires when an elseIf branch is overlay/swap (then/else non-overlay); no rewrite when no branch is overlay/swap; CreateReactionsInput.safeParse passes.
- **decoder**: covered by live verification (plugin code is not unit-tested in this repo — `encodeActionForListEcho` runs in the Figma sandbox); the round-trip is asserted in Section 7 Probe.

## Section 7 — Live verification

Reuse MCP_Test_11 (page "ㄴ MCP - Test Zone"): FLOAT variable `step`, buttons (button01 `1073:30`, button02 `1073:28`), frames screen02 `1072:10` / screen03 `1072:16`, overlay popup01 `1072:13`.

- **Probe A — 3-block chain (low-level + proto)**: `proto_conditional` on a button: `if step>=3 → navigate screen03`, `elseIf step>=2 → navigate screen02`, `else → overlay popup01`. Expect `successCount: 1`. `list_reactions` decodes to `{ condition (>=3), then: [navigate screen03], elseIf: [{ condition (>=2), then: [navigate screen02] }], else: [overlay popup01] }`. Transition = DISSOLVE (overlay branch present → rewrite). Figma Prototype panel shows If / Else if / Else with the right targets.
- **Probe B — regression**: a plain v0.23.0-style if/else (no elseIf) still decodes to `{ condition, then, else }` with NO `elseIf` key (confirms backward-compat echo).
- Append PASS/FAIL to this spec; clean up probe; no production change unless a probe surfaces an issue.

## Section 9 — Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Decoder mis-splits elseIf vs else | Med | Rule: only the LAST block may be condition-less; intermediate condition-less block → raw fallback. Probe A asserts the exact decoded shape. |
| Existing 2-block echo changes (regression) | Low | `elseIf` omitted when empty → byte-identical echo for v0.23.0 reactions. Probe B + builder regression test guard this. |
| toggle_variable detection collides with chain | Low | `detectTogglePattern` is a 5-strict-condition match on a specific 2-block shape; an else-if chain can't satisfy it. Precedence unchanged. |
| Live surprise (Figma rejects 3+ blocks or specific ordering) | Low-Med | `conditionalBlocks` is already an array Figma accepts; toggle desugar already writes 2 blocks. Probe A verifies 3 blocks live before tag. |
| Overlay rewrite misses an elseIf overlay branch | Med | `branchUsesOverlayOrSwap` scans then + every elseIf.then + else; compile test covers elseIf-only-overlay case. |

## Section 10 — Non-goals

- Compound conditions (AND/OR/NOT) — separate candidate.
- Nested conditionals inside any branch — still rejected.
- Multi-action branches in `proto_conditional` — low-level `create_reactions` remains the escape hatch (each proto branch is single-action sugar).
- Per-branch motion/trigger — transition stays at the reaction level.
