# v0.24.0 ‚Äî conditional `else-if` chain

Status: **ABANDONED ‚Äî blocked by Figma platform (2026-06-01).** Implementation (Tasks 1‚Äì5) was built, passed all unit tests + typecheck, then live verification proved Figma does not support else-if. Implementation commits were reverted; this spec + the plan are kept as a historical record. See **Outcome** at the bottom.
Predecessor: v0.23.0 (`proto_conditional`).
Release type: minor (not shipped).
Scope: extend the existing `conditional` action from a single if/else into an **if / else-if* / else** chain, exposed on BOTH the low-level `create_reactions` conditional action and the high-level `proto_conditional` tool. Backward-compatible (`elseIf` is a new optional field; existing `then`/`else` unchanged). Each block remains a **single comparison** ‚Äî compound AND/OR/NOT conditions are an explicit non-goal (separate future candidate).

## Goal

Designer-intent natural language like "stepÏù¥ 3 Ïù¥ÏÉÅÏù¥Î©¥ AÎ°ú, 2 Ïù¥ÏÉÅÏù¥Î©¥ BÎ°ú, ÏïÑÎãàÎ©¥ CÎ°ú" resolves to a single conditional reaction with an ordered chain of branches, instead of forcing the LLM to either nest conditionals (rejected) or chain multiple reactions. Figma's native `conditionalBlocks` is already an ordered array ‚Äî this feature surfaces that capability that the v1.16 / v0.23.0 implementation artificially capped at two blocks.

## Decisions (from brainstorm)

- **Surface**: both low-level `create_reactions` conditional action AND high-level `proto_conditional`. Mirrors the v1.16 ‚Üí v0.23.0 pattern (low-level capability first, proto_* sugar on top). The builder must support N blocks regardless, so exposing it on both costs little extra.
- **API shape ‚Äî additive `elseIf` array (backward-compatible)**, chosen over a breaking `branches: [{condition?, actions}]` restructure. Existing `then`/`else` and all 335 tests + the shipped v0.23.0 `proto_conditional` stay unchanged. `elseIf` is an optional array inserted semantically between `then` and `else`.
- **Single comparison per block**: each `elseIf` entry carries exactly one `condition` (low-level) / `if` (proto), same as the primary block. No compound conditions ‚Äî that was the rejected sibling candidate.
- **Field naming per surface**: low-level `elseIf: [{ condition, then }]` (mirrors low-level `condition`/`then`); proto `elseIf: [{ if, then }]` (mirrors proto `if`/`then` sugar). Deliberate divergence, consistent with the existing per-surface naming.
- **Ordered evaluation**: compiled `conditionalBlocks = [then-block(condition), ...elseIf-blocks(condition), else-block(no condition)]`. Figma evaluates in order, first match wins, the trailing condition-less block is the else.
- **`elseIf` minimum 1 when present**: an empty `elseIf: []` is rejected (`.min(1)`) ‚Äî it's meaningless and signals an LLM mistake.
- **Nesting still rejected**: `elseIf[*].then` uses the same `NonConditionalActionInput` union (no `conditional`, no `toggle_variable`) as `then`/`else`. The no-nesting + no-toggle-inside-conditional rules hold for every block.
- **Overlay/swap ‚Üí DISSOLVE rewrite scans the whole chain**: if `then`, ANY `elseIf[*].then`, or `else` is `{ overlay }` / `{ swap }`, apply the SMART_ANIMATE ‚Üí DISSOLVE rewrite (transition is shared across the reaction). `branchUsesOverlayOrSwap` is extended to scan `elseIf`.
- **list_reactions round-trip extended to N blocks**: the decoder's `standardPattern` (currently capped at 1‚Äì2 blocks) is generalized ‚Äî `blocks[0]` ‚Üí `then`, intermediate condition-bearing blocks ‚Üí `elseIf`, a trailing condition-less block ‚Üí `else`. `detectTogglePattern` keeps precedence (its 5-strict-condition match cannot collide with an else-if chain).
- **History**: `proto_conditional` already captures the full parsed input via `recordedHandler`; `elseIf` is recorded automatically. No history code change.

## Section 2 ‚Äî Low-level schema (`create_reactions`)

```ts
ConditionalActionInput = z.object({
  type: z.literal("conditional"),
  condition: ConditionInput,                       // Í∏∞Ï°¥ (primary if)
  then: z.array(NonConditionalActionInput).min(1),  // Í∏∞Ï°¥
  elseIf: z.array(z.object({                        // NEW, optional
    condition: ConditionInput,
    then: z.array(NonConditionalActionInput).min(1),
  })).min(1).optional(),
  else: z.array(NonConditionalActionInput).min(1).optional(),  // Í∏∞Ï°¥
});
```

Validation: `elseIf` optional; if present ‚â•1 entry; each entry has a `condition` + non-empty `then`. `else` independent of `elseIf` (chain may end with or without a fallback). Existing nesting rejection (branches use `NonConditionalActionInput`, which excludes `conditional` + `toggle_variable`) applies unchanged to every `elseIf[*].then`.

## Section 3 ‚Äî Plugin builder + condition construction

`reaction-builder.ts`:

```ts
export interface ConditionalBuildInput {
  trigger: TriggerInput;
  afterTimeoutSeconds?: number;
  condition: unknown;                                       // primary if (built by plugin)
  thenActions: BuiltAction[];
  elseIfBlocks?: { condition: unknown; actions: BuiltAction[] }[];  // NEW
  elseActions?: BuiltAction[];
}

export function buildConditionalReaction(input: ConditionalBuildInput): BuiltReaction {
  const blocks: ConditionalBlockShape[] = [
    { condition: input.condition, actions: input.thenActions },
  ];
  for (const b of input.elseIfBlocks ?? []) {
    blocks.push({ condition: b.condition, actions: b.actions });
  }
  if (input.elseActions && input.elseActions.length > 0) {
    blocks.push({ actions: input.elseActions });            // no condition = else
  }
  return { trigger: buildTrigger(input.trigger, input.afterTimeoutSeconds),
           actions: [{ type: "CONDITIONAL", conditionalBlocks: blocks }] };
}
```

`code.ts` conditional handler: for each `elseIf` entry, call the existing `buildCondition({ variable, operator, value })` for the condition and `buildNonConditionalAction` for each then-action ‚Äî exactly the paths already used for the primary `then`/`else`. No new validation logic; type-mismatch / variable-not-found errors propagate per-reaction as today.

## Section 4 ‚Äî list_reactions round-trip decoder (`encodeActionForListEcho`)

Generalize the `standardPattern` (currently `blocks.length <= 2`):

```
- detectTogglePattern(blocks) first (unchanged precedence).
- standardPattern: blocks.length >= 1, blocks[0].condition defined,
  every block EXCEPT possibly the last has a condition.
  (i.e. only the final block may be condition-less = else)
- Decode:
    condition = decode(blocks[0].condition); then = decode(blocks[0].actions)
    elseIf = blocks[1..] that HAVE a condition ‚Üí [{ condition, then }]
    else   = trailing block WITHOUT condition (if any) ‚Üí actions
- Non-conforming shapes (e.g. a condition-less block in the middle) ‚Üí { type: "CONDITIONAL", raw: blocks } fallback (unchanged).
```

Echo shape: `{ type: "CONDITIONAL", condition, then, elseIf?: [{ condition, then }], else? }`. `elseIf` omitted when empty (keeps the existing 2-block echo byte-identical ‚Üí no regression for v0.23.0 reactions).

## Section 5 ‚Äî `proto_conditional` sugar (`protoTools.ts`)

```ts
ProtoConditionalEntry = {
  from, trigger?, motion?,
  if: { variable, operator?, value },
  then: BranchAction,                                  // single branch sugar (Í∏∞Ï°¥)
  elseIf?: [{ if: { variable, operator?, value }, then: BranchAction }].min(1),  // NEW
  else?: BranchAction,
}
```

`compileProtoConditional` maps each `elseIf[i]` to a low-level `elseIf` entry: `condition` from `if` (operator default `==` applied by zod), `then: [compileBranchAction(entry.then)]`. Reuses `compileBranchAction` unchanged.

Overlay rewrite: `branchUsesOverlayOrSwap` is called on `then`, every `elseIf[*].then`, and `else`; if any returns true, `rewriteForOverlay` runs on the shared transition (same logic as v0.23.0, extended to scan the chain). Each branch sugar stays `.strict()` (no per-branch motion/trigger).

## Section 6 ‚Äî Test plan

- **Low-level zod** (`tests/tools.test.ts`): accept conditional with 1 elseIf; accept 2+ elseIf; accept elseIf + else; accept elseIf without else; reject empty `elseIf: []`; reject elseIf entry missing `condition`; reject elseIf entry with empty `then`; reject `conditional`/`toggle_variable` inside an elseIf then (nesting guard).
- **Builder** (`tests/reaction-builder.test.ts`): `buildConditionalReaction` with elseIfBlocks produces `conditionalBlocks` in order [then(cond), elseIf1(cond), elseIf2(cond), else(no cond)]; without elseIf the output is byte-identical to today (regression guard).

## Outcome ‚Äî BLOCKED by Figma platform (2026-06-01)

The feature was fully implemented (Tasks 1‚Äì5: low-level zod `elseIf`, N-block builder, plugin handler, N-block decoder, proto sugar + compile), all unit tests + typecheck green (357 tests). **Live verification proved Figma does not support else-if chains**, so the implementation was reverted (commits `ba33fa3`‚Üí`2134d2c`, unpushed ‚Äî recoverable via reflog) back to the plan commit `ebf14a3`. Spec + plan kept as historical record.

### Diagnostic evidence (systematic-debugging)

Instrumented every boundary (server‚Üíplugin wire, builder output, Figma read-back):

| Boundary | Observed |
|---|---|
| Server compile (`compileProtoConditional`) | Correct 3-part action: `then` + `elseIf:[‚Ä¶]` + `else` |
| Wire sent to plugin (server log) | Correct 3-block action (`then`/`elseIf`/`else`) |
| Plugin builder output (`__builtBlocks`) | **3** conditionalBlocks built correctly |
| Figma after `setReactionsAsync` read-back (`__storedBlocks`) | **2** ‚Äî Figma silently collapsed 3 ‚Üí 2 |
| Figma stored shape | `[{cond ‚â•3, navA}, {no-cond, navB}]` ‚Äî kept block0's condition, took block1's *actions* as the condition-less else, **dropped block1's condition and block2 (the real else) entirely** |
| Figma typings | `ConditionalBlock[]` (any length) ‚Äî but the documented example uses only 2 blocks (if + else) |
| **Figma prototype UI** | **No "Else if" option** ‚Äî the product only offers a single `If` + `Else`. |

### Root cause

Figma's prototype **conditional interaction is fundamentally single-level if/else** ‚Äî there is no else-if at the product level (the UI has no "Else if"), and the plugin API's `setReactionsAsync` only persists one conditional block + one condition-less else block. A multi-block `conditionalBlocks` array is accepted by the typings but the runtime keeps only the first condition + a single else. This is the same **runtime-vs-typings mismatch / silently-dropped-input** pattern as `initialVelocity` (v1.12), `deprecatedVersion` (v1.13), and readonly overlay frame props (v1.4-blocked).

### Lessons

1. **Prototype-capability check belongs in brainstorming** ‚Äî before designing a prototype-interaction feature, confirm Figma's **product UI actually exposes it** (not just that the typings allow the shape). The typings accepting `ConditionalBlock[]` led us to assume else-if was supported; the UI never offered it. Add this check alongside the existing readonly-check (from [[v1.4-blocked-by-figma]]).
2. **Live-probe block counts, not just success** ‚Äî `successCount: 1` was returned even though Figma silently dropped 2 of 3 blocks. The decisive diagnostic was instrumenting `__builtBlocks` vs `__storedBlocks` (build-vs-read-back count) ‚Äî a reusable technique for any "did Figma actually persist what we sent?" question.
3. **The implemented code was correct** ‚Äî the block was 100% platform, 0% our code. Reverting (not patching) was right.

If Figma ever ships else-if in prototypes, this spec + plan are a ready implementation blueprint; re-verify with the `__builtBlocks`/`__storedBlocks` probe before shipping.
