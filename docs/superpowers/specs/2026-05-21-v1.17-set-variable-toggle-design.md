# figma-prototype-mcp v1.17 — SET_VARIABLE + toggle_variable 설계 문서

- 작성일: 2026-05-21
- 상태: 초안 (사용자 승인 완료)
- 기반 버전: v0.15.0
- 시리즈: 새 action type `set_variable` (Figma `SET_VARIABLE`) + 단축 sugar action `toggle_variable` (BOOLEAN 전용, 내부 desugar = CONDITIONAL with 2 SET_VARIABLE blocks). VARIABLE_ALIAS / EXPRESSION 값, set_variable_mode, 라이브러리 변수는 후속 release.
- 참조:
  - `node_modules/@figma/plugin-typings/plugin-api.d.ts:5146-5150` — `{ type: 'SET_VARIABLE', variableId, variableValue? }`
  - `node_modules/@figma/plugin-typings/plugin-api.d.ts:7889-7910` — inline doc 예시 (toggleButton: CONDITIONAL + 2 SET_VARIABLE)
  - v1.16 `resolveVariableByName`, `buildCondition` 헬퍼 — 그대로 재사용

## 1. 목적

v0.15.0 기준 conditional reaction은 분기 가능하지만 **분기 안에서 할 수 있는 게 navigate/scroll/overlay/back/close/url/swap뿐**이라 가장 자연스러운 토글 패턴(`if x==true: set x=false; else: set x=true`)이 불가능. 결과: "햄버거 누르면 메뉴 열고 닫기" 같은 디자이너 표준 인터랙션이 자연어로 안 됨.

v1.17은 두 action을 노출:

1. **`set_variable`** — 변수에 리터럴 할당. conditional의 then/else 안에서도 쓰고 top-level에서도 씀.
2. **`toggle_variable`** — BOOLEAN 변수 단축 토글. 내부적으로 CONDITIONAL + 2 SET_VARIABLE로 desugar. 우리 API는 단일 액션처럼 보임.

## 2. Spec-단계 readonly/runtime 체크

Figma typings 5146-5150:

```ts
| {
    readonly type: 'SET_VARIABLE'
    readonly variableId: string | null
    readonly variableValue?: VariableData
  }
```

`readonly` 키워드는 declaration-level — 우리는 plain object를 `setReactionsAsync`에 통째로 넘기는 패턴이라 차단 없음. ✅

`variableValue?: VariableData` — optional이지만 우리 API에서는 항상 제공 (`value` 필수). `variableId: string | null` — null = 미설정. 우리 API에서는 항상 valid 변수 id 제공.

**Runtime 거부 위험**: typings-vs-runtime 미스매치가 발생할 수 있어 **bypass-probe 필수** (Task 1). v1.16에서 CONDITIONAL persist 확인했지만 SET_VARIABLE은 별개 — direct setReactionsAsync 호출 후 raw read로 같은 shape persist 검증.

**Toggle desugar의 raw shape**: CONDITIONAL action 안에 SET_VARIABLE 액션이 nested. v1.16 doc 예시 (line 7889-7910)에서 정확히 이 패턴을 보여줌 — 이론적으로 OK이지만 라이브 검증 필수.

## 3. 스코프

### v1.17 포함

- 새 action type **`set_variable`** — `{ variable: string, value: boolean | number | string }`.
- 새 action type **`toggle_variable`** — `{ variable: string }`. BOOLEAN 전용. desugar to CONDITIONAL.
- 변수 이름 → id 해석 (v1.16 `resolveVariableByName` 재사용).
- 타입 호환성 검증 (v1.16 buildCondition의 type-check 로직과 동일: BOOLEAN→boolean, FLOAT→number, STRING→string).
- list_reactions echo: SET_VARIABLE action 디코드 + CONDITIONAL action에 대한 **toggle 패턴 매칭** (매칭 시 `toggle_variable` 친화 shape으로 디코드, 매칭 실패 시 기존 conditional 디코드 fallback).
- 새 시나리오 23 (set_variable / toggle / conditional+set_variable 조합 / 에러).

### v1.17 비-목표

- `VARIABLE_ALIAS` 값 (`var = otherVar`).
- `EXPRESSION` 값 (`var = var + 1`, arithmetic operators).
- `SET_VARIABLE_MODE` action (Light/Dark 모드 전환 등).
- 라이브러리 (cross-file) 변수.
- nested conditional (v1.16 제약 유지).

## 4. 사용자 API

### 4.1 set_variable

```ts
{
  type: "set_variable",
  variable: string,                          // local variable name
  value: boolean | number | string,          // literal — type must match variable's resolvedType
}
```

**위치**: top-level OR conditional의 then/else 내부. `NonConditionalActionInput`에 포함.

### 4.2 toggle_variable

```ts
{
  type: "toggle_variable",
  variable: string,                          // local BOOLEAN variable name
}
```

**위치**: top-level **only**. conditional의 then/else 안에는 못 들어감 (Zod-rejected). 이유: desugar 결과가 nested CONDITIONAL action을 만들어 v1.16의 무중첩 규칙을 우회하게 됨.

### 4.3 사용 예시

```ts
// "메뉴 열기 버튼"
{ type: "set_variable", variable: "showMenu", value: true }

// "햄버거: 메뉴 토글"
{ type: "toggle_variable", variable: "showMenu" }

// "포인트 +10"  — number는 직접 값만; 산술은 비-목표
{ type: "set_variable", variable: "count", value: 10 }

// "if loggedIn==false then [count=10, navigate home]" — multi-action then
// (Conditional의 then이 이미 array라 가능)
{
  type: "conditional",
  condition: { variable: "loggedIn", operator: "==", value: false },
  then: [
    { type: "set_variable", variable: "count", value: 10 },
    { type: "navigate", targetFrameId: "home_id" },
  ],
}
```

## 6. Plugin handler 변경

### 6.1 새 헬퍼: buildSetVariableData(variable, value)

variable이 이미 resolve된 상태에서 호출. 타입 호환성 + VariableData wrapping을 한 곳에 캡슐화.

```ts
function buildSetVariableData(
  variable: Variable,
  value: boolean | number | string,
): unknown {
  // 타입 호환성 검증 (v1.16 buildCondition과 동일 로직)
  const valueType = typeof value;
  const expected =
    variable.resolvedType === "BOOLEAN" ? "boolean"
    : variable.resolvedType === "FLOAT" ? "number"
    : variable.resolvedType === "STRING" ? "string"
    : "unknown";
  if (expected === "unknown") {
    throw new Error(`Variable "${variable.name}" has unsupported type ${variable.resolvedType} for set_variable (supported: BOOLEAN, FLOAT, STRING)`);
  }
  if (valueType !== expected) {
    throw new Error(`Variable "${variable.name}" is ${variable.resolvedType}; cannot assign ${valueType} literal (expected ${expected})`);
  }
  return {
    type: variable.resolvedType,
    resolvedType: variable.resolvedType,
    value,
  };
}
```

(v1.16 `buildCondition`도 이 로직을 가지고 있는데 — 동일하다면 한 곳으로 추출하는 게 깔끔. Plan에서 결정.)

### 6.2 buildNonConditionalAction 확장 (set_variable branch)

기존 7-way switch에 8번째 case 추가:

```ts
if (action.type === "set_variable") {
  const { variable, warning } = await resolveVariableByName(action.variable);
  const variableValue = buildSetVariableData(variable, action.value);
  const built: BuiltAction = { type: "SET_VARIABLE", variableId: variable.id, variableValue };
  return { built, warning };
}
```

⚠️ 변경 사항: `buildNonConditionalAction` 시그니처가 **async**가 됨 (resolveVariableByName이 async). 기존 시그니처 `sync` → `async`로 바꿔야 함. 이 변경은 호출 사이트 (handleCreateReactions의 top-level 및 conditional then/else 루프)도 영향. 모두 `await`로 변경.

기존 7 branch도 `await` 가능하지만 (사실상 동기) — Promise wrapping은 microsecond 단위 비용. 회귀 위험 0.

### 6.3 handleCreateReactions top-level branch

기존 if/else 끝부분:

```ts
if (conn.action.type === "conditional") { ... }
else if (conn.action.type === "toggle_variable") {
  // toggle: desugar to CONDITIONAL { if x==true: set x=false; else: set x=true }
  const { variable, warning: resolveWarning } = await resolveVariableByName(conn.action.variable);
  if (variable.resolvedType !== "BOOLEAN") {
    throw new Error(`Cannot toggle non-BOOLEAN variable "${conn.action.variable}" (type: ${variable.resolvedType}); toggle_variable requires BOOLEAN`);
  }
  if (resolveWarning) warning = resolveWarning;

  const condition = {
    type: "EXPRESSION", resolvedType: "BOOLEAN",
    value: {
      expressionFunction: "EQUALS",
      expressionArguments: [
        { type: "VARIABLE_ALIAS", resolvedType: "BOOLEAN",
          value: { type: "VARIABLE_ALIAS", id: variable.id } },
        { type: "BOOLEAN", resolvedType: "BOOLEAN", value: true },
      ],
    },
  };
  const setFalseAction: BuiltAction = {
    type: "SET_VARIABLE",
    variableId: variable.id,
    variableValue: { type: "BOOLEAN", resolvedType: "BOOLEAN", value: false },
  };
  const setTrueAction: BuiltAction = {
    type: "SET_VARIABLE",
    variableId: variable.id,
    variableValue: { type: "BOOLEAN", resolvedType: "BOOLEAN", value: true },
  };

  newReaction = buildConditionalReaction({
    trigger: conn.trigger,
    afterTimeoutSeconds: conn.afterTimeoutSeconds,
    condition,
    thenActions: [setFalseAction],
    elseActions: [setTrueAction],
  });
} else {
  // existing non-conditional path
  const { built, warning: branchWarning } = await buildNonConditionalAction(...);
  ...
}
```

set_variable의 경우 별도 if 필요 없음 — `buildNonConditionalAction`이 처리 (set_variable이 NonConditionalActionInput에 포함됨).

### 6.4 list_reactions echo decoder 확장

`encodeActionForListEcho`:

```ts
if (action.type === "CONDITIONAL") {
  // 1) toggle 패턴 매칭 먼저
  const toggleVar = detectTogglePattern(action.conditionalBlocks);
  if (toggleVar) {
    const v = await figma.variables.getVariableByIdAsync(toggleVar.id);
    return { type: "toggle_variable", variable: v?.name ?? `<id:${toggleVar.id}>` };
  }
  // 2) 기존 conditional 디코드 (v1.16 로직)
  ...
}

if (action.type === "SET_VARIABLE") {
  const v = action.variableId ? await figma.variables.getVariableByIdAsync(action.variableId) : null;
  return {
    type: "set_variable",
    variable: v?.name ?? `<id:${action.variableId}>`,
    value: action.variableValue?.value,
  };
}
```

`detectTogglePattern(blocks)` — toggle desugar의 정확한 shape 매칭:
- blocks.length === 2
- block[0].condition: EXPRESSION + EQUALS + [VARIABLE_ALIAS(varA), BOOLEAN(true)]
- block[0].actions === [{ SET_VARIABLE, variableId === varA, variableValue: BOOLEAN(false) }]
- block[1].condition === undefined
- block[1].actions === [{ SET_VARIABLE, variableId === varA, variableValue: BOOLEAN(true) }]
- 매칭 시 `{ id: varA }` 반환, 실패 시 null.

## 7. Builder (reaction-builder.ts)

### 7.1 BuiltAction union 확장

```ts
export type BuiltAction =
  | { type: "NODE"; ... }
  | { type: "CLOSE" }
  | { type: "BACK" }
  | { type: "URL"; ... }
  | { type: "CONDITIONAL"; conditionalBlocks: ConditionalBlockShape[] }
  | { type: "SET_VARIABLE"; variableId: string; variableValue: unknown };
```

### 7.2 buildSetVariableReaction (top-level용)

set_variable이 top-level일 때 호출. `buildNonConditionalAction`이 이미 처리하지만 (단일 action 반환), full reaction wrapping용 builder 함수도 일관성 차원에서 추가:

```ts
export interface SetVariableBuildInput {
  trigger: TriggerInput;
  afterTimeoutSeconds?: number;
  variableId: string;
  variableValue: unknown;
}

export function buildSetVariableReaction(input: SetVariableBuildInput): BuiltReaction {
  return {
    trigger: buildTrigger(input.trigger, input.afterTimeoutSeconds),
    actions: [{ type: "SET_VARIABLE", variableId: input.variableId, variableValue: input.variableValue }],
  };
}
```

토글 builder는 안 만듬 — plugin handler가 `buildConditionalReaction` (v1.16 기존)을 직접 호출.

### Fixture (자동 생성, 기존 재사용 우선)

### Sub-cases

(a) **set_variable top-level**: button → `set_variable count = 10`. wire success. Figma raw read에 `SET_VARIABLE action`, variableId, variableValue=`{ FLOAT 10 }` 확인.

(b) **toggle_variable top-level**: button → `toggle_variable loggedIn`. wire success. Figma raw read에 CONDITIONAL action with 2 blocks. list_reactions echo는 `{ type: "toggle_variable", variable: "loggedIn" }`로 응답 (toggle 패턴 매칭).

(c) **conditional + set_variable**: `if loggedIn==false then [set count=10, navigate home]`. 단일 connection이 multi-action then 분기. Figma raw에 conditionalBlocks[0].actions 길이 2 확인.

(d) **Error — variable not found**: `set_variable doesNotExist = true` → "Variable not found".

(e) **Error — type mismatch**: BOOLEAN `loggedIn`에 `value: 42` → "BOOLEAN ... cannot assign number".

(f) **Error — toggle non-BOOLEAN**: FLOAT `count`를 toggle → "Cannot toggle non-BOOLEAN variable".

(g) **Defense — toggle inside conditional**: `then: [{ type: "toggle_variable", variable: "x" }]` → Zod reject (NonConditionalActionInput에 toggle 없음).

(h) **list_reactions echo 정확성**: (a) 후 echo가 `{ type: "set_variable", variable, value }`로 디코드. (b) 후 echo가 `{ type: "toggle_variable", variable }`로 디코드 (toggle 패턴 매칭 성공).

(i) **Manual Play-mode**: Figma Present 모드에서 toggle/set 동작 시각 확인 — 변수 패널에서 값 변화 관찰.

## 10. 라이브 리스크

1. **Bypass-probe 의무** (Task 1): SET_VARIABLE shape이 우리가 보내는 그대로 Figma persist 확인. 별도로 toggle desugar shape (CONDITIONAL + 2 SET_VARIABLE)도 1차 확인. 둘 다 persist OK여야 Task 2 진행.

2. **buildNonConditionalAction async화** — 시그니처 변경. v1.16 코드 (Task 5 추출 helper)를 sync에서 async로 변경. 호출 사이트 모두 await 추가. 회귀 위험 낮지만 typecheck로 사이트 누락 검증.

3. **Toggle 패턴 매칭의 정확성** — list_reactions에서 detectTogglePattern이 너무 엄격하면 toggle wire가 비-toggle conditional로 잘못 디코드. 너무 느슨하면 사용자가 직접 만든 conditional이 toggle로 잘못 인식. 정확한 매칭 (5 조건 동시) 필수.

4. **v1.16 buildCondition 타입 호환성 검증 로직과 중복** — buildSetVariableData가 같은 검증 함. plan에서 한 곳으로 추출 권장 (`validateVariableLiteralCompat(variable, value)` 같은 헬퍼).

5. **toggle_variable의 desugar는 write-only sugar** — Figma 저장 후 다시 list_reactions로 읽을 때 toggle-pattern matcher가 매칭해야 round-trip이 보장됨. 사용자가 직접 만든 (UI 또는 다른 도구로) CONDITIONAL+2 SET_VARIABLE도 매칭되면 toggle로 응답할 텐데, 이게 의도와 맞음 (functionally identical이라 사용자가 toggle로 보든 conditional로 보든 동작 같음).
