# figma-prototype-mcp v1.3 — Open Overlay + Close Overlay 설계 문서

- 작성일: 2026-05-20
- 상태: 초안 (사용자 리뷰 대기)
- 기반 버전: v0.2.0 (v1 + v1.1 + v1.2 누적)
- 직전 spec: `docs/superpowers/specs/2026-05-20-v1.1-scroll-to-design.md`

## 1. 목적

v0.2.0까지 지원되는 액션은 Navigate To와 Scroll To 두 가지다. 실제 프로토타이핑 작업에서 가장 흔한 다음 패턴은 모달·시트·드로어로, 이는 Open Overlay와 Close Overlay의 짝으로 표현된다. v1.3은 이 한 짝을 추가해 모달 패턴을 자연어 프롬프트로 완결한다.

## 2. 스코프

### v1.3 포함

- 새 action 타입: **Open Overlay** (Figma 내부 `type: "NODE"` + `navigation: "OVERLAY"`)
- 새 action 타입: **Close Overlay** (Figma 내부 `type: "CLOSE"`)
- 두 신규 variant를 `create_reactions` `action` discriminated union에 추가
- 사용자-노출 키워드: `action.type: "overlay" | "close"`

### v1.3 비-목표

- Swap Overlay (`navigation: "SWAP"`) — 다단계 오버레이 패턴
- Back action (`type: "BACK"`)
- Overlay 위치 옵션 (Center/Top-Left/...) — `overlayRelativePosition` 좌표 노출 안 함, 디폴트 `(0, 0)`
- Background dim/color 옵션 — 디폴트 `{ type: "NONE" }`
- `overlayBackgroundInteraction` 옵션 노출 — 디폴트 `"CLOSE_ON_CLICK_OUTSIDE"`
- 그 외 action 타입 (URL / SET_VARIABLE / SET_VARIABLE_MODE / CONDITIONAL / UPDATE_MEDIA_RUNTIME)
- 사용자가 Figma에서 직접 만든 오버레이 reaction을 변환·이전하는 마이그레이션

## 3. 변경 요약

| 영역 | v0.2.0 | v1.3 |
|---|---|---|
| `action` union variant 수 | 2 (navigate, scroll) | 4 (+ overlay, close) |
| 빌더 함수 | `buildNavigateReaction`, `buildScrollReaction` | + `buildOverlayReaction`, `buildCloseReaction` |
| Figma 내부 wire shape | NODE(navigate), NODE(scroll) | + NODE(overlay), CLOSE |
| 응답 result 구조 | 변경 없음 | 변경 없음 (`warning?`는 scroll 전용 유지) |

기존 툴(`get_canvas_overview`, `find_nodes`, `list_reactions`, `clear_reactions`)은 모두 변경 없음. `list_reactions`는 v0.2.0에서 추가된 `navigation` 필드 덕에 OVERLAY 액션도 자연스럽게 식별된다.

## 4. 툴 API

### 4.1 `create_reactions` (입력)

```ts
{
  connections: [
    {
      sourceNodeId: string,
      trigger?: "ON_CLICK" | "ON_HOVER" | "ON_PRESS",     // default ON_CLICK
      transition?: "INSTANT" | "DISSOLVE" | "SMART_ANIMATE", // default INSTANT
      action:
        | { type: "navigate", targetFrameId: string }
        | { type: "scroll",   targetNodeId: string }
        | { type: "overlay",  targetFrameId: string }     // NEW
        | { type: "close" }                                // NEW
    }
  ],
  replaceExisting?: boolean
}
```

Zod 표현은 기존 `z.discriminatedUnion("type", [...])`에 두 개의 새 object 스키마 추가.

`replaceExisting` 의미는 v1.1에서 정의한 그대로 — action 타입 무관, source 노드별로 기존 reactions 전체 제거 후 추가.

`transition`은 close 액션의 wire shape에 존재하지 않는다. 입력 레벨에서 사용자가 close에 transition을 지정해도 handler가 무시하고 빌더에 전달하지 않는다 (스키마 단위에서 거부하지 않음 — pass-through-ignore).

### 4.2 `create_reactions` (응답)

변경 없음. 단, overlay/close에는 별도 warning이 없다 (warning은 v1.1 scroll 전용으로 유지). `result.status`는 성공/에러 둘 중 하나.

검증 순서:
1. `sourceNodeId` 존재 / `setReactionsAsync` 가능 노드 확인
2. action 타입별 분기:
   - navigate: `targetFrameId` 존재 + FRAME 타입
   - scroll: `targetNodeId` 존재, 그 뒤 `findScrollableAncestor` (v1.1 warning)
   - overlay: `targetFrameId` 존재 + FRAME 타입
   - close: 추가 검증 없음
3. 빌더 호출 → `setReactionsAsync` 호출

`BuiltAction` union은 다음 셋으로 넓어진다 (정확한 라이브 shape은 §10 라이브 검증 후 확정):

```ts
export type BuiltAction =
  | {
      type: "NODE";
      destinationId: string;
      navigation: "NAVIGATE" | "SCROLL_TO" | "OVERLAY";
      transition: TransitionShape;
      preserveScrollPosition: false;
      // OVERLAY-only fields:
      overlayRelativePosition?: { x: number; y: number };
      overlayBackgroundInteraction?: "NONE" | "CLOSE_ON_CLICK_OUTSIDE";
      overlayBackground?: { type: "NONE" } | { type: "SOLID_COLOR"; color: { r: number; g: number; b: number; a: number } };
    }
  | { type: "CLOSE" };
```

> **라이브 API 검증으로 수정됨 (commit `75f398e`):** 초안에서는 `buildOverlayReaction`이 `overlayRelativePosition`, `overlayBackgroundInteraction`, `overlayBackground` 셋을 action 객체에 명시했다. 라이브 `setReactionsAsync`가 두 번 거부:
> 1. `overlayBackgroundInteraction`/`overlayBackground`는 reaction action이 아닌 destination overlay **frame의 property**다 (frame.overlayBackground 등).
> 2. `overlayRelativePosition`은 action에 있어도 되지만 destination frame의 `overlayPositionType === "MANUAL"`일 때만. 그 외면 "Actions only support overlayRelativePosition when navigation is OVERLAY and the destination is a frame with overlayPosition equal to MANUAL"로 거부.
>
> v1.3은 minimum scope — 세 필드 모두 제거하고 bare `{ type: "NODE", destinationId, navigation: "OVERLAY", transition, preserveScrollPosition: false }`만 emit. 오버레이 위치·배경·외부클릭 동작은 사용자가 Figma에서 destination frame property로 설정. v1.1 SCROLL_TO 정정과 같은 패턴.

## 9. 위험과 완화

- **Overlay wire shape 미정**: `overlayRelativePosition`, `overlayBackgroundInteraction`, `overlayBackground` 셋이 정말 필수인지·필드명·값 enum이 위 추측과 일치하는지 불확실. v1.1 SCROLL_TO 사고(`486d69c`)와 동일 패턴 — 단위 테스트 통과해도 라이브 API가 거부할 수 있다. **§8의 step 7을 의도적으로 라이브 first에 가깝게**: 구현 직후 시도, 거부되면 에러 메시지에서 정확한 필드 이름을 읽어 빌더 수정.
- **`CLOSE` 트리거 가능 여부**: `type: "CLOSE"` 액션은 Figma 어떤 trigger와도 조합 가능한지 확실하지 않음. ON_CLICK은 분명히 동작. ON_HOVER 등은 일단 pass-through.
- **`CommandQueue` 영향 없음**: handler 진입점이 동일한 `commandQueue.enqueue` 경로를 통과하므로 직렬화 안전. (v1.0.1 `d598703` 보장 그대로.)
- **`list_reactions`는 자동 호환**: v0.2.0의 `navigation` 노출 덕에 OVERLAY 액션도 시각 가능. CLOSE 액션은 `type: "CLOSE"`로 노출되며 destination이 없는 점에서 자명. 추가 변경 불필요.
