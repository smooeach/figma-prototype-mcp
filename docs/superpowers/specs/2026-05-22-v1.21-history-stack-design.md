# v1.21 — In-memory history stack + `proto_get_last_history`

Status: design (2026-05-22).
Predecessor: v0.19.0 (Phase B: proto_* + motion presets).
Origin: item #5 of the original 5-point Phase A/B/C spec the user provided ("인메모리 히스토리 스택 구현 (상태 유지)"). Also folds in the leftover item #3 disclaimer (description文구 on every proto_* tool).

## Goal

Record every successful `proto_*` tool call in an in-memory FIFO ring buffer (capacity 10) tied to the server's lifetime, and expose a new `proto_get_last_history` tool so an LLM can recover the source/target IDs + motion preset of recent actions when the user says things like "방금 만든 거 타이밍만 조금 수정해줘".

Also add a one-line Korean disclaimer ("이 도구는 오직 피그마의 프로토타입 인터랙션/애니메이션 생성·수정 목적으로만 사용합니다.") to the description of all 4 proto_* tools.

## Decisions (from brainstorm)

- **Recording scope:** `proto_wire` / `proto_overlay` / `proto_scroll` only. Low-level escape-hatch tools (`create_reactions`, `clear_reactions`, `set_frame_scroll`, etc.) are NOT recorded.
- **Per-entry data:** rich — full parsed input + tool name + minimal result summary + UUID + timestamp.
- **Query tool:** single `proto_get_last_history({ count?: 1..10 })`, default `count = 1`, always returns an array.
- **Eviction + lifecycle:** FIFO ring buffer, capacity 10, drop oldest on overflow. Cleared only on server restart. No reset tool. No clear-on-plugin-disconnect.
- **Partial success:** one tool call = one entry, only when `successCount > 0`. If a batch had 3 entries and 2 succeeded, the entry's `result` shows `successCount: 2, errorCount: 1`.
- **Disclaimer:** added to all 4 proto_* descriptions in v1.21.

## Section 1 — Architecture & module boundaries

```
src/
├── shared/
│   └── motionPresets.ts                 # unchanged
└── server/

    ├── history.ts                        # NEW — HistoryStore class + types + summarizeResult helper
    └── index.ts                          # instantiates HistoryStore, passes into registerToolHandlers
```

Rules:
1. `history.ts` is plugin-independent. It is a pure in-memory FIFO buffer with no awareness of `PluginSession`. Recording happens by explicit `historyStore.record(...)` calls from proto_* handlers — no auto-hook on `sendCommand`. The flow is explicit and easy to trace.
2. `HistoryStore` has no external dependencies (only `crypto.randomUUID()`). Fully unit-testable.
3. `src/server/index.ts` creates one `HistoryStore` instance and injects it into `registerToolHandlers(mcp, session, historyStore)`. Singleton-by-injection (mockable for tests).

## Section 2 — HistoryStore API + Entry type

### Entry shape

```ts
export type ProtoToolName = "proto_wire" | "proto_overlay" | "proto_scroll";

export interface HistoryEntry {
  historyId: string;          // UUID v4 from crypto.randomUUID()
  timestamp: number;          // Date.now()
  tool: ProtoToolName;        // which proto_* was called
  input: unknown;             // the fully-parsed input (ProtoWireInput | ProtoOverlayInput | ProtoScrollInput)
  result: {
    successCount: number;
    errorCount: number;
    warningCount: number;
  };
}
```

The `input` field is typed `unknown` because the union of three input types would pin the store to the proto-specific shapes. Callers discriminate by reading `entry.tool` and then narrowing the input themselves; the LLM does this trivially at the prompt level.

`result` is intentionally minimal — counts only, not the per-connection result array. The full `input` plus the count summary is enough to reconstruct what happened.

### HistoryStore class

```ts
import { randomUUID } from "node:crypto";

export class HistoryStore {
  private buffer: HistoryEntry[] = [];
  private readonly capacity: number;

  constructor(capacity = 10) { this.capacity = capacity; }

  record(tool: ProtoToolName, input: unknown, result: HistoryEntry["result"]): HistoryEntry | null {
    if (result.successCount === 0) return null;            // policy: no successes → no entry
    const entry: HistoryEntry = {
      historyId: randomUUID(),
      timestamp: Date.now(),
      tool, input, result,
    };
    this.buffer.push(entry);
    if (this.buffer.length > this.capacity) this.buffer.shift();   // FIFO drop oldest
    return entry;
  }

  getLast(count = 1): HistoryEntry[] {
    if (count < 1) return [];
    return this.buffer.slice(-Math.min(count, this.buffer.length));
  }

  size(): number { return this.buffer.length; }
  // No reset() — server-lifetime only per design decision Q4.
}
```

**Ordering convention:** `getLast(n)` returns `[oldest_of_n, …, newest]`. The last element of the returned array is the most recent call. LLM consumers using `arr.at(-1)` or `arr[arr.length-1]` get the latest.

**Successcount === 0 policy:** lives inside `record()` so the three call sites in `tools.ts` don't repeat the same `if`. Slightly couples the store to the "success" concept, but the savings outweigh the SRP purity cost.

### `summarizeResult()` helper (also in history.ts)

```ts
export function summarizeResult(raw: unknown): HistoryEntry["result"] {
  const r = raw as { successCount?: number; errorCount?: number; warningCount?: number };
  return {
    successCount: r.successCount ?? 0,
    errorCount:   r.errorCount   ?? 0,
    warningCount: r.warningCount ?? 0,
  };
}
```

The existing `create_reactions` plugin response already carries `successCount` / `errorCount` / `warningCount` (verified live during v1.20 probe — e.g. `{ "results": [...], "successCount": 1, "errorCount": 0, "warningCount": 0 }`). proto_* handlers receive that raw response from `session.sendCommand`; this helper strips it down to the 3 fields the store needs.

### `proto_get_last_history` tool zod

Co-located in `src/mcp-server/protoTools.ts` for parity with the other proto_* schemas:

```ts
export const ProtoGetLastHistoryInput = z.object({
  count: z.number().int().min(1).max(10).default(1),
});

export type ProtoGetLastHistoryInput = z.infer<typeof ProtoGetLastHistoryInput>;
```

### Handler in `src/server/tools.ts`

```ts
{
  name: "proto_get_last_history",
  description:
    "이 도구는 오직 피그마의 프로토타입 인터랙션/애니메이션 생성·수정 목적으로만 사용합니다. " +
    "Returns the most-recent successful proto_* tool calls as an array of HistoryEntry objects, " +
    "newest-last. Use when the user references \"the last thing I made\" / \"방금 만든 거\" " +
    "to recover the source/target IDs and motion preset, then re-call the corresponding proto_* " +
    "with replaceExisting=true to apply a modification.",
  schema: ProtoGetLastHistoryInput,
  handler: async (input) => {
    const { count } = input as ProtoGetLastHistoryInput;
    return { entries: historyStore.getLast(count) };
  },
},
```

The handler signature (`(input, session) => Promise<unknown>`) is shared with proto_* tools; this handler simply ignores `session` because history queries don't involve the plugin.

## Section 3 — proto_* description disclaimer + record() integration

### Disclaimer

Prepend to the description of all 4 proto_* tools (the 3 existing + the new history tool) the one-line Korean disclaimer:

> "이 도구는 오직 피그마의 프로토타입 인터랙션/애니메이션 생성·수정 목적으로만 사용합니다."

The existing English description continues immediately after. This is purely a documentation change inside `src/server/tools.ts` — no behavior change.

### record() integration

Each proto_* handler in `src/server/tools.ts` gains one `historyStore.record(...)` call after the `session.sendCommand` resolves and before returning. Pattern:

```ts
{
  name: "proto_wire",
  description: /* disclaimer + existing description */,
  schema: ProtoWireInput,
  handler: async (input, session) => {
    const parsedInput = input as ProtoWireInput;
    const compiled = compileProtoWire(parsedInput);
    const result = await session.sendCommand("CREATE_REACTIONS" as CommandName, compiled);
    historyStore.record("proto_wire", parsedInput, summarizeResult(result));
    return result;
  },
},
```

Same shape for `proto_overlay` and `proto_scroll`. The `record()` policy of skipping when `successCount === 0` keeps zero-success calls out of history.

### Dependency injection

`src/server/index.ts`:
```ts
import { HistoryStore } from "./history.js";
// …
const session = new PluginSession();
const historyStore = new HistoryStore();
// …
registerToolHandlers(mcp, session, historyStore);
```

`registerToolHandlers` signature changes from `(mcp, session)` to `(mcp, session, historyStore)`. The 6 low-level tools' dispatch is unchanged; only the 3 proto_* handlers and the new `proto_get_last_history` close over `historyStore`.

### Out of scope (explicit YAGNI)

- No `proto_reset_history` tool (per Q4).
- No persistence (no SQLite / JSON file). Server restart = clean history.
- No multi-client isolation (single PluginSession already enforces single-active).
- No diff display between entries (LLM constructs the diff if needed).
- No token-saving compression on the response (10-entry cap keeps response size bounded).
- No `historyId` filter on `proto_get_last_history` (only `count`).
- Low-level tools (`create_reactions`, `clear_reactions`, `set_frame_scroll`, etc.) are NOT recorded.

## Section 4 — Testing strategy + live verification

### Unit tests on top of the existing 221

**`tests/history.test.ts`** — pure HistoryStore tests, no plugin involvement.
- `record()` generates a UUID-shaped historyId + a numeric timestamp.
- `record()` skips entry creation when `result.successCount === 0`; `size()` stays unchanged and returns `null`.
- FIFO ring: pushing 11 entries leaves `size() === 10`, and the first-pushed entry is gone.
- `getLast(count)` defaults to 1, clamps to 10, clamps to `buffer.length` when `count > buffer.length`.
- `getLast()` returns entries in oldest-to-newest order; last element is the most recent.
- `summarizeResult()` handles empty object / partial fields / full object.

**`tests/protoTools-history.test.ts`** — integration test: proto_* handlers record correctly.
- `session.sendCommand` is stubbed via `vi.fn()` to return controlled response shapes.
- `proto_wire` success → store size +1, entry.tool === "proto_wire", entry.input matches the parsed input.
- `proto_overlay` success → entry.tool === "proto_overlay".
- `proto_scroll` success → entry.tool === "proto_scroll".
- Response with `successCount: 0` → store size unchanged.
- 11 sequential `proto_wire` calls → store size === 10, oldest dropped.

**`tests/tools.test.ts`** — append a couple `ProtoGetLastHistoryInput` zod cases:
- Accepts empty input (default count = 1).
- Accepts count: 5.
- Rejects count: 0.
- Rejects count: 11.

Expected total: 221 → ~245 (+24).

### Live verification — bypass-probe (3 shots)

v1.20 lesson: every probe exercises a full `tools/call` roundtrip, not just connection establishment.

| # | Sequence | Verifies |
|---|---|---|
| 1 | `proto_wire` × 1 → `proto_get_last_history` | `size === 1`; returned entry's `tool === "proto_wire"`, `input.wires[0].from/to` matches, `result.successCount === 1`. |
| 2 | `proto_overlay` × 1 → `proto_wire` × 1 → `proto_get_last_history({count: 2})` | Returns 2-element array; index 1 (newest) is the `proto_wire`, index 0 is the `proto_overlay`. Confirms ordering convention. |
| 3 | 11 sequential `proto_wire` (any combo) → `proto_get_last_history({count: 10})` | `size === 10`; oldest dropped; returned array length 10, timestamps strictly ascending. |

All 3 pass → ship. Any failure → fold into the spec as a runtime quirk and fix before continuing (same protocol as v1.20).

## Live verification outcome (2026-05-22)

All three bypass-probes passed against the real Figma file (MCP_Test_10 fixture, plugin connected via ws://localhost:3000/ws):

- **Probe 1** (`proto_wire 1073:30 → 1072:10`, then `proto_get_last_history`) — PASS. Returned entries array of length 1; entry had a valid UUID `historyId`, numeric `timestamp` close to call time, `tool: "proto_wire"`, full parsed `input` matching the call, and `result.successCount: 1`.
- **Probe 2** (`proto_overlay open 1073:28 → 1072:13`, then `proto_wire 1073:30 → 1072:10`, then `proto_get_last_history({count: 2})`) — PASS. entries.length === 2; `entries[0].tool === "proto_overlay"` (oldest), `entries[1].tool === "proto_wire"` (newest); `entries[0].timestamp < entries[1].timestamp`. Ordering convention (oldest-to-newest) confirmed live.
- **Probe 3** (11 sequential `proto_wire` calls, then `proto_get_last_history({count: 10})`) — PASS. entries.length === 10 exactly; timestamps strictly ascending; all entries `tool === "proto_wire"`. FIFO eviction at capacity confirmed live.

No live-API surprises this release — pure in-process state, no Figma-side runtime quirks to catch. The bypass-probe protocol is still mandatory regardless of where the risk surface seems to lie.
