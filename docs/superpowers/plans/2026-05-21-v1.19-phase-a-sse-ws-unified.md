# v1.19 Phase A — SSE + WS unified Express server Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the 3-process architecture (stdio MCP + standalone WS relay + Figma plugin) with a single 24/7 Express server that handles BOTH MCP-over-SSE (for Claude) and WebSocket (for the Figma plugin) on the same port. Preserve all 10 tools, all 173 unit tests' substantive coverage, and all natural-language behavior verified in scenarios 1–24.

**Architecture:** Express on `PORT` (default 3000). `GET /sse` is the MCP SSE endpoint; `POST /messages` is the SSE back-channel for client → server JSON-RPC. WebSocket upgrades on path `/ws` are dispatched to a `PluginSession` (single-active plugin — latest connection wins). Existing Zod schemas, reaction-builder, and plugin code.ts are unchanged. The `mcp-server/plugin-bridge.ts` (external WS client) is replaced by direct in-process `PluginSession.sendCommand`. Plugin manifest + ui.html migrate to the new URL.

**Tech Stack:** Express + `@modelcontextprotocol/sdk` (SSE transport) + `ws` (WebSocket upgrade on the Express http.Server) + existing Zod schemas + Vitest.

**Source spec:** `docs/superpowers/specs/2026-05-21-v1.19-phase-a-sse-ws-unified-design.md`

**Designer milestone protocol:** Each task ends with a one-line "디자이너 요약" note for the UXUI designer reader. Implementation subagents must include this note in their report.

```javascript
import { WebSocketServer } from "ws";
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { SSEServerTransport } from "@modelcontextprotocol/sdk/server/sse.js";
import { ListToolsRequestSchema } from "@modelcontextprotocol/sdk/types.js";

const PORT = 3001;  // probe uses 3001 to avoid colliding with real 3000

const mcp = new Server({ name: "probe", version: "0.0.0" }, { capabilities: { tools: {} } });
mcp.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{ name: "probe_ping", description: "probe", inputSchema: { type: "object" } }],
}));

const app = express();
const transports = new Map();
app.get("/sse", async (_req, res) => {
  const transport = new SSEServerTransport("/messages", res);
  transports.set(transport.sessionId, transport);
  res.on("close", () => transports.delete(transport.sessionId));
  await mcp.connect(transport);
});
app.post("/messages", express.json(), async (req, res) => {
  const t = transports.get(String(req.query.sessionId ?? ""));
  if (!t) { res.status(400).send("unknown session"); return; }
  await t.handlePostMessage(req, res);
});

const httpServer = app.listen(PORT, () => console.log(`[probe] http://localhost:${PORT}`));
const wss = new WebSocketServer({ noServer: true });
httpServer.on("upgrade", (req, socket, head) => {
  if (req.url === "/ws") wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
  else socket.destroy();
});
wss.on("connection", (ws) => {
  console.log("[probe] ws client connected");
  ws.send(JSON.stringify({ type: "ready" }));
  ws.on("message", (raw) => console.log("[probe] ws msg:", raw.toString()));
});
```

```typescript
import { randomUUID } from "node:crypto";
import type { WebSocket } from "ws";

interface Pending {
  resolve: (v: unknown) => void;
  reject: (e: Error) => void;
  timer: NodeJS.Timeout;
}

export interface PluginSessionOptions {
  /** Per-command timeout in ms (default 30s). */
  commandTimeoutMs?: number;
  /** How long sendCommand will wait for a plugin to become active before failing (default 3s). */
  connectWaitMs?: number;
}

export class PluginSession {
  private active: WebSocket | null = null;
  private pending = new Map<string, Pending>();
  private waiters = new Set<() => void>();
  private readonly commandTimeoutMs: number;
  private readonly connectWaitMs: number;

  constructor(opts: PluginSessionOptions = {}) {
    this.commandTimeoutMs = opts.commandTimeoutMs ?? 30_000;
    this.connectWaitMs = opts.connectWaitMs ?? 3_000;
  }

  isConnected(): boolean {
    return this.active !== null && this.active.readyState === 1;
  }

  setActive(ws: WebSocket): void {
    if (this.active && this.active !== ws) {
      try { this.active.send(JSON.stringify({ type: "system", message: "Replaced by newer connection" })); } catch {}
      try { this.active.close(); } catch {}
      this.failAllPending(new Error("Plugin connection replaced by newer connection"));
    }
    this.active = ws;
    try { ws.send(JSON.stringify({ type: "ready" })); } catch {}
    this.notifyWaiters();
  }

  clearActive(ws: WebSocket): void {
    if (this.active === ws) {
      this.active = null;
      this.failAllPending(new Error("Plugin disconnected"));
    }
  }

  handleResponse(msg: { id: string; status: "ok" | "error"; result?: unknown; error?: { message?: string } }): void {
    const p = this.pending.get(msg.id);
    if (!p) return;
    this.pending.delete(msg.id);
    clearTimeout(p.timer);
    if (msg.status === "ok") p.resolve(msg.result);
    else p.reject(new Error(msg.error?.message ?? "plugin error"));
  }

  async sendCommand(command: string, params: unknown): Promise<unknown> {
    if (!this.isConnected()) {
      await this.waitForConnection(this.connectWaitMs);
      if (!this.isConnected()) {
        throw new Error("피그마 플러그인 연결을 확인해주세요");
      }
    }
    const id = randomUUID();
    return new Promise<unknown>((resolve, reject) => {
      const timer = setTimeout(() => {
        this.pending.delete(id);
        reject(new Error(`Command ${command} timed out after ${this.commandTimeoutMs}ms`));
      }, this.commandTimeoutMs);
      this.pending.set(id, { resolve, reject, timer });
      try {
        this.active!.send(JSON.stringify({ type: "command", id, command, params }));
      } catch (e) {
        clearTimeout(timer);
        this.pending.delete(id);
        reject(e instanceof Error ? e : new Error(String(e)));
      }
    });
  }

  private waitForConnection(timeoutMs: number): Promise<void> {
    if (this.isConnected()) return Promise.resolve();
    return new Promise((resolve) => {
      const timer = setTimeout(() => {
        this.waiters.delete(waiter);
        resolve();
      }, timeoutMs);
      const waiter = () => {
        clearTimeout(timer);
        this.waiters.delete(waiter);
        resolve();
      };
      this.waiters.add(waiter);
    });
  }

  private notifyWaiters(): void {
    for (const w of [...this.waiters]) w();
  }

  private failAllPending(err: Error): void {
    for (const [id, p] of this.pending) {
      clearTimeout(p.timer);
      p.reject(err);
      this.pending.delete(id);
    }
  }
}
```

```typescript
import type { Server as HttpServer } from "node:http";
import { WebSocketServer, type WebSocket } from "ws";
import type { PluginSession } from "./sessions.js";

const PLUGIN_PATH = "/ws";

/**
 * Attach a WebSocket upgrade handler at `/ws` to the existing http.Server.
 * Routes the connected socket into the single-active PluginSession.
 */
export function attachPluginWebSocket(httpServer: HttpServer, session: PluginSession): WebSocketServer {
  const wss = new WebSocketServer({ noServer: true });

  httpServer.on("upgrade", (req, socket, head) => {
    if (req.url === PLUGIN_PATH) {
      wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
    } else {
      socket.destroy();
    }
  });

  wss.on("connection", (ws: WebSocket) => {
    session.setActive(ws);
    ws.on("message", (raw) => {
      let msg: unknown;
      try { msg = JSON.parse(raw.toString()); } catch { return; }
      if (typeof msg === "object" && msg !== null && (msg as { type?: unknown }).type === "response") {
        session.handleResponse(msg as { id: string; status: "ok" | "error"; result?: unknown; error?: { message?: string } });
      }
    });
    ws.on("close", () => session.clearActive(ws));
    ws.on("error", () => session.clearActive(ws));
  });

  return wss;
}
```

```typescript
  { capabilities: { tools: {} } }
);
registerToolHandlers(mcp, session);

const app = express();
const transports = new Map<string, SSEServerTransport>();

app.get("/sse", async (_req: Request, res: Response) => {
  const transport = new SSEServerTransport("/messages", res);
  transports.set(transport.sessionId, transport);
  res.on("close", () => transports.delete(transport.sessionId));
  await mcp.connect(transport);
});

app.post("/messages", express.json(), async (req: Request, res: Response) => {
  const t = transports.get(String(req.query.sessionId ?? ""));
  if (!t) {
    res.status(400).send("unknown session");
    return;
  }
  await t.handlePostMessage(req, res);
});

const httpServer = app.listen(PORT, () => {
  console.log(`[server] listening on http://localhost:${PORT}`);
  console.log(`[server]   MCP SSE endpoint: GET /sse`);
  console.log(`[server]   Plugin WebSocket:  ws://localhost:${PORT}/ws`);
});

attachPluginWebSocket(httpServer, session);

process.on("unhandledRejection", (err) => {
  console.error("[server] unhandledRejection:", err);
});
process.on("uncaughtException", (err) => {
  console.error("[server] uncaughtException:", err);
});
```

```markdown
**Why:** The 3-process architecture (stdio MCP child + standalone WS relay + Figma plugin) was an operational pain point: 3 terminals, channel name management, session-scoped lifecycle. v1.19 collapses to 1 process suitable for 24/7 background operation. **The transport change is the foundation for v1.20 (proto_* + presets) and v1.21 (history stack) — those phases can't start until the unified server is stable.**

**Patterns introduced:**

1. **Same-port HTTP + WS via `noServer: true`.** Express's \`http.Server\` instance shared between Express routes and \`WebSocketServer\` upgrade handling on a specific path (\`/ws\`). Standard Node pattern but worth documenting because it's the foundation for any future server features that want to use the same port.

2. **In-process bridge replaces external WS client.** v0.17.0's \`PluginBridge\` was a WS *client* connecting to a separate relay. v0.18.0's \`PluginSession\` directly tracks the plugin's connected WebSocket as a server-side counterparty. The dispatch surface (Promise-based \`sendCommand(command, params)\`) is intentionally identical so the \`registerToolHandlers\` consumer doesn't change.

3. **Single-active replaces channel-based.** Channel-based pub/sub was overkill for solo dev workflows. \`PluginSession.setActive(ws)\` evicts the prior connection (system message + close) before accepting the new one. This loses multi-channel isolation but adds zero configuration burden.

4. **3-second connect-wait at the tool-call boundary.** When a tool is invoked but no plugin is connected, \`sendCommand\` waits up to 3s for a plugin to attach (typical reconnect window) before returning the Korean user-facing error. Prevents indefinite hangs without sacrificing the legitimate reconnect-mid-call case.

**Bypass-probe applied as usual.** Task 1 verified \`@modelcontextprotocol/sdk\` SSE transport + Express + \`ws\` upgrade integration before any production code change.

**Designer milestone summaries** were included per task — a small UX improvement for the human reader watching a long subagent execution.

See also [[v1.18-shipped]] (prior release), [[repo-and-releases]] (v0.18.0 entry).
```

## Risk + Fallback

- **Task 1 probe failure (SDK SSE not as expected)**: STOP. The plan assumes \`SSEServerTransport\` exists at \`@modelcontextprotocol/sdk/server/sse.js\`. If the import path moved or the API changed, record actual shape and update Task 6 accordingly. If SSE was retired in favor of a newer transport (Streamable HTTP), pivot to that.
- **Task 7 plugin ui.html ↔ sandbox code mismatch**: the existing \`src/figma-plugin/code.ts\` listens for \`pluginMessage.type === "command"\` and emits \`pluginMessage.type === "response"\`. The new \`ui.html\` must bridge the \`envelope\` shape correctly. If scenario 25 (a–e) fail after deployment with \`command_unknown\` or no response, the bridging in \`ui.html\`'s \`window.onmessage\` is the prime suspect.
- **Task 8 leaves dangling imports**: deleting \`src/mcp-server/index.ts\` cleanly works only if no other file imports from it. Run \`grep -r "mcp-server/index" src tests\` after deletion. If anything still imports, the import is stale — update or remove.
- **Plugin reload doesn't pick up new manifest**: rare, but Figma's plugin dev loader sometimes caches the manifest. If the plugin still tries port 3055 after reload, instruct the user to **remove + re-import** the plugin from `dist/figma-plugin/manifest.json`.
- **Claude SSE config not picked up**: Claude Desktop or Claude Code may need a full restart, not just a config reload. Document this clearly in Task 10 Step 4.
- **3-second timeout false positives during legitimate reconnect**: 3s is a deliberate compromise. If a user reports false-positive errors during plugin reload, document the workaround (wait + retry the prompt) — not a code change in v1.19.
- **Subagent unable to read user's Claude config**: Task 10 Step 4 is user-actionable, not subagent-actionable. The implementing subagent should produce the EXACT JSON snippet and instruct the controller to pause for user confirmation.
