# v1.4 `configure_overlay_frame` — Implementation Plan (BLOCKED)

> **🚫 STATUS:** Implementation attempted on 2026-05-20, blocked by Figma plugin API readonly limitation. See `docs/superpowers/specs/2026-05-20-v1.4-configure-overlay-frame-design.md` (the companion spec) for the full discovery and learnings.
>
> The five implementation commits were reverted from `main` (squashed back to v0.3.0 = commit `7f429ec`). This plan is preserved as a historical record and as a ready-to-execute template if Figma later exposes setters for overlay frame properties.

## Summary of the blocker

Figma marks all three overlay frame properties as `readonly`:

```ts
readonly overlayPositionType: OverlayPositionType
readonly overlayBackground: OverlayBackground
readonly overlayBackgroundInteraction: OverlayBackgroundInteraction
```

Direct assignment from the plugin returned `"no setter for property"`. No `setOverlay*Async` or equivalent mutation method exists in the typings. The MCP cannot programmatically configure these without an API change from Figma.

## What was attempted before revert

Five implementation commits, all reverted via `git reset --hard 7f429ec` (none had been pushed):

- `dbbc386` docs: README v1.4 — configure_overlay_frame tool, scenario 8

- `99c422e` feat(tools): add ConfigureOverlayFrameInput schema and types
- `0681aed` feat(overlay-config): add pure mapping module for overlay frame props

The pure mapping module (`overlay-config.ts`) and its 12 unit tests would still work as-is. Only the frame-property writes inside `handleConfigureOverlayFrame` are blocked. If Figma later exposes setters, recover that commit via `git show 0681aed` and continue with task 2 of the plan body below.

## Re-activation criteria

Figma exposes any of:
- `readonly` removed from the three properties
- New `setOverlay*Async` methods
- Document-level prototype-settings API
- Other mutation paths for overlay frame appearance
